# V6.0「信眾時間軸」交付說明

本輪目標：搜尋到一位信眾後，能一眼看到這位信眾與所屬家戶歷年的完整紀錄。
本輪為**唯讀功能**，不刪除、不修改、不覆蓋任何既有資料，也沒有建立任何假資料。

---

## 一、修改檔案清單

### 新增檔案
| 檔案 | 用途 |
|---|---|
| `src/lib/timeline.ts` | 時間軸資料整合邏輯（唯讀），整合 RitualRecord + 舊 Activity 資料 |
| `src/app/household/[id]/timeline/page.tsx` | 時間軸頁面（`/household/{id}/timeline`） |
| `src/components/timeline/TimelineScreen.tsx` | 時間軸主畫面（視角切換、年度篩選、年度分組） |
| `src/components/timeline/TimelineEntryCard.tsx` | 單筆紀錄卡片（摘要 + 展開查看詳細內容） |

### 修改既有檔案
| 檔案 | 修改內容 |
|---|---|
| `src/components/SearchBar.tsx` | 搜尋結果點擊時，把 `memberId` 一起帶進網址（`?member=xxx`），供時間軸頁面預設切到該成員視角 |
| `src/app/household/[id]/page.tsx` | 讀取網址上的 `?member=` 參數（驗證真的屬於這一戶才採用），往下傳給操作面板 |
| `src/components/household/QuickActionsPanel.tsx` | 新增「📜 歷年紀錄」按鈕，點擊開啟時間軸頁面（保留成員資訊） |

**沒有變動**：`prisma/schema.prisma`（無新增資料表/欄位）、`package.json`（無新增套件）、
任何既有的普渡登記/列印/API 邏輯（時間軸只讀取，不呼叫任何修改用的 API）。

## 二、時間軸資料來源說明

| 來源 | 狀態 | 說明 |
|---|---|---|
| `RitualRecord`（+ `UniversalSalvationDetail` + 登記項目） | ✅ 目前唯一有實際內容 | 普渡登記資料；年度燈/宮慶/補印尚未開發，等這些模組有資料時會自動一起出現在時間軸上，不需要修改 `timeline.ts` |
| `Activity`（舊資料表） | ⚠️ 目前是空的 | 只讀顯示、不再寫入（V2.0 起明確停用）。程式已經整合這個資料來源，但**目前正式環境與種子資料裡這張表完全沒有任何一筆資料**，所以時間軸目前不會顯示任何來自這裡的項目——這不是 bug，是誠實反映目前資料庫的實際狀態 |
| 家戶資料異動紀錄 | 🚫 目前沒有這個資料來源 | 系統目前沒有任何「家戶欄位在何時被誰改成什麼」的紀錄機制（`AuditLog` 資料表目前只服務財務模組，且系統完全沒有登入/操作者身份）。本輪**沒有為此新增任何資料表或假資料**，這是已知限制，見下方 |

時間軸會把以上來源攤平成同一種格式，依年度由新到舊排序；同一年度內，先普渡、
再依活動類型字母序排列；沒有年度資訊的舊資料（目前不存在，但程式有防呆）會
另外分到「年度不詳」分組，排在最後面。

## 三、測試方式

部署完成、取得網址後，建議依下列步驟測試（以種子資料 F00009 王家為例）：

1. **入口**：首頁搜尋「王小明」→ 點進王家 → 家戶頁「今天要辦」區塊應該會看到
   新的「📜 歷年紀錄」按鈕。
2. **成員視角預選**：點擊「歷年紀錄」，時間軸頁面應該預設顯示「王小明」視角
   （視角切換列裡「王小明」按鈕應該是選中狀態），且網址列會帶著
   `?member=xxx`。
3. **切回整戶視角**：點擊「👪 整戶視角」，應該改成顯示整戶（含家戶共同紀錄）
   的所有紀錄。
4. **年度切換**：分別點擊「全部年度」「今年」「去年」「前年」，以及用下拉選單
   直接選一個年度（例如 114），畫面上顯示的紀錄應該會即時改變，**過程中不應該
   看到任何 loading 狀態或網路請求**（因為篩選純粹是前端運算，資料只在一開始
   載入一次）。
5. **卡片內容**：114/115 年的普渡紀錄卡片，應該看得到：活動類型、狀態、
   對應成員或「家戶共同」、摘要（是否已報名、四類筆數、是否贊普、普渡桌）、
   建立/更新日期。
6. **展開查看**：點擊卡片上的「展開詳細內容」，應該看到完整的歷代祖先/個人
   乙位正魂/冤親債主/無緣子女清單，以及陽上姓名、安奉位置、贊普資料、
   普渡桌、備註——**畫面上不應該出現任何可以編輯或刪除的按鈕**。
7. **資料安全確認**：測試完成後回到普渡登記畫面（`.../rituals/universal-salvation`），
   確認資料完全沒有被時間軸頁面影響過（因為時間軸從頭到尾沒有呼叫任何
   PATCH/POST/DELETE API）。

## 六、已知限制

1. **舊 Activity 資料目前是空的**：程式已經整合這個資料來源，但正式環境跟
   種子資料裡都沒有任何一筆，時間軸暫時看不到任何「舊資料」項目——不是功能
   沒做，是目前真的沒有資料可以顯示。
2. **家戶資料異動紀錄目前無法整合**：系統沒有任何家戶欄位修改歷史的紀錄
   機制，也沒有登入/操作者身份，這個資料來源本輪沒有做（也不會用假資料
   充數）。如果之後需要這個功能，需要另外規劃一張變更紀錄表跟登入機制。
3. **目前所有普渡 RitualRecord 的 memberId 都是空的**：現有的普渡登記畫面
   （V3.0～V3.2）建立資料時，本來就不會指定特定成員（一戶一年一筆，家戶
   共同）。所以「成員視角」目前實際看到的內容，跟「整戶視角」幾乎一樣
   （都是家戶共同紀錄）——這是資料現況，不是時間軸的篩選邏輯有問題；
   之後如果有需要把某筆普渡登記指定給特定成員的功能，時間軸會自動正確
   分流顯示，不需要修改這支畫面。
4. **本輪程式碼尚未部署/尚未經過瀏覽器實測**：跟前幾輪一樣，需要你上傳到
   GitHub 讓 Render 重新部署後，我再用瀏覽器工具實際驗證並補上截圖。
5. **V3.2～V5.1 的部署狀態也都還沒收到確認**：建議這次連同 V6.0 一起上傳，
   我可以一次幫你把所有還沒驗證過的版本都測試一輪。
