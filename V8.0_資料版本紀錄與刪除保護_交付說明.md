# V8.0「資料版本紀錄與刪除保護」交付說明

本輪是「備份、還原與版本管理系統」十大項需求裡的**第一個子模組**，也是
V6.2～V7.1.1 一連串規格文件之後，**這一輪真正動到程式碼**（`prisma/schema.prisma`／
`src/lib/`／`src/app/api/`／`src/components/`），不是規格文件。

依照你的選擇：「開始實際開發」＋「資料版本紀錄＋刪除保護（推薦）」，本輪只做這
兩項，其餘八項（自動備份、手動備份、資料還原核心、程式版本管理頁面、更新保護、
備份儲存位置、權限強化）都還沒有開始，等這一輪確認沒問題後再個別討論下一步。

---

## 一、這一輪做了什麼

### 1. 資料版本紀錄

新增 `RecordVersion` 資料表，記錄「誰、什麼時候、對哪一筆資料、做了什麼動作
（新增/修改/刪除/還原/回復到舊版本/永久刪除）、改之前是什麼、改之後是什麼」。

實際接上這套機制的資料：

- **家戶資料**：修改（PATCH）會留一筆修改紀錄。
- **信眾成員**：新增會留一筆紀錄。
- **普渡登記**（整筆登記主檔＋明細＋每一筆登記項目）：建立、修改、刪除都會
  留下對應的紀錄，這是目前唯一有完整「建立→修改→刪除」全流程的資料。

家戶詳細頁新增「🕘 修改紀錄」按鈕，點開可以看到這筆資料的完整歷史，展開查看
改了哪些欄位、改之前/改之後的值，也可以把某一次的修改「回復」回去。

### 2. 刪除保護（回收區）

普渡登記（整筆或單一登記項目）刪除時，不再是真的從資料庫刪除，而是先移入
「回收區」，至少保留 30 天，這段期間都可以還原；超過 30 天才能在回收區裡執行
「永久刪除」。

首頁新增「🗑 回收區」入口，可以看到目前回收區裡的所有資料，逐筆還原或（超過
保留期限後）永久刪除。永久刪除前也會留一筆紀錄，這樣即使資料真的消失了，
系統裡還是留得住「這裡曾經有過什麼、什麼時候被誰刪除」的軌跡。

### 3. 資料庫變更

- `households`／`members`／`ritual_records`／`universal_salvation_entries`
  四張表新增 `deletedAt`／`deletedByName` 兩個欄位（軟刪除用）。
- 新增 `record_versions` 資料表。
- 這是**這一輪唯一的資料庫結構變更**，migration 檔案是
  `prisma/migrations/20260716000000_version_history_recycle_bin/migration.sql`，
  純粹新增欄位/新增資料表，**不會影響任何既有資料**。

## 二、誠實說明：這一輪「沒有」做的事

依照專案「不要自行增加流程」的原則，這一輪只針對「目前系統已經有的修改/刪除
功能」接上版本紀錄與刪除保護，**沒有**額外發明原本不存在的功能：

- **家戶目前沒有「刪除」功能**：資料庫已經加上軟刪除欄位，但沒有新增「刪除
  家戶」的按鈕或 API。未來如果要開放這個功能，資料庫結構已經準備好了，不需要
  再改 schema。
- **信眾成員目前沒有「修改」或「刪除」功能**：所以成員只會留下「新增」的紀錄，
  不會有「修改」或「刪除」的紀錄——因為系統本來就沒有這兩個功能。
- 回收區頁面實際上目前只會出現「普渡登記」相關的資料，因為只有這個功能有
  真正的刪除入口。

這不是漏做，是刻意的範圍控制。詳細對照表跟原因說明在
`docs/BACKUP_VERSION_CONTROL.md`，之後每一輪都可以回去查這份文件，不用重新
爬程式碼推敲設計理由。

## 三、目前最大的限制：系統還沒有登入功能

這一輪的「修改人」「操作人」欄位（`operatorName`），**是使用者自己在畫面上
填寫的文字**，不是系統驗證過的真實身份——因為 ERP 目前完全沒有登入/帳號
機制，沒辦法知道「現在是誰在操作」。

這代表：

- 任何人都可以在這個欄位填任意姓名，或乾脆留空。
- 需求裡「九、權限——只有超級管理員可以操作」這一項，這一輪**沒辦法在後端
  真正擋下**，回收區跟修改紀錄頁面目前對所有人開放，只在畫面上用文字提醒
  這個限制。

這是刻意先做能做的部分、誠實標註限制，而不是為了做這個功能硬做一套假的登入
機制。等系統之後做出真正的登入/帳號功能，這裡預留的欄位跟這份文件裡的說明，
可以直接拿來把「操作人」升級成真正驗證過的身份，並補上後端權限檢查。

## 四、這個沙盒環境做不到、需要你實機測試的部分

跟前面每一輪真正動到程式碼的交付一樣，這個開發環境**連不到 npm 套件庫、
GitHub、Render**，所以沒辦法在這裡執行：

- `npm install` / `prisma generate` / `prisma migrate dev`
- `npm run build`
- 啟動伺服器、實際點擊畫面測試

**我做了這些不需要安裝套件也能執行的替代檢查**，全部通過：

1. Prisma schema 結構檢查：13 個 model、12 個 enum，沒有重複定義；
   `datasource`/`generator` 各只有一個；`migration_lock.toml` 的 provider
   跟 schema 一致；`prisma/migrations/` 底下確認是三個資料夾（`init`／
   `ritual_core`／這次新增的 `version_history_recycle_bin`），沒有孤立或
   衝突的 migration。
2. 全部這一輪新增/修改的 26 個檔案，大括號/小括號/中括號逐一配對檢查，
   沒有發現任何不平衡。
3. `src/` 底下全部 95 個 `@/` 別名 import 逐一解析，沒有找不到的路徑。
4. Client/Server 邊界檢查：確認新增的 `"use client"` 元件
   （`VersionHistoryPanel`／`RecycleBinScreen`／`ConfirmDialog`）都沒有在
   執行期 import 伺服器專用的 `lib` 模組。
5. 沒有新增任何 npm 套件依賴，`package.json` 不需要異動相依套件清單（只把
   `version` 欄位從 `6.1.0` 更新為 `8.0.0`，純粹版本標籤）。
6. 逐一確認每一個會「修改資料＋寫入版本紀錄」的地方，都放在同一個資料庫
   交易（`prisma.$transaction`）裡執行——避免「資料改了、但版本紀錄沒寫成功」
   或反過來的情況。

**部署後請務必先在測試環境跑過一次** `prisma migrate deploy`，然後實際操作
一輪完整流程：新增一筆普渡登記 → 修改 → 刪除 → 到回收區還原 → 再刪除一次
→ 等（或先手動調整資料庫時間來測試）超過 30 天 → 永久刪除 → 到「修改紀錄」
查看完整歷史 → 回復到舊版本，確認每一步都符合預期再正式上線使用。
