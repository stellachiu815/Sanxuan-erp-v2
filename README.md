2. **PostgreSQL** 其中一種：
   - Docker Desktop（推薦，最乾淨）：https://www.docker.com/products/docker-desktop/
   - 或 Homebrew：`brew install postgresql@16 && brew services start postgresql@16`
   - 或 Postgres.app：https://postgresapp.com

## 啟動方式（Mac）

雙擊專案資料夾裡的 **`start.command`**。

第一次啟動會自動：

1. 安裝套件（`npm install`，需要網路）
2. 建立 `.env`（資料庫連線設定）
3. 如果你有裝 Docker，自動用 `docker compose` 啟動一個乾淨的 PostgreSQL
4. 同步資料庫結構（`prisma migrate deploy`，依照 `prisma/migrations/` 裡的異動歷史檔案
   建立資料表，保留完整的異動紀錄，不會再用 `db push`）
5. 匯入測試種子資料（F00009 王家）
6. 啟動網站並自動開啟瀏覽器 `http://localhost:3000`

如果你已經有自己的 PostgreSQL（不是用 Docker），請先把 `.env.example` 複製成 `.env`，
把裡面的 `DATABASE_URL` 改成你自己的連線字串，再雙擊 `start.command`。

之後每次啟動都一樣，直接雙擊 `start.command` 即可，不會重複安裝或重複建立測試資料。

停止網站：在跳出來的終端機視窗按 `Control + C`。

## 測試 Excel 批次匯入

1. 啟動系統後，到首頁最下面點「家戶資料 Excel 批次匯入」，或直接開啟 `/import`。
2. 上傳附件裡的 **`匯入範本_5戶測試資料.xlsx`**（5 戶、共 7 筆成員資料，欄位格式範例都在裡面）。
3. 上傳後畫面會顯示：總筆數、可匯入、錯誤、待確認（重複）四個數字，以及每一列的詳細狀態：
   - 「可匯入」：格式正確、家戶編號是全新的，會真正被匯入
   - 「錯誤」：格式有問題（例如生日格式看不懂、必填欄位空白），永遠不會被匯入，
     要修正 Excel 內容後重新上傳
   - 「待確認（家戶編號重複）」：家戶編號已經存在資料庫，**不會覆蓋**，只會記錄下來，
     可以到 `/import/pending` 查看
4. 確認「可匯入」的資料沒問題後，按「確認匯入這 N 筆資料」，畫面會顯示實際新增了
   幾戶、幾位成員、幾筆祭祀資料。
5. 測試「重複家戶編號會被擋下」：把同一個檔案**再上傳一次**，這次應該全部 5 戶都變成
   「待確認（家戶編號重複）」，不會產生重複資料，也不會覆蓋剛剛匯入的內容。
6. 確認 5 戶測試資料驗證都沒問題後，再把正式的家戶資料整理成同樣的欄位格式上傳。

Excel 欄位格式（因為需求只列出欄位名稱，沒有規定儲存格內要怎麼填，以下是目前程式
使用的建議格式，寫在 `src/lib/importRules.ts` 最上面的註解，正式使用前請先確認符合
你們登記資料的習慣，需要調整請告訴我）：

- 國曆生日／農曆生日：`yyyy-MM-dd`（例如 `1958-03-12`），農曆生日如果是閏月，
  請在後面加 `(閏)`，例如 `1960-06-05(閏)`；兩個生日欄位只填一種即可，系統會自動換算另一種、
  生肖跟實歲/虛歲
- 是否已辭世：填「是」或「否」，空白視為「否」
- 歷代祖先／個人乙位正魂：有填文字才會建立一筆祭祀資料，文字內容就是祭祀資料的名稱；
  兩欄都填會建立兩筆祭祀資料，共用同一列的「陽上姓名」「安奉位置」
- 同一個家戶編號的多列（同一戶的不同成員），家戶層級欄位（家戶名稱/主要聯絡人/電話/地址/
  公司名稱）請填一樣的內容，不一致的話這個家戶所有列都會被判定為錯誤

## 財務模組架構預留

這一輪**沒有新增任何看得到、按得到的功能**，只是先把財務模組未來需要的資料模型跟權限規則
定義好，方便之後直接開發、不用重新設計架構。實際驗收時可以檢查：

- `npm run prisma:studio` 打開資料庫檢視工具，應該能看到新增的 `users`、
  `finance_records`、`audit_logs` 三張表（種子資料裡有一筆 `admin@sanxuan.local`
  的 `SUPER_ADMIN` 帳號，但因為系統還沒有登入功能，這個帳號目前不會被用到）
- 家戶管理（搜尋、家戶頁、新增家人、修改資料、新增祭祀資料）跟之前完全一樣，沒有受影響

權限規則（`src/lib/permissions.ts`，目前只是定義，還沒有任何 API 真正呼叫它）：

- `SUPER_ADMIN`：財務模組的查看、新增、修改、作廢、匯出全部開放
- `STAFF`（目前系統的預設角色）：完全看不到財務功能，選單也不會顯示
- `FINANCE_CLERK`（預留角色，尚未開放給任何人）：只能新增草稿，不能修改已確認資料、
  不能作廢、不能看完整報表

之後真的開發財務模組時，務必遵守：

1. 財務資料不可永久刪除，只能作廢（`FinanceRecordStatus` 只有 `DRAFT`／`CONFIRMED`／`VOID`
   三種狀態，沒有刪除這個選項）
2. 每一支財務 API route 開頭都要呼叫 `canFinance()` 或 `assertFinancePermission()`
   做後端權限檢查，不能只在前端把按鈕藏起來
3. 新增、修改、作廢都要寫一筆 `AuditLog`（操作者、時間、修改前後內容、原因）
4. 開發財務模組之前，系統必須先有真正的登入/session 機制，`getCurrentUser()`
   目前是回傳 `null` 的預留函式，還沒有真正實作

## 部署到 Render（提供線上測試網址）

**為什麼沒辦法直接幫你部署好**：我（Claude）目前是在一個雲端工作環境裡開發這個專案，
這個環境的網路被限制住，連不到 Render、GitHub 等任何網站，也沒辦法幫你申請 Render
帳號或連接 GitHub 帳號授權——這兩件事本來就必須由帳號本人操作，即使我能連上網路也一樣
不會幫你做這兩步。以下步驟你只需要滑鼠點一點、瀏覽器操作，不需要用到 Terminal：

### 步驟

1. **把程式碼放到 GitHub**（免費）：
   - 到 https://github.com 註冊/登入，右上角「+」→「New repository」，建一個新的
     private repository（例如叫 `sanxuan-erp`）
   - 進到這個新 repo，點「uploading an existing file」，把附件裡
     `三玄宮ERP.zip` 解壓縮後的內容**除了 `node_modules`、`.next`、`.git` 這三個資料夾**
     以外全部拖拉上傳（GitHub 網頁版支援直接拖拉整批檔案，不需要 git 指令）
2. **到 Render 建立 Blueprint**：
   - 到 https://dashboard.render.com 註冊/登入（可以直接用 GitHub 帳號登入，順便完成
     GitHub 授權）
   - 右上角「New +」→「Blueprint」→ 選擇剛剛上傳的 `sanxuan-erp` repo
   - Render 會自動讀到專案根目錄的 `render.yaml`，同時建立：
     - `sanxuan-erp`：網站服務（Node.js，免費方案）
     - `sanxuan-erp-db`：PostgreSQL 資料庫（免費方案），並自動把連線字串設成網站的
       `DATABASE_URL` 環境變數
   - 按「Apply」，等待幾分鐘（第一次要 `npm install` 加 build，會比較久）
3. **部署完成**：Render 會給一個 `https://sanxuan-erp-xxxx.onrender.com` 的網址，
   這就是可以直接點開測試的線上網址，不需要密碼、不需要裝任何東西
4. **（選用）載入測試種子資料**：在 Render 的網站服務頁面點「Shell」分頁，執行
   `npm run seed`，會建立 F00009 王家的測試資料，方便你照下面「測試方式」操作
5. **之後怎麼更新**：以後把新的程式碼 push 到這個 GitHub repo，Render 會自動重新部署，
   不需要再手動做任何事——這就是「更新方式」

### 免費方案的限制（Render 官方規則，不是我們系統的問題）

Render 的免費網站服務閒置一段時間後會自動休眠，下次有人打開網址時要等待約
30 秒到 1 分鐘喚醒，之後就會恢復正常速度；免費 PostgreSQL 資料庫則有 90 天效期限制，
到期前 Render 會寄信提醒。這對「測試網站」來說完全足夠，但如果之後要正式讓行政人員
每天使用，建議升級成付費方案（不會自動休眠、資料庫沒有效期限制）。

## Prisma Migration（資料庫異動歷史管理）

這次改成用 **Prisma Migration** 管理資料庫結構，不再使用 `prisma db push`：

- `prisma/migrations/20260101000000_init/migration.sql`：目前的第一筆（也是唯一一筆）
  migration，內容涵蓋 households / members / worship_records / activities（家戶管理）、
  users / finance_records / audit_logs（財務模組架構預留）、import_batches / import_rows
  （Excel 匯入）全部資料表，等同於目前 `schema.prisma` 的完整結構。這個檔案已經在沙盒內
  的 PostgreSQL 16 實際套用測試過（`psql -f migration.sql`），確認可以正確建立所有資料表、
  外鍵、預設值，且刪除家戶會正確連動刪除底下的成員/祭祀資料（cascade）。
- **正式環境（Render 或你的 Mac）套用 migration**：`npm run migrate:deploy`
  （對應 `prisma migrate deploy`，只會套用還沒套用過的 migration，不會互動詢問，
  適合自動化部署）
- **之後如果要修改資料表結構**（例如之後開發年度燈/普渡等模組時）：先改
  `prisma/schema.prisma`，然後在你自己的電腦上執行 `npm run migrate:dev -- --name 說明文字`
  （對應 `prisma migrate dev`），它會自動產生新的 migration 檔案並套用，記得把新產生的
  `prisma/migrations/` 資料夾一起放進 GitHub repo，這樣 Render 下次部署時才會套用到

## 農曆換算核對

第一次 `npm install` 完成後，建議執行：

```bash
npx tsx scripts/verify-lunar.ts
```

會列出幾組已知的農曆新年日期、生肖、閏月換算結果，對照輸出是否正確
（這台雲端開發環境沒有對外網路，無法先幫你實際跑過，請務必在 Mac 上核對一次）。

```

    page.tsx               搜尋首頁

    api/households/[id]/worship/   新增祭祀資料 API（POST）

    api/import/[batchId]/          查詢單一匯入批次結果（GET）
    api/import/[batchId]/commit/   Excel 匯入：確認匯入（POST，真正寫入資料庫）
    api/import/pending/            查詢所有「待確認」的匯入列（GET）

    household/

      AddMemberModal.tsx     新增家人表單
      EditHouseholdModal.tsx 修改家戶資料表單
      AddWorshipModal.tsx    新增祭祀資料表單
      formStyles.ts          三個表單共用的樣式
```

## 把專案打包給 ChatGPT（或其他人）檢查

在專案資料夾外層開啟終端機，執行：

```bash
cd 三玄宮ERP的上一層資料夾
zip -r 三玄宮ERP.zip 三玄宮ERP -x "*/node_modules/*" "*/.next/*" "*/.git/*"
```

這樣會排除掉 `node_modules`、`.next`、`.git` 這些不需要、體積又大的資料夾，
只打包原始程式碼，方便別人檢查或匯入其他工具。
