# V12.0 效能檢查紀錄

對應指令「十八、效能要求」：至少支援 1,000 家戶／5,000 家戶成員／50,000 筆活動及收款歷史紀錄。

⚠️ 本沙盒沒有這個規模的真實測試資料，也無法啟動 Next.js 開發伺服器量測真實回應時間（`node_modules` 不存在）。以下全部是**程式碼層級的查詢設計審查**，不是實際負載測試結果。

## 一、逐項對照

| 要求 | 設計方式 | 檔案 |
|---|---|---|
| 1. 名單使用分頁 | `skip`/`take`，資料庫層級分頁；`MAX_PAGE_SIZE=100` 上限，不會一次抓全部 | `devoteeList.ts` |
| 2. 時間軸使用批次載入 | 360 總覽一次性 `Promise.all` 抓齊該信眾的全部區塊資料（活動/供品/收款/收據/互動等），時間軸是對「已經抓到的資料」做記憶體內排序組裝，不是額外分批查詢——因為時間軸資料量是「單一信眾」的歷史，不是全宮資料，數量級遠小於全宮 50,000 筆 | `devotee360.ts` `buildTimeline()` |
| 3. 搜尋欄位有必要索引 | `Member.name`/`Household.id,phone,address,companyName,contactName` 等既有欄位本身在既有 schema 已有部分索引（`@@index([householdId])` 等）；`contains` 模糊搜尋在 PostgreSQL 預設無法用一般 B-tree 索引完全命中，仍為全表掃描风险，屬既知限制（見下方「三、已知效能限制」） | `devoteeList.ts`／`devoteeSearch.ts` |
| 4. 不得一次載入全部歷史資料 | 名單/搜尋皆有 `take` 上限（`PER_CATEGORY_LIMIT=10`／`MAX_PAGE_SIZE=100`）；360 總覽只抓「單一信眾」範圍的資料，不會抓全宮 | 全部新增查詢函式 |
| 5. 不得在前端做大量全資料比對 | 名單「本月生日」篩選的應用層精篩，只作用在「已經分頁後的固定筆數」（最多 100 筆）上，不是對全宮資料做全表比對 | `devoteeList.ts` |
| 6. 統計使用資料庫聚合 | 首頁 10 張統計卡片全部用 `prisma.count()`/`groupBy()`，沒有把全部信眾抓進應用層再用 JS 算數量 | `devoteeStats.ts` |
| 7. 避免 N+1 查詢 | 名單「最近參加活動」「最近收款」兩欄：先分頁取出該頁 `memberId`，再用 2 個批次 `groupBy` 查詢（`IN memberIds`），全部名單固定 3 支查詢（分頁本體＋2 個批次聚合），不隨頁面筆數增加而增加查詢次數 | `devoteeList.ts` |
| 8. 360 總覽可用 tabs 或 lazy loading | 前端 9 個分頁一次性抓取整包 `overview` 資料後在前端切換分頁顯示（非逐分頁各自呼叫 API）——這是本輪的取捨：換取「切換分頁不用重新打 API」的體驗，代價是初次載入需要一次抓齊全部區塊資料；由於單一信眾的歷史資料量遠小於全宮規模，判斷為可接受，但**沒有實測極端情況**（例如單一信眾有數百筆收款/活動紀錄時的載入時間） | `devotee-center/[memberId]/page.tsx` |
| 9. 搜尋輸入使用 debounce | 信眾名單與全宮搜尋兩個頁面皆用 350ms debounce，避免每個按鍵都打一次 API | `devotee-center/list/page.tsx`／`devotee-center/search/page.tsx` |
| 10. API 回應有合理分頁與筆數上限 | 名單 `pageSize` 上限 100；全宮搜尋每分類上限 10 筆（`PER_CATEGORY_LIMIT`）；首頁最近清單固定 10 筆（`RECENT_LIST_LIMIT`）、統計卡片內預覽固定 5 筆（`PREVIEW_LIMIT`） | `devoteeList.ts`／`devoteeSearch.ts`／`devoteeStats.ts` |

## 二、疑似重複信眾比對的效能取捨（誠實揭露）

`findDuplicateMatches()` 是 O(n²) 逐兩兩比對演算法，5,000 位信眾規模下是 2,500 萬次比對量級。程式碼註解已明確記錄這是「本來就是離線批次檢視頁面」的可接受設計判斷（不在指令「十八」的高頻互動效能要求範圍內），但**沒有實際跑過 5,000 筆資料量測真實耗時**，如果實務上發現這個頁面載入過慢，需要之後再優化為預先分組（例如先依電話/姓名做 Hash 分組，只在同一組內比對）而不是全域兩兩比對。

## 三、已知效能限制（未在本輪解決）

1. 姓名/電話/地址等欄位的 `contains`（模糊比對）查詢在 PostgreSQL 一般 B-tree 索引下無法完全命中索引，資料量成長後可能變慢；要真正優化需要 `pg_trgm` 擴充功能＋ GIN 索引，這是既有系統（V1.1 `/api/search`）本來就有的限制，本輪的新查詢延續同樣的既知限制，沒有引入新問題、也沒有解決舊問題。
2. 全宮搜尋一次查詢 9 個分類（9 支平行 Promise），資料庫連線池的併發負擔隨搜尋頻率增加，沒有實測高併發情境。
3. 360 總覽一次抓齊全部區塊資料的載入時間，隨單一信眾歷史資料量增加而增加，沒有實測資料量很大的信眾（例如同一人 20 年間有數百筆活動/收款紀錄）的載入時間。

## 四、結論

程式碼設計已依照指令「十八」逐項落實對應的效能考量（分頁/批次查詢/避免N+1/聚合統計/debounce/筆數上限），但**所有數字都是設計審查結論，不是實測結果**——本沙盒無法產生 1,000 家戶/5,000 成員/50,000 筆歷史紀錄規模的測試資料並量測真實回應時間，這一項在使用者的正式環境或有真實資料量的測試環境重新驗證前，維持「未完成實測」狀態。
