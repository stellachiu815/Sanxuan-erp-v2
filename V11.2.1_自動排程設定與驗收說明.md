# V11.2.1 自動備份排程設定與驗收說明

對應指令「十二、自動備份排程檢查」。

## 一、誠實結論（先講重點）

**目前無法確認外部排程服務是否已實際建立並持續觸發每日/每週/每月自動
備份。** 這不是程式碼問題，是因為：

1. Render 免費方案沒有內建排程（Cron Job）能力，一定要依賴「外部排程
   來源」定時對系統的 API 發送請求。
2. 這個外部排程來源（cron-job.org、GitHub Actions 排程、或升級 Render
   付費方案後的內建 Cron Job）**是否已經設定、設定得對不對，屬於部署
   維運層面的操作，不是這次交付的程式碼本身能夠自我證明的事**——系統
   自己沒有辦法「看見」有沒有一個外部服務按時呼叫它。

因此系統管理頁面依指令原文規定，會誠實顯示：

```
系統 API 已準備完成，但外部排程服務尚未確認。
```

**不會**顯示為「已啟用」，除非資料庫裡真的出現至少一筆
`isAutomatic: true` 的成功 BackupLog，才代表外部排程確實觸發過一次。

## 二、逐項確認（指令「十二」列出的 9 個檢查點）

| # | 檢查項目 | 狀態與說明 |
|---|---|---|
| 1 | 哪個外部排程服務負責觸發 | **尚未由使用者決定／建立**。render.yaml 已列出兩種選項：（a）升級 Render 付費方案使用內建 Cron Job；（b）維持免費方案，改用 cron-job.org 或 GitHub Actions 排程 workflow。這是部署維運層面的選擇，依工作原則「所有重大修改都必須等待確認」，不由我代為決定或假裝已建立。 |
| 2 | 排程服務是否已實際建立 | 未確認，需要使用者提供或親自到對應平台設定後告知 |
| 3 | 呼叫網址是否正確 | 網址格式為 `https://sanxuan-erp.onrender.com/api/system-center/backup/scheduled-trigger?type=daily`（`type` 可為 `daily`/`weekly`/`monthly`），程式碼端已就緒，但「排程服務實際設定的呼叫網址是否正確」需要使用者在設定排程服務時確認 |
| 4 | `BACKUP_CRON_SECRET` 是否正確帶入 | API 端要求 header `x-backup-cron-secret`，值需與 Render 環境變數 `BACKUP_CRON_SECRET` 完全一致；沒有設定這個環境變數時 API 一律回傳 503 拒絕執行（不會「沒設定就跳過檢查」）。實際排程服務端有沒有正確帶入這個 header，需要使用者確認 |
| 5 | 時區是否使用 `Asia/Taipei` | **已實作，程式邏輯已確認**：`scheduled-trigger/route.ts` 的 `taipeiDateKey()` 用 UTC+8 offset 換算判斷「今天（台北時間）」，用於「同一天已經成功執行過就不重複執行」的判斷 |
| 6 | 排程是否會因 Render 休眠或重新部署失效 | 已在 render.yaml 說明：外部排程服務發出的 HTTP 請求本身就會把 Render 免費方案休眠中的服務喚醒，這是選用外部排程（而非只依賴 Render 內部機制）的用意之一；「重新部署」不會影響排程本身（排程來源是外部的，不隨著 Render 服務重啟而消失），但需要使用者選定的排程平台本身要穩定運作 |
| 7 | 同一天重複觸發時是否有防重複機制 | **已實作並用邏輯驗證**：查詢同類型（daily/weekly/monthly）最近一筆 `isAutomatic:true` 且 `status:"SUCCESS"` 的 BackupLog，若其 `finishedAt` 換算台北時間跟現在同一天，直接回傳 `{ok:true, skipped:true, reason:"..."}`（HTTP 200，不視為錯誤），不會重複執行備份 |
| 8 | 失敗後是否有紀錄 | 已實作：`createBackup()` 失敗會把 BackupLog 標記為 FAILED、記錄 failedStage/errorCode/failureReason（已過濾敏感資訊，見資安驗收紀錄），可在備份紀錄頁面查詢 |
| 9 | 是否可由管理者手動測試一次排程觸發 | 可以：`scheduled-trigger` API 本身可以由管理者手動用 `curl` 或 Postman 帶正確的 `BACKUP_CRON_SECRET` 呼叫一次，效果跟排程服務自動觸發完全相同（同一支 API），可以用這個方式驗證 API 本身能不能正確執行，但這樣手動呼叫一次不能證明「排程服務會自動、按時、持續觸發」——這件事只有等外部排程真的運作一段時間、看到資料庫裡按預期頻率出現新的 BackupLog，才能確認 |

## 三、系統管理頁面顯示內容

`SystemSettingsScreen.tsx` 顯示：
- 每日／每週／每月備份排程狀態
- 下一次預定時間（`nextRunAt()`，依 Taipei 時區推算的理論排程時間，不代表外部排程服務真的會在那個時間觸發）
- 上一次執行時間、上一次執行結果
- `everConfirmedAutomatic`：**只有資料庫裡真的存在至少一筆該類型、`isAutomatic:true` 的 BackupLog 時才是 `true`**——這是「外部排程已確認運作」這個宣稱唯一的證據來源，不是靠猜測或假設

## 四、建議的下一步（給使用者參考，不代為決定）

1. 決定要採用哪一種外部排程來源（升級 Render 付費方案 vs. 外部免費排程服務），這屬於「重大修改，需要使用者確認」的範圍。
2. 設定完成後，先手動呼叫一次 `scheduled-trigger` API（帶正確的 secret）驗證 API 本身可以正確執行、寫入 BackupLog。
3. 讓排程實際運作一段時間（至少一天，觀察每日備份是否真的按時出現），之後系統管理頁面的「每日備份排程狀態」才會顯示為已確認運作。
