## 四、架構決策說明（誠實揭露，非隱性偷換需求）

### 4.1「左側主選單」的實作方式
指令「二」要求新增「左側主選單」項目，但這套系統目前整體是「單頁 HomeCard＋文字連結」架構（`src/app/page.tsx`），沒有任何既有模組使用真正貫穿全站的側邊選單。為了不違反工作原則「不得大幅更換全站風格」「不得重新設計已完成的功能」，本輪改用一張新的 `DevoteeCenterHomeCard`（比照 `OfferingHomeCard`/`CollectionHomeCard`/`SystemCenterHomeCard` 既有慣例）＋首頁文字連結做為入口，而不是新增側邊欄。信眾關係中心內部（`/devotee-center` 首頁）則用「功能磚塊」（tiles）取代真正的側邊選單，同樣是沿用既有系統管理中心（`/system-center`）的磚塊式導覽慣例。

### 4.2「360°總覽」分頁數量
指令「十九」建議 10 個分頁（1.總覽 2.時間軸 3.活動 4.收款 5.收據 6.供品 7.祭祀與祭改 8.家戶成員 9.互動紀錄 10.資料異動），本輪實作了前 9 個分頁；第 10 個「資料異動」（該信眾專屬的稽核歷史列表）**尚未實作為獨立分頁**——目前只有全宮共用的「最近資料異動」清單在首頁，沒有「單一信眾的完整異動歷史」查詢頁面。這是本輪的已知缺口，列在「十一、未完成/未測試項目清單」。

### 4.3 權限矩陣的欄位層級判斷
指令「十六」ADMIN 只能「修改一般信眾資料」，SUPER_ADMIN 沒有這個限定詞。本輪的 `updateDevoteeProfile()` 沒有做欄位層級的角色差異（`mobile`/`lineId`/`email`/`companyName`/`personalNote`/`isDisabled`/`disabledReason` 全部欄位 ADMIN 都能改），因為指令沒有進一步列出哪些延伸資料欄位算「敏感」。詳細判斷記錄在 `src/lib/permissions.ts` 對應段落的註解。

### 4.4 捐款統計類別涵蓋範圍
依 `src/lib/receivableAdapters.ts` 既有的 `RESERVED_SOURCE_NOTES`，這套系統目前「真正有收款資料」的來源只有四種：供品認捐（花果/福壽龜/其他供品）、普渡贊普、祭改、收款中心自建臨時應收。添油香、補庫、太歲燈類單獨收費、宮慶獨立應收目前完全沒有對應的登記或收費資料表——`getDonationStats()` 對這些類別一律顯示 0 並在 `note` 欄位附上文字說明，不虛構金額（詳見 `src/lib/devotee360.ts` 檔案開頭說明與「六、資料庫關聯說明」文件）。

### 4.5 農曆生日提醒的範圍限制
沿用既有 `src/lib/lunar.ts`（真正的 `lunar-javascript` 函式庫，非簡化演算法），但本輪刻意只做「本月農曆生日」（直接比對儲存的農曆月份，可靠）,不做「農曆生日換算成今年對應國曆日期」的跨年度換算（需要處理農曆新年跨年、閏月等邊界情況，本沙盒無法安裝套件、無法真正執行測試驗證這類邊界換算的正確性）。只登記農曆生日、沒有國曆生日的信眾，不會出現在「今日國曆生日」「未來七日國曆生日」名單，只會出現在「本月農曆生日」名單。

## 六、TypeScript 編譯結果

**未完成實測**。原因：`node_modules` 不存在（`npm install` 被登錄檔拒絕，見「一、誠實總結」），無法執行 `npx tsc --noEmit`。

本輪改用「逐一比對 Prisma schema 真實欄位/關聯名稱」的程式碼審查方式作為替代驗證手段：
- 開發過程中發現並修正一個真實型別錯誤：`devotee360.ts` 的 `getReceiptNumbersForSource(sourceType, sourceId)` 原本把 `sourceType` 宣告為 `string`，但實際傳入/使用於 Prisma `where: { sourceType }` 時對應的欄位型別是 `ReceivableSourceType` enum，`string` 無法指派給該 enum 型別，會是真實的 tsc 編譯錯誤——已改為 `import type { ReceivableSourceType } from "@prisma/client"` 並修正函式簽章。
- 逐一確認所有新查詢使用的 model／欄位／關聯名稱（`Member`/`Household`/`RitualRecord`/`PurificationEntry`/`UniversalSalvationEntry`/`OfferingClaim`/`PaymentTransaction`/`PaymentAllocation`/`ReceiptLine`/`Receipt`/`TempleEvent`）皆與 `prisma/schema.prisma` 實際定義逐字相符，包含必選/可選關聯是否需要 `is`/`isNot` 包裝（必選關聯直接巢狀查詢，可選關聯才用 `is`）。
- 這個方式**能抓出型別層級的欄位/關聯拼錯，但無法抓出所有可能的 tsc 錯誤**（例如某些泛型推論邊界情況），因此仍誠實列為「未完成實測」，不宣稱「TypeScript 檢查通過」。
