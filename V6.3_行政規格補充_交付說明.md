# V6.3「三玄宮行政規格補充」交付說明

本輪**沒有開發任何新 UI 或功能、沒有新增任何資料表 migration、沒有修改
正式資料庫**。目標是把年度燈燈種限制、犯太歲名單、信眾提醒、延用去年
報名等行政規則補齊到正式規格文件裡。

---

## 一、更新後規格文件

- **`docs/FINANCE_AND_ACTIVITY_SPEC.md`**（更新）：新增第十五節「年度燈與
  犯太歲規格」，收斂第五節「普渡規格」對外收款的隱藏規則（四個類別全部
  不對外顯示，不是只隱藏兩個），ER Diagram 與資料表用途一覽都補上
  `TaiSuiYearZodiac`，風險與尚待確認事項新增兩項（延用去年報名的批次
  複製 API、犯太歲名單的重複防呆）。
- **`docs/ADMINISTRATION_RULES.md`**（新增）：專門收錄「行政操作規則」
  （畫面怎麼表現、什麼情況要提醒、按鈕該叫什麼名字、流程先後順序），
  跟規格書的「資料架構」分開，方便你之後對照行政流程時不用在一堆 Schema

  （狀態/背景/決策/後果）記錄七個影響架構的重要決策，包含 V6.2 已經做過
  的四個決策（普渡兩層架構、流水帳沿用既有表、防重複計算設計、南北巡
  財務容器共用），以及 V6.3 新增的三個決策（犯太歲名單資料表設計、延用
  去年報名的批次 API 設計、普渡收款隱藏規則收斂）。

規格書第十一節、第十五節都已更新，新增一張表：`TaiSuiYearZodiac`

## 三、新增行政規則摘要

1. **年度燈只有光明燈、太歲燈兩種**，不預設其他燈種。
2. **犯太歲名單依年度設定**（例如 115 年：馬/鼠/雞/兔），管理者可自由
   修改，不寫死在程式。
3. **信眾提醒**：搜尋信眾後，今年犯太歲就黃色提醒「⚠ 本年度犯太歲，
   建議確認是否安奉太歲燈」，不犯太歲不提醒。
4. **「延用去年報名」正式流程**（按鈕名稱固定，不是「跟去年一樣」）：
   預設勾選光明燈/祭改/普渡，不自動勾太歲燈；如果今年犯太歲，另外獨立
   詢問是否新增太歲燈。
5. **普渡對外收款規則收斂**：只顯示功德主姓名，歷代祖先/個人乙位正魂/
   冤親債主/無緣子女四個類別全部不對外顯示（V6.2 原本只要求隱藏後兩者）。
6. 普渡固定順序（歷代祖先→個人乙位正魂→冤親債主→無緣子女→贊普→白米）
   在畫面/列印/報表/收費全部一致——沿用既有既定順序，本輪重新確認延伸到
   收費也要一致。
7. 贊普、白米沒有固定價格，每次手動輸入。
8. 南巡/北巡以車次為財務統計單位，名單只作乘車/住宿管理用途，不作為
   財務依據——沿用 V6.2 確認的規則。
9. 報名 ≠ 收款，再次確認——沿用 V6.2 確認的規則。

完整文字見 `docs/ADMINISTRATION_RULES.md`。

## 四、哪些地方未來需要修改資料表

- **新增 `TaiSuiYearZodiac`**：目前完全沒有這張表，年度犯太歲清單這個
  功能一定需要新增這張表才能實作。
- **`ActivityPriceItem`（年度燈）需要在 API/畫面邏輯層級限制只能有
  「光明燈」「太歲燈」兩個 `itemKey`**：不是資料庫層級的欄位限制，是
  應用邏輯的限制（年度燈的新增收費項目入口本身就不該讓管理者自訂燈種）。
- **「延用去年報名」需要一支新的批次複製 API**（見 `docs/ADR.md`
  ADR-0006）：不能只是疊加呼叫三次既有的單一活動複製邏輯，需要包在同一個
  資料庫交易裡。

## 五、哪些地方不用修改

- **普渡對外收款的隱藏規則收斂（四類別全部隱藏）完全不需要修改任何資料
  模型或程式碼**——因為 V6.2 的架構決策（對外收款畫面架構上天生就不會
  碰到 `UniversalSalvationEntry` 明細）已經涵蓋這個情況，只是文件用詞
  更新（見 `docs/ADR.md` ADR-0007）。
- **信眾提醒（黃色警示文字）不需要新的資料表**，純粹是「今年度 +
  這位信眾的生肖」是否存在於 `TaiSuiYearZodiac` 的查詢判斷。
- **普渡固定順序**沿用既有的 `ENTRY_CATEGORY_ORDER`/
  `universalSalvationEntryCategoryOrder`（V2.0 起就有），不需要修改。
- **贊普/白米金額不預設**沿用 V6.2 已經確認的 `ActivityPriceItem.
  unitPrice` 可為空的設計，不需要修改。
- **南巡/北巡以車次為單位、名單限制用途**沿用 V6.2 已經確認的
  `TourVehicle`/`TourRoomCharge` 設計，不需要修改。
- **報名 ≠ 收款**沿用 V6.2 已經確認的 `RegistrationGroup`/`Payment`/
  `FinanceRecord` 三層架構，不需要修改。
