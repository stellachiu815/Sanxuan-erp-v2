## 一、112 個既有 TypeScript 錯誤修正情況

延續前一輪已經確認的結論：這 112 個「非套件連鎖的真實型別錯誤」（用
獨立最小重現腳本驗證過，不是憑空歸類），全部逐一檢視，結果分成兩類：

### （A）真的是程式邏輯錯誤，本輪已修正的（不使用 any/@ts-ignore/
@ts-nocheck/型別斷言/關閉 strict）

1. **`notFound()` 之後的 null 窄化問題**（5 個檔案：`household/[id]/page.tsx`、
   `household/[id]/timeline/page.tsx`、`activities/[id]/page.tsx`、
   `purification/[yearId]/page.tsx`、`purification/[yearId]/print/page.tsx`）——
   在 `notFound()` 後面加一行 `throw new Error(...)`，讓 TypeScript 自己
   就能證明「往下走一定不是 null」，不依賴 `next/navigation` 的型別宣告
   （因為套件沒裝，`notFound()` 的回傳型別本來就無法被正確推斷）。

2. **`Buffer` 隱性 global 問題**（8 個檔案：7 支 API route + `smartImport.ts`）——
   改成明確 `import { Buffer } from "node:buffer";`，不依賴 Node.js 的
   ambient global 型別宣告。

3. **測試檔案 import 路徑多了 `.ts` 副檔名**（5 個測試檔）——移除多餘的
   `.ts`，改用專案其他測試檔案一致的寫法。

4. **缺少 `next-env.d.ts` 導致的 CSS import 型別問題**——新增
   `types/global.d.ts`，內容只有誠實的 `declare module "*.css";`，並在
   註解說明這是因為套件沒裝、無法自動產生 `next-env.d.ts` 的替代方案。

5. **`ActivityHomeScreen.tsx` 真正的 enum 比較邏輯錯誤**——原本
   `(searchParams.get("tab") as Tab) === "import"` 型別斷言後拿去跟一個
   不屬於 `Tab` 聯集的字面值比較，改成先比較原始字串再決定要不要轉型，
   邏輯本身修正，不是只消除警告。

6. **`RecycleBinScreen.tsx` 與 `VersionHistoryPanel.tsx` 各自維護一份
   過時的本地型別**，跟真正的來源（`src/lib/recycleBin.ts`、
   `src/lib/recordVersion.ts`）已經兜不起來——改成直接 `import type`
   引用權威來源。其中 `VersionHistoryPanel.tsx` 這個是**真正會造成生產
   環境問題的 bug**：舊版本地型別沒有涵蓋收據的 `PRINT`／`VOID`／
   `REISSUE` 三種異動動作，若不修正，收據的列印/作廢/換開異動紀錄在
   畫面上會顯示空白或 `undefined`。

7. **`activities/[id]/page.tsx` 的 `Date` 序列化問題**——修正 null 窄化
   之後才顯現出來的殘留型別不符：`checklist[].completedAt` 原本是
   `Date` 物件，直接傳給預期 `string | null` 的 Client Component，
   改成先 `.toISOString()` 再傳入。

## 二、使用者身分與權限機制

### 沿用既有資料模型，沒有另建第二套

檢查 `prisma/seed.ts` 發現專案已經有一個 `model User`（含 `role: Role`
enum：`SUPER_ADMIN`/`STAFF`/`FINANCE_CLERK`），是先前為未來財務模組
預留、但從來沒有被任何 API 或畫面真正查詢過的資料表。**本輪直接沿用
這個既有的 `User` 模型**，只在既有的 `Role` enum 上**新增**
`ADMIN`／`READONLY` 兩個值（純附加，不影響既有三個值的語意），沒有
另外建立第二套互不相通的人員資料——這是遵照指令「二」明確要求
（「若已有，請直接沿用；不要另外建立第二套互不相通的人員資料」）的
決定。

> 過程中曾經先建立一個獨立的 `StaffMember`／`StaffRoleValue` 模型
> （含完整 migration），後來讀 `seed.ts` 才發現既有的 `User` 模型，
> 已經整個復原刪除、改用現有模型——這個修正是自己在完成前發現並修正的，
> 沒有等到被指出才改。

角色對應（依指令「二」「至少要能辨識」的四種身分）：

| 角色 | 對應 Role enum | 說明 |
|---|---|---|
| 最高管理員 | `SUPER_ADMIN` | 全部收據操作皆開放，含號碼規則管理 |
| 管理員 | `ADMIN`（新增） | 授權管理人員，可作廢/換開/標記不需開立，但不能改號碼規則 |
| 一般工作人員 | `STAFF` | 日常操作（查看/開立/列印/補印/匯出/查看紀錄），不能核准類操作 |
| 唯讀人員 | `READONLY`（新增） | 只能查看/匯出/查看異動紀錄 |
| （既有預留，未開放） | `FINANCE_CLERK` | 收據相關權限全部沒有（`[]`），保留給未來財務模組 |

### 最小可用身分機制（明確不是真正的登入系統）

因為系統目前完全沒有帳號密碼/session 機制，本輪刻意**不**去做一套真正
的登入系統（超出本輪範圍，也沒有被指令要求），而是做「最小可用」的
身分辨識：

- 新增 `src/lib/operator.ts`：`resolveOperator(userId)` 依 `userId` 查
  `User` 資料表、確認 `isActive`，回傳 `{id, name, role}`；
  `assertReceiptPermissionForOperator(userId, action)` 進一步呼叫
  `canReceipt(role, action)` 做權限判斷，回傳 401（身分無效/已停用）
  或 403（角色沒有這項權限）。
- 新增 `src/app/api/system/users/route.ts`：`GET` 列出所有 `isActive`
  的使用者（只回傳 id/name/role，不含 email 等個資），提供畫面「選擇
  目前操作人員」使用。
- 前端新增 `src/lib/operatorClient.tsx`（`OperatorProvider`/`useOperator`）
  ＋ `src/components/system/OperatorBar.tsx`：使用者從下拉選單選出自己
  是誰，選擇結果存進瀏覽器 `localStorage`，之後每次呼叫收據相關 API
  都會帶上這個 `userId`。已經接進收據中心 5 個互動頁面（待開立收據、
  已開立收據／查詢、收據詳細、收據設定，以及收據中心首頁不需要因為
  沒有互動操作）。

**誠實揭露這個機制的限制**（這一點已經明確寫在 `operator.ts` 與
`api/system/users/route.ts` 的程式碼註解裡，不是事後才想到才講）：
`GET /api/system/users` 這支 API **刻意沒有權限檢查**——因為在選出
操作人員之前，根本不會有 `operatorUserId` 可以拿來檢查權限（雞生蛋
問題）。也就是說，任何打得到這個系統的人都能看到「有哪些人員、各自是
什麼角色」，並且可以在畫面上選擇假冒任何一個人（因為 `userId` 只是
瀏覽器端選擇、沒有密碼或任何憑證驗證使用者真的是本人）。這不是漏洞
被忽略，而是指令明確要求的「最小可用」機制的必然限制——**真正要防止
冒名，需要一套真正的登入機制（帳號密碼或 SSO＋伺服器端 session）**，
這是下一輪如果要做，需要另外明確確認的重大修改，本輪沒有自行擴大範圍
去做。

## 三、「標記不需開立」獨立權限

新增獨立的 `markNoReceiptRequired` 權限值（跟一般 `issue` 開立權限分開，
不會混在一起）。權限矩陣：只有 `SUPER_ADMIN`／`ADMIN` 擁有，
`STAFF`／`READONLY`／`FINANCE_CLERK` 都沒有。

實作內容：
- `MarkNoReceiptRequiredInput` 的 `reason` 與 `operatorUserId` 都改成
  **必填**（原本是選填的 `note`/`operatorName`），伺服器端會先驗證
  `reason.trim()` 非空、再驗證操作人身分與權限，兩者都通過才會寫入。
- 記錄操作人（`createdByName` 採用伺服器驗證過的真實姓名）、記錄操作
  時間（資料表既有的 `createdAt`）、原因寫進 `note` 欄位
  （`標記不需開立原因：xxx`）。
- 新增 `revokeNoReceiptRequired(receiptId, {reason, operatorUserId})`：
  允許授權人員撤銷標記，撤銷時同樣必填原因與操作人身分，撤銷後把收據
  狀態從 `NO_RECEIPT_REQUIRED` 轉為 `VOIDED`（沿用既有的作廢語意與
  異動紀錄機制），並呼叫 `recalculateAllocationReceiptStatus` 讓這筆
  收款重新回到待開立收據清單。

## 四、收據作廢／換開的核准控制

`voidReceipt()`／`reissueReceipt()` 的輸入都改成必填
`operatorUserId`／`approverUserId`（原本是選填的 `operatorName`／
`approvedByName` 自由文字欄位，沒有真正驗證）。新增私有輔助函式
`resolveVoidOrReissueParties()`，實際驗證：

1. 操作人（`operatorUserId`）身分有效、且擁有 `void`/`reissue` 權限。
2. 核准人（`approverUserId`）身分有效、且擁有
   `canApproveReceiptVoidOrReissue()`（只有 `SUPER_ADMIN`／`ADMIN`）。
3. **操作人與核准人不可為同一人**——除非核准人（同時也是操作人）是
   `SUPER_ADMIN`，且明確傳入 `isEmergencyOverride: true` 與非空的
   `emergencyReason`（緊急處理特殊原因），否則一律拒絕。

畫面上新增 `ReceiptVoidReissueDialog.tsx`（取代原本用 `window.prompt`
分別問「原因」「核准人姓名」的做法）：核准人一律從實際的使用者名單裡
選（存的是 `userId`，不是自由輸入的姓名），下拉選單會把操作人本人設為
`disabled`；只有目前操作人員角色是 `SUPER_ADMIN` 時才會顯示「緊急處理」
勾選框，勾選後改成要求填寫特殊原因、核准人自動視為操作人本人。這些
畫面端的檢查都只是體驗優化，**真正的強制驗證在伺服器端
`resolveVoidOrReissueParties()` 完成**，不是只在資料表預留欄位卻沒有
驗證。
