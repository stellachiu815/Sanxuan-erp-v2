### 2.6 財務防重複識別 financeSourceKey 三命名空間（對應指令八）

確認並補齊三個互不衝突的命名空間：

- `{sourceType}:{sourceId}:{paymentTransactionId}`——收入型，PaymentAllocation。
- `ADJUSTMENT:{allocationId}:{adjustmentId}`——退款/轉款/作廢，明確跟收入分開。
- `RECONCILIATION:{agentName}:{對帳紀錄id}`（本輪新增）——代收繳回是資金移轉，**不是**第二次收入，因此獨立一個命名空間，不會跟收入型 key 衝突，也不會被誤算進收入。

三方合併付款情境（供品認捐 2000 + 普渡贊普 1500 + 祭改 500 = 4000）實測驗證，三筆 `PaymentAllocation` 的 `financeSourceKey` 各自唯一：

```
OFFERING_CLAIM              | OC_FLORAL_1 | 2000.00 | OFFERING_CLAIM:OC_FLORAL_1:PT_3WAY_MERGED
UNIVERSAL_SALVATION_SPONSOR | USD_3WAY_1  | 1500.00 | UNIVERSAL_SALVATION_SPONSOR:USD_3WAY_1:PT_3WAY_MERGED
PURIFICATION_ENTRY          | PE_3WAY_1   |  500.00 | PURIFICATION_ENTRY:PE_3WAY_1:PT_3WAY_MERGED
```

一筆 `PaymentTransaction`（總額 4000）底下三筆 `PaymentAllocation`，加總正確等於 4000，三個不同來源的餘額分別正確扣減至 0。

### 2.7 重複送出防護機制（對應指令九）——本輪用真實併發資料庫驗證，不是模擬

機制設計：

- **前端**：`QuickPaymentScreen.tsx` 新增 `idempotencyKeyRef`，送出前產生一次性 UUID（`crypto.randomUUID()`，無此 API 時退回時間戳+亂數組合），送出後立即鎖定按鈕、顯示處理中；只要購物籃內容改變（加入/移除/改金額），這組 key 就會失效重新產生。
- **後端**：`createMergedPaymentTransaction()` 先用 `idempotencyKey` 做一次 `findUnique` 快速檢查（處理連續點兩下的常見情況）；即使兩個請求真的幾乎同時打進資料庫，最終仍由資料庫 `idempotencyKey` 的 `@unique` 唯一索引擋下，程式端 `catch` 區塊偵測 Prisma 的 `P2002`（唯一鍵衝突）錯誤碼，直接回傳先前已建立好的那筆交易，而不是報錯或建立第二筆。

**真實驗證方式（不是邏輯推演，是真的背景執行兩個 PostgreSQL session）**：

```
Session A：用 idempotencyKey=IDEM_TEST_1 建立收款，交易中停留 1.5 秒
Session B：幾乎同時，帶「同一組」idempotencyKey 再送一次
→ Session B 收到：ERROR: duplicate key value violates unique constraint
   "payment_transactions_idempotencyKey_key"
   DETAIL: Key ("idempotencyKey")=(IDEM_TEST_1) already exists.
   （實際等待了 1200.821 ms 才拿到這個結果，證明真的是在等 Session A 的鎖，不是巧合）
```

同樣的真實併發方法也驗證了另外兩個機制：

- **同一筆供品認捐被兩人同時搶收**：Session A 先扣款成功（2000/1000），Session B 的條件式 UPDATE 等了 1505.215 ms 之後拿到 `UPDATE 0`（0 rows affected），確認不會超收。
- **收款編號併發搶號**：Session A 取到 `currentValue=1` 並持有交易 1.5 秒，Session B 等了 1205.197 ms 之後才拿到 `currentValue=2`，序號正確遞增、不重複、不遺漏。

這三個測試都是真正背景執行的兩個獨立 `psql` 連線（用 `pg_sleep()` 撐開一段真實可觀測的鎖定時間、`\timing on` 量測實際等待毫秒數），比單一 session 依序執行 SQL 語句的驗證方式更能代表真實使用者連點兩下或多位行政人員同時收款的情境。
