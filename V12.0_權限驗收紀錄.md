## 二、指令逐句對照

| 指令原文 | 對照結果 |
|---|---|
| SUPER_ADMIN：查看/新增/修改/標籤管理/互動紀錄管理/關懷名單管理/查看完整收款與捐款統計/查看稽核紀錄 | 矩陣中 SUPER_ADMIN 擁有 `view`/`createProfile`/`updateProfile`/`manageTags`/`manageInteractions`/`manageCareList`/`viewFullFinancialStats`/`viewAuditLog`，逐項對應。 |
| ADMIN：查看/修改一般信眾資料/新增互動紀錄/套用既有標籤/查看活動及收款摘要 | 矩陣中 ADMIN 擁有 `view`/`updateProfile`/`createInteraction`/`applyTag`/`viewFinancialSummary`，逐項對應。 |
| ADMIN 不可：管理系統權限 | ADMIN 沒有 `manageTags`（標籤定義本身的管理），也沒有信眾關係中心以外的系統權限（`SystemAction` 矩陣完全獨立，未給予）。 |
| ADMIN 不可：執行資料合併 | `mergeDevotees` 對所有角色（含 SUPER_ADMIN）皆為 `false`，因為指令「十三」明確規定本次不開放合併功能。 |
| ADMIN 不可：刪除歷史財務資料 | 信眾關係中心本身**沒有任何** API 提供刪除 `PaymentTransaction`/`Receipt` 的功能（唯讀串接），這個限制是靠「根本沒有入口」達成，比權限矩陣擋下來更安全。 |
| ADMIN 不可：查看敏感系統設定 | ADMIN 沒有 `viewFullFinancialStats`／`viewAuditLog`，也沒有系統管理中心（`SystemAction`）的任何權限。 |
| READONLY：僅可查看，不可新增/修改/刪除/套用標籤/新增互動紀錄 | 矩陣中 READONLY 只有 `view`／`viewFinancialSummary`，沒有任何寫入類 action。 |
| 所有後端 API 都必須重新驗證登入身分與角色 | 全部 15 支 `route.ts` 皆呼叫 `assertDevoteePermissionForOperator(userId, action)`，該函式會重新查詢 `User` 資料表確認帳號存在且未停用，再依查到的角色檢查權限矩陣，不信任前端送出的角色字串。 |

## 三、94 項 tsx 執行斷言涵蓋範圍

1. 完整矩陣逐一角色 × 逐一 action 的預期結果（5 角色 × 12 action = 60 項基本斷言）。
2. READONLY 對全部寫入類 action（`createProfile`/`updateProfile`/`manageTags`/`applyTag`/`createInteraction`/`manageInteractions`/`manageCareList`）皆回傳 `false`。
3. ADMIN 對敏感類 action（`viewFullFinancialStats`/`viewAuditLog`/`manageTags`/`manageInteractions`/`manageCareList`/`createProfile`）皆回傳 `false`。
4. SUPER_ADMIN 對除 `mergeDevotees` 外的全部 action 皆回傳 `true`。
5. `mergeDevotees` 對全部 5 個角色皆回傳 `false`。
6. 未知角色字串（非 `Role` 型別的字串）安全回傳 `false`，不會拋出例外。

執行紀錄：
```
$ npx tsx v11_1_1_verification/verify_devotee_permissions.ts
... （94 行 PASS）
總結：94 項通過，0 項失敗。
```

## 四、STAFF／FINANCE_CLERK 選單隱藏

`canSeeDevoteeMenu(role)` 直接呼叫 `canDevotee(role, "view")`，STAFF／FINANCE_CLERK 兩者矩陣皆為空陣列（`[]`），因此這兩個角色：
- 前端 `DevoteeCenterGate` 會顯示「沒有查看權限」訊息，不渲染任何信眾資料。
- 就算繞過前端直接呼叫任何一支 API，伺服器端 `assertDevoteePermissionForOperator` 一樣會回傳 403（角色檢查在伺服器端重新執行，非只靠前端隱藏按鈕）。
