# V5.1「年度快照（Year Snapshot）」架構文件

本輪不新增資料表、不改 Schema、不開發 UI，只在既有的 V2.0 祭祀資料核心上
補齊「年度」這個維度的查詢能力，並把「每一年都是獨立資料、互不覆蓋」這個
從 V2.0 就存在的設計明確記錄下來。

---

## 一、每一年都是獨立資料（互不覆蓋）——已經是既有架構，本輪確認/記錄

`RitualRecord` 從 V2.0 建立時就用 `@@unique([householdId, year, activityType])`
當唯一鍵，也就是說「一戶、一年、一種祭典類型」永遠是獨立的一筆資料：

- 114 年、115 年、116 年的普渡資料，是三筆完全獨立的 `RitualRecord`（各自
  帶著自己的 `UniversalSalvationDetail` 與登記項目），彼此不會互相覆蓋。
- 這件事已經在 V2.0 部署驗收時實際測試過（複製 114→115 之後，114 年資料
  原封不動；再次嘗試複製到 115 年時系統正確回傳 409，不會覆蓋）。

本輪重新逐一檢查了普渡目前所有的「修改/刪除」函式（`src/lib/ritual.ts`），
確認每一支都符合「二、五」的要求：

| 函式 | 修改前如何鎖定範圍 |
|---|---|
| `updateUniversalSalvationDetail` | 先用 `(householdId, year, activityType)` 重新查一次 `RitualRecord`，查不到就回 404，不會誤改到別的年度 |
| `createUniversalSalvationEntry` | 同上，且新項目一律掛在「這次查到的那筆」`universalSalvationId` 底下 |
| `updateUniversalSalvationEntry` / `deleteUniversalSalvationEntry` | 除了鎖定 `(householdId, year)`，還會再確認 `entryId` 真的屬於「這個年度」查到的登記項目清單，才允許修改/刪除——就算有人猜到別的年度的 `entryId`，也不可能透過這個年度的網址誤改到 |
| `deleteUniversalSalvationRecord` | 同樣先用三個鍵重新查一次，才刪除 |
| `copyUniversalSalvationFromPreviousYear` / `createBlankUniversalSalvationRecord` | 目標年度已存在資料一律回傳 409，不覆蓋 |

**結論：「任何修改只修改目前年度、不得影響其他年度」這個要求，在資料存取層
（`src/lib/ritual.ts`）已經有實際保護，不是只靠網址上的年度參數自律。這次
沒有發現任何漏洞，也沒有需要修正的地方。**

---

## 二、任何祭典（普渡／年度燈／宮慶）未來都依附年度——已經是既有架構

`RitualRecord.activityType` 是 enum（`UNIVERSAL_SALVATION` / `ANNUAL_LANTERN` /
`TEMPLE_CELEBRATION` / `REPRINT`），跟 `year` 一起構成唯一鍵。這代表：

- 之後年度燈、宮慶要開發時，**不需要新建一張「年度」表**，只需要比照
  `UniversalSalvationDetail` 的模式，各自新增一張自己的明細表
  （例如 `AnnualLanternDetail`），用 1:1 掛在 `RitualRecord` 底下即可，
  年度隔離、防覆蓋這些機制全部自動繼承，不用重新設計。

## 三、ER Diagram

```
Household (家戶主檔，不分年度)
    │ 1
    │
    │ N
    ▼
RitualRecord (祭祀紀錄主檔，「一戶＋一年＋一種活動類型」唯一)
  - householdId
  - year                 ← 年度隔離的關鍵欄位
  - activityType         ← 普渡 / 年度燈 / 宮慶 / 補印
  - status（DRAFT/CONFIRMED/CANCELLED）
  - @@unique([householdId, year, activityType])
    │
    │ 1:1（activityType = UNIVERSAL_SALVATION 時）
    ▼
UniversalSalvationDetail (普渡明細)
    │ 1:N
    ▼
UniversalSalvationEntry (歷代祖先/個人乙位正魂/冤親債主/無緣子女)

    │ 1:1（activityType = ANNUAL_LANTERN 時，尚未開發）
    ▼
（未來）AnnualLanternDetail

    │ 1:1（activityType = TEMPLE_CELEBRATION 時，尚未開發）
    ▼
（未來）TempleCelebrationDetail

Member (成員主檔，不分年度) 1───N RitualRecord（memberId 可為空）
```

**關鍵設計**：`Household`／`Member` 是「不分年度」的主檔資料（人一直都在，不會
每年重複建立）；只有「這個人/這一戶今年做了什麼祭祀」才會落在
`RitualRecord` 上、按年度分開。這正是「年度快照」的意思——同一個家戶，
114/115/116 年可以各自有一份完全獨立的祭祀資料快照。

## 四、API 清單（本輪新增 2 支，其餘皆為既有 API，未變動）

### 新增：年度總覽（供「歷史年度瀏覽」下拉、「今年/去年/前年快速切換」使用）

```
GET /api/households/{id}/rituals/years
```

回傳範例：
```json
{
  "household": { "id": "F00009", "name": "王家" },
  "currentRitualYear": 115,
  "recentYears": [115, 114, 113],
  "years": [
    { "year": 115, "activities": [{ "activityType": "UNIVERSAL_SALVATION", "status": "DRAFT" }] },
    { "year": 114, "activities": [{ "activityType": "UNIVERSAL_SALVATION", "status": "CONFIRMED" }] }
  ]
}
```
- `recentYears`：今年／去年／前年，不管有沒有資料都列出（快速切換用）。
- `years`：只列「已經有資料」的年度，由新到舊（歷史年度瀏覽下拉選單用）；
  不限普渡，之後年度燈/宮慶只要開始寫入資料，會自動一起出現在這裡。

### 新增：年度快照（一次看到某年度、所有活動類型的資料）

```
GET /api/households/{id}/rituals/{year}
```

回傳範例：
```json
{
  "household": { "id": "F00009", "name": "王家" },
  "year": 115,
  "activities": {
    "UNIVERSAL_SALVATION": { "...": "跟 GET .../universal-salvation/115 回傳的 record 內容一樣" },
    "ANNUAL_LANTERN": null,
    "TEMPLE_CELEBRATION": null,
    "REPRINT": null
  }
}
```
- 目前只有 `UNIVERSAL_SALVATION` 有實際內容，其餘尚未開發的模組固定回傳
  `null`（不是錯誤，是「這個年度這個活動類型還沒有資料/這個模組還沒開發」）。
- 之後年度燈、宮慶的明細資料模型做好後，只需要在
  `getRitualYearSnapshot`（`src/lib/ritual.ts`）裡各補一段查詢，這支 API
  的網址跟既有欄位都不用改。

### 既有 API（本輪未變動，僅重新確認符合「年度隔離」原則）

| API | 用途 |
|---|---|
| `GET /api/households/{id}/rituals/universal-salvation/{year}` | 查詢某戶某年度完整普渡資料 |
| `PATCH /api/households/{id}/rituals/universal-salvation/{year}` | 修改該年度普渡明細 |
| `DELETE /api/households/{id}/rituals/universal-salvation/{year}` | 刪除該年度普渡登記 |
| `POST /api/households/{id}/rituals/universal-salvation` | 建立全新空白年度資料 |
| `POST /api/households/{id}/rituals/universal-salvation/copy-from-previous-year` | 複製前一年（或指定年度）資料成新一年 |
| `POST/PATCH/DELETE /api/households/{id}/rituals/universal-salvation/{year}/entries[/{entryId}]` | 新增/修改/刪除單一登記項目 |
| `GET /api/households/{id}/rituals/universal-salvation/{year}/print` | 列印資料格式（V4.0/V4.1 列印中心使用） |

## 五、資料流程

### 情境 A：行政人員想「瀏覽某戶的歷史年度」
1. 前端呼叫 `GET /rituals/years` 取得 `years`（有資料的年度清單）與
   `recentYears`（今年/去年/前年）。
2. 畫面用 `years` 畫出「歷史年度瀏覽」下拉選單；用 `recentYears` 畫出
   「今年／去年／前年」快速切換按鈕。
3. 行政人員選定一個年度後，前端呼叫 `GET /rituals/{year}`（年度快照）或
   `GET /rituals/universal-salvation/{year}`（只要普渡），取得該年度資料
   顯示；**不會、也不需要**同時查詢其他年度。

### 情境 B：行政人員修改「今年」的某一筆登記
1. 前端一律帶著「目前正在檢視的年度」呼叫 PATCH/POST/DELETE。
2. `src/lib/ritual.ts` 裡的每一支修改函式，都會先用
   `(householdId, year, activityType)` 這把唯一鍵重新查一次 `RitualRecord`，
   確認存在才動作；登記項目的修改/刪除還會再確認 `entryId` 屬於這個年度
   查到的清單。
3. 因為查詢條件一定包含 `year`，資料庫層面就不可能改到其他年度的資料——
   不是靠「前端只顯示這個年度」這種畫面約束，是資料存取層本身的保護。

### 情境 C：新年度開始，普渡登記還沒開始建立
1. 前端呼叫 `GET /rituals/years`，`recentYears` 裡的「今年」如果不在
   `years`（有資料的年度）清單中，代表今年還沒有任何資料。
2. 沿用既有的 V3.0 流程：問「今年跟去年一樣嗎？」→ 是 → 呼叫
   `copy-from-previous-year`；否 → 呼叫建立空白年度資料的 API。
3. 這兩支 API 都已經有「目標年度已存在則 409、不覆蓋」的保護。
