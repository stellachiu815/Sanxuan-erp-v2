# V11.2.1 更新前備份流程檢查報告

對應指令「十三、更新前備份流程檢查」。

## 一、現有 `render.yaml` 與 `scripts/preDeployBackup.ts` 執行順序檢查

`render.yaml` 的服務定義：

```yaml
buildCommand: npm install && npm run build
startCommand: npm run backup:before-update && npm run migrate:deploy && npm run start
```

逐項對照指令「十三、2. 正確順序」：

| 指令要求的順序 | 對應到哪一步 | 是否符合 |
|---|---|---|
| 取得舊版正式資料庫 | Render 每次部署都是連到同一個既有的 PostgreSQL 服務（`sanxuan-erp-db`），資料庫本身不會因為部署新程式碼而重建 | 符合 |
| 建立更新前備份 | `npm run backup:before-update`（`scripts/preDeployBackup.ts`），排在 `startCommand` 最前面，且此時 `buildCommand` 已完成但**尚未**執行任何 migration | 符合 |
| 確認備份成功 | `preDeployBackup.ts` 備份失敗會 `process.exit(1)`，shell 的 `&&` 語意讓後面的指令不會被執行 | 符合 |
| 執行 Prisma migration | `npm run migrate:deploy`，排在備份成功之後 | 符合 |
| 啟動新版系統 | `npm run start`，排在 migration 之後 | 符合 |

**檢查結論：現有順序本來就是正確的，本輪未發現需要調整順序的問題**，
只針對「重複執行」這個既有缺口（見下一節）做補強。

## 二、發現的既有問題：Render 免費方案的「重啟」也會重跑 startCommand

Render 免費方案的服務在以下情況都會重新執行一次 `startCommand`：
1. 推送新程式碼觸發真正的重新部署（build 階段會重新執行）。
2. 服務閒置一段時間後「休眠」，之後被喚醒。
3. 管理員在 Render 主控台手動按「Restart」。

V11.2 版本的 `preDeployBackup.ts` 對這三種情況一律無條件執行一次完整
備份——這代表單純的服務喚醒或手動重啟，也會在沒有任何程式碼變更的情況
下產生一份內容完全相同的 `Before_Update` 備份，長期下來會浪費 Google
Drive 空間、也讓備份紀錄畫面充滿無意義的重複項目。

## 三、修正方式：用 `RENDER_GIT_COMMIT` 辨識「真正部署」vs「單純重啟」

Render 官方會在每一次「真正的部署」（build 階段重新執行）時，把當次的
git commit SHA 寫入環境變數 `RENDER_GIT_COMMIT`；單純重啟／喚醒服務不會
重新跑 build 階段，這個值維持不變。

`scripts/preDeployBackup.ts` 新增比對邏輯：
1. 讀取目前的 `RENDER_GIT_COMMIT`。
2. 跟 `SystemSetting.lastBeforeUpdateCommit`（上一次成功執行更新前備份時
   記錄的 commit）比對。
3. **相同** → 判定為單純重啟／喚醒，跳過重複備份，印出說明訊息。
4. **不同（或第一次執行、沒有記錄）** → 判定為真正的新版本部署，執行
   備份，成功後更新 `lastBeforeUpdateCommit`。
5. **`RENDER_GIT_COMMIT` 完全不存在**（例如部署平台不是 Render，或方案/
   設定沒有提供這個變數）→ **無法可靠判斷**，依指令「十三、7. 若現有
   機制無法可靠判斷，先提出最安全的可行方案並實作，不得假裝已安全」，
   採用「不確定就備份」的保守預設（維持 V11.2 原本行為：每次都備份），
   並在 console 印出清楚的警告，說明這個環境無法辨識重啟與部署的差異
   ——這不是「假裝安全」，是明確告知使用者目前處於「無法辨識、所以選擇
   比較浪費空間但不會漏備份」的狀態。

## 四、這項修正「驗證了多少」——誠實揭露

- **邏輯本身**：已用程式碼審查方式確認條件分支完整（相同 commit 跳過、
  不同 commit 備份、無 commit 資訊時保守預設備份），三種情況都有對應的
  console 輸出訊息，方便日後從 Render 的部署 Log 追查發生了哪一種判斷。
- **實際在 Render 環境執行、確認 `RENDER_GIT_COMMIT` 真的存在且行為正確**：
  **未實測**。原因：這需要在真正的 Render 部署環境裡才能驗證（本沙盒不是
  Render 環境，無法取得真實的 `RENDER_GIT_COMMIT` 值），必須等這次的
  `sanxuan-erp_V11.2.1_完整專案.zip` 實際部署到 Render 之後，比對兩次
  部署之間備份紀錄的行為（第二次「單純重啟」是否真的沒有產生新的
  Before_Update 備份）才能確認。
- **第一次部署的例外情況**（尚未連結 Google Drive 時，`SKIP_PRE_DEPLOY_BACKUP=1`
  環境變數跳過此步驟）：這是 V11.2 既有設計，本輪未變更，僅在程式碼
  註解中重新確認其合理性——全新環境第一次部署時資料庫本來就是空的，
  沒有需要保護的既有資料，適合略過這一步。

## 五、備份失敗仍然禁止繼續更新（確認未被本輪修改破壞）

`preDeployBackup.ts` 主流程：備份失敗 → `console.error` 印出清楚原因 →
`process.exit(1)` → shell `&&` 語意讓 `migrate:deploy`／`start` 都不會
執行。本輪唯一新增的分支（判定為單純重啟而跳過備份）用的是 `return`
正常結束（`process.exit(0)` 語意），不會誤觸發失敗流程；`main().catch()`
仍然涵蓋所有未預期的例外狀況，同樣會 `process.exit(1)`。
