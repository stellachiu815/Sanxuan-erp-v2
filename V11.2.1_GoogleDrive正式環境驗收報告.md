## 六、立即備份正式驗收（對應指令「六」）——**本輪修正的關鍵缺口**

### 發現的落差

檢查 V11.2 既有程式碼時發現：`formatBackupFileName()` 產生的檔名是 `Backup_YYYY-MM-DD_HHMM.zip`（沒有秒數、前綴也不是指令要求的），跟指令「六」明確指定的
`SanxuanERP_Manual_YYYY-MM-DD_HHmmss.zip`
不符。前端畫面（`BackupCenterScreen.tsx`）的說明文字其實已經寫著正確格式，但後端實際產生的檔名並不是這個格式——這是一個文件與實作不一致的落差。

### 修正內容

新增 `formatManualBackupFileName()`，**只套用於 `MANUAL`（立即備份）類型**，完全依照指令文字（含秒數）：

```ts
function formatManualBackupFileName(now: Date): string {
  const y = now.getFullYear();
  const m = pad2(now.getMonth() + 1);
  const d = pad2(now.getDate());
  const hh = pad2(now.getHours());
  const mm = pad2(now.getMinutes());
  const ss = pad2(now.getSeconds());
  return `SanxuanERP_Manual_${y}-${m}-${d}_${hh}${mm}${ss}.zip`;
}
```

`DAILY`／`WEEKLY`／`MONTHLY`（自動排程備份）維持既有的 `Backup_YYYY-MM-DD_HHMM.zip` 格式不變——指令「六」的檔名格式只針對「立即備份」，沒有要求連動修改自動排程備份的既有命名規則，依工作原則「不重新設計已完成的功能」，這裡刻意不擴大修改範圍。

### 驗收清單逐項狀態

## 八、備份失敗處理補強（對應指令「八」）——**本輪發現並修正的資安缺口**

### 發現的實際風險（非理論性揣測）

`dumpDatabase()` 呼叫 `pg_dump` 時，把完整的 `DATABASE_URL`（可能含帳號密碼）當作指令的一個參數傳入；`restore.ts` 的 `pg_restore --dbname` 呼叫方式相同。Node.js 的 `child_process.execFile` 在指令失敗時，錯誤物件的 `.message` 依慣例會包含「Command failed: `<指令> <所有參數>`」整段內容——也就是說，如果 pg_dump／pg_restore 因為任何原因失敗，**這組連線字串（含密碼）會被整段存進 `BackupLog.failureReason`，並直接顯示在【系統管理中心 → 備份紀錄】畫面上**，違反指令「十五、6. 錯誤訊息不得包含完整 DATABASE_URL」與「八、4. 不得在前端洩漏...資料庫連線密碼」。

這不是本輪新增的邏輯造成的問題，是 V11.2 既有程式碼裡原本就存在、這一輪重新檢查程式碼時才發現的既有缺口。

### 修正方式

新增 `redactSensitive()`（`src/lib/backupErrorClassifier.ts`），用正規表示式偵測並隱藏任何 `postgres://`／`postgresql://`／`mysql://`／`mongodb://` 開頭的連線字串片段，換成 `[已隱藏：資料庫連線字串]`，其餘錯誤說明文字（例如 pg_dump 實際印出的錯誤原因）維持不變。套用位置：

- `backup.ts`：`createBackup()` 主要 catch 區塊、保留政策清理失敗的 catch 區塊。
- `restore.ts`：還原失敗的 catch 區塊。
- `health/route.ts`：資料庫健康檢查失敗訊息。

**已用 `tsx` 真實執行驗證**（見 `V11.2.1_備份完整性驗證紀錄.txt`），用真實的 pg_dump/pg_restore 失敗訊息格式（含假的測試用密碼字串）餵給 `redactSensitive()`，確認密碼字串被完全移除、其餘錯誤說明文字被完整保留（沒有誤刪不該刪的內容）。

### 十一種錯誤分類

`classifyBackupError()` 涵蓋指令「八」列出的全部十一種類別（含未知系統錯誤），並且用「執行階段」（`stage`）輔助判斷 `ENOENT`（找不到指令）到底是 pg_dump 還是 zip 找不到，避免 Node 對「找不到指令」一律回傳同一種錯誤代碼造成誤判。**15 項分類測試 + 3 項 redactSensitive 測試，共 18 項全部通過**（真實執行 `tsx`，不是模擬）。
