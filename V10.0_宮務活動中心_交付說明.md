## 一、最重要的架構決定：祭改整個搬進通用架構

規格第一句話是：「本次開發完成後，所有宮務活動（普渡、祭改、光明燈、
太歲燈、全家燈、補庫、宮慶及未來新增活動）皆必須使用同一套流程，不得
各自重新開發。」

但上一輪（V9.0）祭改是建立在自己獨立的三張表
（`PurificationYear`／`PurificationRegistration`／`PurificationPrintBatch`）
上，跟普渡本來就在用的通用 `RitualRecord` 架構是分開的兩套系統。這是
一個會影響資料庫結構、需要重新設計的真正架構分岔，所以這一輪開始前
先跟你確認方向，你選擇了：**把祭改整個搬進通用 Activity/RitualRecord
架構**（而不是保留祭改原本的表、只加一個新的建立入口）。這一輪就是照
這個方向做的，實際變動：

- 新增 `TempleEvent`（活動主檔，取代原本 `PurificationYear` 的角色，
  一般化成「活動類型＋年度」都能用，不是祭改專屬）。
- 既有的 `RitualRecord`（V2.0 就有，普渡一直在用）新增一個可為空的
  `templeEventId` 欄位，指回這個活動主檔；普渡的既有資料完全不受影響
  （這個欄位對普渡永遠是空的也沒關係）。
- 祭改的報名資料改名為 `PurificationEntry`，掛在 `RitualRecord` 底下
  （`activityType = PURIFICATION`），不再自己存 `householdId`——跟普渡
  的登記資料完全走同一套「家戶＋年度＋活動類型」的容器。
- 原本 `PurificationYear`／`PurificationRegistration`／
  `PurificationPrintBatch` 三張表整個移除，資料模型上不再有「祭改
  專屬」的年度／報名／列印批次容器，統一用 `TempleEvent`／
  `RitualRecord`／`TempleEventPrintBatch`。
- **祭改所有業務規則完全不變**（編號跳號規則、貼紙欄位、性別建生瑞生、
  虛歲計算、版面最佳化、列印批次分離……），只是資料掛載的位置從專屬表
  改成通用表，`src/lib/purification.ts` 每一個對外函式的名稱／參數／
  回傳格式都刻意維持跟上一輪一模一樣，所以既有的 11 支祭改 API 路由
  跟前端畫面**完全不需要修改**（只有一個必要的行為變更，見下方「誠實
  限制」第 4 點）。

這個決定的結果就是：這一輪之後，祭改、普渡、以及所有新增的活動類型
（光明燈／太歲燈／全家燈／補庫／宮慶／其他），資料庫底層真的是同一套
表格結構、同一套建立/複製/匯入流程，不是「看起來統一、其實各自維護一份」。

## 二、活動精靈：唯一的建立入口

首頁新增「＋建立宮務活動」——這是**唯一**建立任何宮務活動的入口，固定
三步驟，任何活動類型都走同一個精靈，不會有個別活動類型自己的建立畫面：

- **Step1 選擇活動類型**：中元普渡／光明燈／太歲燈／全家燈／祭改／
  補庫／宮慶／其他。
- **Step2 基本資料**：活動名稱（自動依年度＋活動類型組字，可手動覆蓋）／
  年度／農曆日期／國曆日期／狀態（籌備中／進行中／已結案／已取消）／
  備註。
- **Step3 建立方式**（四選一）：
  1. 空白建立。
  2. 沿用去年資料——三個勾選項：去年參加名單／去年設定（禁用編號等）／
     去年收費紀錄，各自獨立勾選，不是全有全無。
  3. Excel/CSV 智慧匯入（見下方第三節）。
  4. 直接在 ERP 輸入。

建立完成後，系統會**自動**（不需要人工一步一步操作）備好：報名/參加
資料容器、信眾/家戶關聯、收款欄位、支出紀錄、列印佇列、功德名錄（普渡
專屬）、活動首頁、統計數字、Checklist——符合規格「不得要求逐步設定」
的要求。

---

## 三、Excel/CSV 智慧匯入

- 上傳檔案後，系統先**分析**再顯示結果，不會直接匯入：新增筆數／更新
  筆數／重複筆數／資料不全筆數／需要人工確認筆數，一次全部列出來，
  行政人員看過摘要再決定要不要真的執行。
- 欄位對應用「智慧辨識」：常見欄位名稱（戶號、姓名、電話、地址、金額……）
  自動猜對應的 ERP 欄位；猜不到的欄位人工手動選一次之後，**這個對應會
  永久記住**（存在 `ImportFieldMapping`，依「匯入種類」分開記憶），下次
  同一種匯入、同樣的 Excel 欄位名稱，不用重新設定。
- 支援手動覆蓋自動辨識結果、支援跳過某一欄不匯入。
- 重複資料偵測：同一位信眾/同一戶在同一個活動年度已經有資料時，會列在
  「重複」清單，讓行政人員選擇「更新」「跳過」或「合併」，不會靜默
  覆蓋或靜默略過。

## 四、自動化建立（不是逐步設定）

活動精靈完成後，系統自動（無需人工介入）建立：

- 報名／參加者資料容器（依活動類型自動選對容器：普渡用
  `UniversalSalvationDetail`，祭改用 `PurificationEntry`，其他類型用
  通用名單）。
- 信眾／家戶關聯（沿用去年資料時自動帶入去年的家戶關聯）。
- 收款欄位（依活動類型預設）。
- 支出紀錄容器（`TempleEventExpense`，活動建立當下就備好，不需要另外
  啟用）。
- 列印佇列（`TempleEventPrintBatch`，第一次列印前是空的，不會憑空產生
  假資料）。
- 功德名錄（普渡專屬，沿用既有的 `UniversalSalvationEntryCategory`）。
- 活動首頁（見第五節）。
- 統計數字（人數／收款／支出，活動首頁即時計算）。

## 十一、誠實說明：這一輪的限制

### 1. 沙盒環境沒有資料庫，這一輪最大量的程式碼無法真正執行測試

跟前幾輪一樣，這個沙盒沒有 PostgreSQL、無法連線 GitHub/Render、無法
`npm install`（`node_modules` 完全是空的，`@prisma/client`／`xlsx` 都
無法真的載入執行）。這一輪新增的程式碼中：

- **可以誠實驗證的部分**：3 個新抽出的純邏輯模組（活動名稱組字、
  Checklist 預設項目、Excel 欄位智慧辨識）刻意設計成不依賴 Prisma／
  xlsx 執行期，16 個測試是真的跑過的。
- **無法在沙盒裡執行驗證的部分**：`templeEvents.ts`／`templates.ts`／
  `smartImport.ts`／重寫後的 `purification.ts`，以及全部新增的 API
  路由——這些本質上是資料庫協調邏輯（建立/查詢/交易），只做到了
  「靜態檢查通過」：全專案 144 個檔案括號配對檢查、Prisma 模型名稱/
  複合唯一鍵名稱逐一比對 schema、所有 import 路徑逐一驗證可解析、
  App Router 動態路由命名衝突逐一檢查，**這些都做了，也都通過**，但
  跟 V9.0 一樣，「靜態檢查通過」不等於「真的對著資料庫跑過一次」，
  這需要部署到有真實資料庫的環境才能做最後驗證。

### 2. 只有祭改跟普渡有完整的專屬業務規則，其他活動類型目前是通用框架

光明燈／太歲燈／全家燈／補庫／宮慶／其他這 6 種活動類型，這一輪只有
規格裡沒有再細分規則的部分（活動精靈、Excel匯入、Checklist、支出、
通用名單、列印模板分類）——因為你這次給的規格本身沒有像祭改（十六節）
那樣針對這幾種活動給出專屬的業務規則（例如光明燈燈牌實際版面尺寸、
太歲燈疏文內容格式）。這幾種活動目前可以正常建立、匯入名單、列印
（用官方列印模板分類，但實際版面內容要等對應的範本檔案送達後才能真正
產生正確格式），統計/Checklist/支出都能正常使用，只是還沒有專屬的
業務邏輯——這是刻意的範圍控制（規格沒給的細節不自行增加），不是漏做。

### 3. Excel 智慧匯入的「分析」是無狀態的，跟家戶匯入的既有流程不完全一樣

既有的家戶資料匯入（`src/app/api/import/preview/route.ts`）會把預覽
結果存成 `ImportBatch`／`ImportRow`，之後「確認匯入」是讀這批已存的
預覽紀錄。這一輪的宮務活動 Excel 智慧匯入為了讓「分析」跟「確認匯入」
兩步驟能各自獨立呼叫（畫面上可以先分析看摘要，過一陣子才回來確認），
選擇了**無狀態**設計：確認匯入時重新解析一次使用者上傳的檔案，而不是
去讀一筆之前存的批次紀錄。這代表：如果分析完之後、確認匯入之前使用者
換了一個不同的檔案，系統不會知道「這已經不是原本分析過的那個檔案」——
畫面上會提醒使用者「請不要在確認匯入前更換檔案」，但後端目前沒有做
檔案內容雜湊比對來強制擋下。如果你希望比照家戶匯入一樣存成正式批次
紀錄（`ImportBatch.importKind` 欄位已經預留好了），下一輪可以直接加上。

### 4. 祭改「臨時報名者」現在必須綁定家戶（行為變更，非新增流程）

上一輪（V9.0）祭改的「臨時報名者」（`isTemporaryName=true`，還沒有
信眾主資料的人）原本可以完全不綁定任何家戶。這一輪祭改搬進通用
`RitualRecord` 架構後，`RitualRecord.householdId` 是必填欄位（這是
V2.0 就定案、普渡一直在遵守的既有規則），所以臨時報名者現在**也必須**
選一個家戶。已經在「臨時報名」的表單裡加上必填的家戶搜尋欄位（沿用
既有的 `/api/search`，本來就支援「只有家戶、還沒有信眾」的搜尋結果），
沒有家戶就無法完成報名，畫面會提示「請選擇這位報名者所屬的家戶」。
這是配合統一架構必須接受的行為變更，這裡誠實列出來，不是自行新增的
流程規定。

### 5. 模板中心的「上傳版本」目前只存檔名/備註，沒有真正的檔案儲存

沙盒環境無法真正儲存二進位檔案（沒有檔案儲存服務可以接），`fileUrl`
欄位這一輪先留著，上傳版本目前只會記錄檔名／版本標籤／備註／是否啟用
這些中繼資料，還沒有真的把上傳的檔案內容存起來。等正式環境接上檔案
儲存服務（例如 S3 或本機檔案系統）後，只需要在
`src/app/api/templates/[id]/versions/route.ts` 補上真正的檔案寫入，
資料模型本身不需要更動。

### 6. 財務模組（支出/收款整合）沒有跟著這一輪一起做

`TempleEventExpense` 是這個活動中心模組**自己內部**的簡易支出記錄
（分類／金額／日期／說明），刻意跟正式的財務流水帳模組
（`FinanceRecord`，目前 schema 裡已經定義、但完全沒有畫面/API）分開——
確認過專案裡目前沒有任何 `FinanceRecord` 相關的路由或畫面（`git log`
顯示這個專案從 V2.0 到這一輪都還沒有正式提交紀錄，倉庫根目錄有幾份
「V7.0_FinanceCore」「V7.1_CollectionCenter」的交付文件/壓縮檔，但對應
的程式碼並不在目前的工作目錄裡）。這一輪沒有順便把財務模組也做掉，
避免無中生有擴充一個還沒排入開發計畫的模組。

## 十二、資料庫變更

- 移除 3 張表：`purification_years`／`purification_registrations`／
  `purification_print_batches`（改由下方新表取代）。
- 新增 9 張表：`temple_events`／`temple_event_banned_numbers`／
  `temple_event_print_batches`／`temple_event_checklist_items`／
  `temple_event_expenses`／`purification_entries`／
  `template_definitions`／`template_versions`／`import_field_mappings`。
- `ritual_records` 新增一個可為空的關聯欄位 `templeEventId`，不影響
  既有的普渡資料。
- `import_batches` 新增 `importKind`（預設 `"HOUSEHOLD"`，既有家戶匯入
  流程完全不受影響）與 `templeEventId` 兩個欄位。
- `ActivityType` 列舉新增 6 個值：`PURIFICATION`／`GUANGMING_LANTERN`／
  `TAISUI_LANTERN`／`FAMILY_LANTERN`／`STORAGE_REPAYMENT`／`OTHER`。
- `purification_banned_numbers`（全域禁用號碼）與
