## 一、這次做了什麼（一句話版本）

把「歷代祖先／個人乙位正魂／冤親債主／無緣子女」底下簡化的「是否有寶袋＋寶袋數量」欄位，換成一套完整、每個寶袋（或牌位/疏文/燈牌/其他列印項目）都是獨立一筆資料的管理機制，涵蓋資料庫、後端邏輯、API、前端畫面、權限規則、Excel 匯入、列印整合、版本紀錄與自動測試。

## 二、需求 16 個章節逐項對照

### 1～3. 附加列印項目資料模型（AdditionalPrintItem）

新增 `AdditionalPrintItem` 資料表，欄位完全對照需求「二」的清單：`id`／`activityId`／`ritualRecordId`／`sourceEntryId`（+`sourceEntryType`）／`householdId`／`memberId`／`itemType`／`printName`／`quantity`／`templateId`／`status`／`note`／`isExtra`／`createdBy`／`createdAt`／`updatedAt`／`deletedAt`（+`deletedByName`）。

`itemType` 對照需求：寶袋(POCKET)／牌位(TABLET)／疏文(PETITION)／燈牌(LANTERN_TABLET)／其他列印項目(OTHER)。`status` 對照需求：待確認(PENDING_CONFIRMATION)／待列印(PENDING_PRINT)／已列印(PRINTED)／取消(CANCELLED)。

每一個寶袋都是獨立一列 `AdditionalPrintItem`，不會用一個數字或布林值表示——需求「三」範例（王姓歷代祖先 → 祖先牌位×1 + 寶袋:王姓歷代祖先×1 + 額外寶袋:王姓歷代祖先×1 + 額外寶袋:王某某×1）對應到資料庫就是 4 筆獨立的 `AdditionalPrintItem`，見 `tests/additionalPrintItemRules.test.ts` 案例 2／4／6。

`sourceEntryId`／`sourceEntryType` 刻意不是強制外鍵，是延續 V8.0 `RecordVersion.entityType`/`entityId` 建立的「多型參照」慣例——目前只有 `UNIVERSAL_SALVATION_ENTRY` 一種來源型別，之後如果要讓其他資料（例如未來的其他祭祀類型）也能掛附加列印項目，不需要重新設計這張表。

### 4. 「寶袋與附加列印」畫面

新增 `AdditionalPrintItemsPanel` 元件，嵌在既有的 `EntryRow`（每一筆歷代祖先／個人乙位正魂／冤親債主／無緣子女登記項目）底下，點擊「寶袋與附加列印」按鈕展開。畫面顯示每一筆項目的：類型、列印名稱、數量、預設／額外標籤、狀態、已列印進度（例如「已列印 2/2 份」）、備註，並提供編輯／取消／恢復／列印／補印／（取消後）永久刪除等操作按鈕——完全對照需求「十二」的王姓歷代祖先範例畫面配置，不會把多筆寶袋合併顯示。

「＋新增寶袋」按鈕展開新增表單，欄位：項目類型、是否沿用原祭祀名稱（勾選時顯示原名稱、取消勾選才出現自訂名稱輸入框）、數量、是否為額外寶袋、備註——對照需求「四」。

### 5. 命名規則

`resolvePrintName(usesSourceName, sourceName, customName)`（`src/lib/additionalPrintItemRules.ts`）：`usesSourceName=true` 回傳原祭祀名稱，`=false` 回傳自訂名稱。自訂名稱只在建立當下算一次、存成 `AdditionalPrintItem.printName` 獨立欄位，不會回寫或反查原始的 `UniversalSalvationEntry.displayName`——之後原祭祀名稱被修改，已建立的列印名稱完全不受影響。對應測試：`tests/additionalPrintItemRules.test.ts` 案例 3／19（含反例）。

### 6. 數量規則

`expandFixedColumnRow()` 展開 Excel 匯入方式一的固定欄位時，即使兩個額外寶袋名稱完全相同，也會展開成兩筆獨立的 `ExpandedBagSpec`，不會合併成「數量 2」；不同名稱的寶袋（例如「王姓歷代祖先×1」+「王某某×1」）同樣展開成兩筆獨立資料。對應測試：案例 2／4／6／10。

### 7. 預設／額外標籤

`AdditionalPrintItem.isExtra`（布林值）+ 前端 `PrintItemRow` 元件用不同底色的標籤（額外＝粉色系、預設＝霧藍色系）清楚區分。活動精靈自動建立的預設寶袋 `isExtra=false`；人工新增／Excel 匯入的額外寶袋預設 `isExtra=true`（可在新增表單裡取消勾選，允許管理者手動補建「預設」寶袋，例如補登記漏建的情況）。

### 8. Excel／CSV 匯入

實作**方式二（明細工作表）**：欄位「家戶編號／報名人／原祭祀類型／原祭祀名稱／附加項目類型／列印名稱／數量／預設或額外／備註」，見 `src/lib/importFieldSuggestion.ts` 的 `ADDITIONAL_PRINT_ITEM_FIELDS`、`src/lib/templates.ts` 的 `ADDITIONAL_PRINT_ITEM_IMPORT` 空白範本。

`analyzeAdditionalPrintItemImport()`（`src/lib/additionalPrintItems.ts`）会依「家戶編號 + 年度 + 原祭祀類型 + 原祭祀名稱」逐列比對是否找得到對應的來源登記項目：找不到的列一律標記為 `NEEDS_CONFIRMATION`（待確認），**不會直接匯入**；欄位缺漏標記為 `MISSING_DATA`；名稱/數量/預設額外完全相同的既有資料標記為 `DUPLICATE`；其餘標記為 `NEW`。前端 `PrintItemsCenter` 的匯入面板會先顯示這份分析預覽（含每一列的狀態與說明），使用者確認後才呼叫 `commitAdditionalPrintItemImport()` 真正寫入。

**方式一（固定欄位）**：純規則邏輯（`expandFixedColumnRow()`）已完成並有完整測試覆蓋，但這次沒有另外接一支對應的 API／匯入 UI（誠實揭露於「已知限制」）——如果之後要開放這個匯入方式，直接呼叫這支既有的展開函式即可，不需要重新設計規則。

### 9. 普渡列印中心

新增跨家戶的「普渡列印中心」畫面（`/universal-salvation/[year]/print-center`，對應元件 `src/components/universal-salvation/PrintItemsCenter.tsx`），可依活動／家戶／報名人／原祭祀類型／原祭祀名稱／寶袋列印名稱／預設額外／狀態篩選，畫面上方有預設／額外／總數／待列印／已列印／已取消六個統計標籤。批次列印支援：全部列印／只印預設寶袋／只印額外寶袋／勾選特定項目列印（涵蓋「指定寶袋補印」「指定家戶列印」「指定名稱搜尋」——後兩者透過先套用篩選、再用清單勾選的方式達成，不需要額外的 UI 分頁）。

### 10. 列印紀錄

`AdditionalPrintItem` 每一筆獨立追蹤 `isPrinted`／`printedAt`／`printedByName`／`printBatchId`（+ 批次本身的 `templateVersionId`）／`printedQuantity`／`reprintCount`。`quantity=2` 時，`printedQuantity` 會準確反映「印過幾份」，`computePrintProgress()` 判斷是否已完整列印兩份。`applyPrintAction()`（純函式）明確保證：不論列印幾次，`quantity`（原始需求數量）永遠不會被修改，補印只會累加 `printedQuantity`／`reprintCount`。對應測試：案例 5／13／14。

### 11. 收款欄位預留

`isChargeable`／`unitPrice`／`quantity`／`subtotal`／`isPaid`／`paymentId` 全部已建立在資料表上。`computeAdditionalPrintItemFee()`：不收費時 `subtotal` 固定是 `0`；收費但單價尚未設定時 `subtotal` 是 `null`（不會假裝是 0，逼著畫面上要求人工補齊單價）；收費且有單價時 `subtotal = unitPrice × quantity`。對應測試：案例 18（三種情境）。

### 12. 畫面範例對照

`PrintItemRow` 元件的呈現完全對照需求給的王姓歷代祖先範例：每一筆寶袋獨立一行、各自的預設／額外標籤、獨立的編輯／取消／列印／補印按鈕，絕對不會合併顯示成「寶袋數量3」。

### 13～14. 刪除保護與權限

「取消」（`cancelAdditionalPrintItem`）：`status` 改為 `CANCELLED`，保留全部歷史與列印紀錄，從待列印清單移除，可用「恢復」（`restoreCancelledAdditionalPrintItem`）復原。

「永久刪除」是獨立的第二層機制：只有已經是「取消」狀態的項目，才能呼叫 `moveAdditionalPrintItemToRecycleBin()` 移入回收區（`deletedAt` 欄位），前端要求「雙重確認」對話框；真正的硬刪除沿用 V8.0 既有的 30 天保留期限機制（`src/lib/recycleBin.ts`，已擴充 `AdditionalPrintItemEntityType` 支援），超過期限才能從 `/api/recycle-bin/purge` 永久刪除。

權限規則已定義在 `src/lib/permissions.ts` 的 `AdditionalPrintItemAction`／`ADDITIONAL_PRINT_ITEM_PERMISSIONS`：SUPER_ADMIN 開放全部操作；STAFF（管理員/櫃台）只開放新增額外寶袋／修改尚未列印的資料／列印／補印。**⚠️ 跟這個專案所有其他權限規則一樣，系統目前沒有登入/session 機制，這些規則還沒辦法在後端真正擋下**，是已知、一致的限制（見下方「已知限制」）。

編輯已列印過的項目時，`updateAdditionalPrintItem()` 會回傳 `alreadyPrintedWarning: true`，前端顯示警告訊息；不論是否已列印，每次修改都會寫入 `RecordVersion`（版本紀錄）。

### 15. 活動精靈整合

`getAdditionalPrintItemActivitySummary(templeEventId)` 提供活動摘要（預設寶袋數量／額外寶袋數量／寶袋總數／待列印數量／已列印數量），API：`GET /api/temple-events/[id]/print-items-summary`。`checklistDefaults.ts` 的普渡（UNIVERSAL_SALVATION）預設待辦清單新增三項：「確認額外寶袋資料」「列印預設寶袋」「列印額外寶袋」。

**已知限制**：活動精靈目前建立普渡活動時，還沒有接上「自動依規則建立預設寶袋」這段邏輯（例如「每一筆歷代祖先自動建一個同名寶袋」的規則）——資料模型、API（`POST .../print-items`）都已經支援手動或匯入建立，只是活動精靈的自動化這一步這次沒有實作，見下方「已知限制」。

### 16. 自動測試

`tests/additionalPrintItemRules.test.ts`：26 個測試，逐一對應需求列出的 20 個案例（案例 1～20，含反例）。`tests/importFieldSuggestion.test.ts`：新增 2 個測試涵蓋新的匯入欄位。全部測試（連同既有的 79 個）合計 **107 個測試，107 個通過**：


## 五、已知限制（誠實揭露）

1. **沙盒環境限制（跟前幾輪交付相同）**：無法 `npm install`（`node_modules` 是空的）、無法連線 GitHub/Render、無法連真正的資料庫、無法真的執行 `next build` 或產生螢幕截圖。所有驗證都是：純邏輯自動測試（`tsx --test`，107/107 通過）＋人工逐檔案交叉比對 Prisma schema 欄位名稱／既有程式慣例＋大括號/小括號/中括號平衡檢查。沒有辦法在這個環境裡跑出「真正連資料庫」的整合測試。
2. **登入/session 機制尚未建立**：`src/lib/permissions.ts` 的 `AdditionalPrintItemAction` 權限矩陣已經定義好規則，但因為系統還沒有登入功能，這些規則目前無法在後端真正擋下（跟財務模組、祭改禁用編號、回收區還原/永久刪除等既有功能的限制完全一致）。
3. **Excel 匯入方式一（固定欄位）沒有對應的 API／UI**：純規則邏輯（`expandFixedColumnRow`）已完成且有測試，但這次只把方式二（明細工作表）接到真正的 API 與畫面。
4. **列印中心的匯入面板沒有手動欄位對應 UI**：跟活動精靈既有的匯入面板不同，這次的匯入面板只依賴系統自動辨識欄位名稱，沒有讓使用者手動調整欄位對應的下拉選單（系統仍然會把使用者上傳時的欄位名稱記憶下來，供下次自動帶出）。
5. **活動精靈尚未自動建立預設寶袋**：資料模型與 API 已經支援「預設寶袋」概念（`isExtra=false`），但活動精靈建立普渡活動時，這次沒有實作「依規則自動幫每一筆歷代祖先/冤親債主/個人乙位正魂/無緣子女建立一筆同名預設寶袋」這段自動化邏輯，目前預設寶袋一樣要透過「＋新增寶袋」手動建立（取消勾選「額外」即可標記成預設）或 Excel 匯入。
6. **無法真的產生 PDF**：列印中心「產生 PDF」的功能只完成資料格式與列印批次紀錄（`TempleEventPrintBatch`），沙盒環境無法產生真正的二進位 PDF 檔案，之後接上正式模板／PDF 引擎即可。
7. **card/會員路由 revalidatePath 覆蓋範圍**：由於這次列印中心是全新的跨家戶頁面路徑，`revalidatePath` 只涵蓋目前已知的頁面路徑，如果之後畫面路徑調整，需要一併更新對應 API 裡的 `revalidatePath` 呼叫。
