本輪**沒有開發任何新功能、沒有修改正式資料庫、沒有套用任何 migration**。
目標是把已確認的行政與財務規則整理成正式規格，並檢查現有架構能不能支援。

完整規格內容見：**`docs/FINANCE_AND_ACTIVITY_SPEC.md`**（十四節，含 ER
Diagram、建議 Schema、開發順序、風險與尚待確認事項——這是本輪真正的主要
交付物，以下只是摘要導覽）。

## 三、現有架構可沿用的部分

好消息：這個系統從很早的版本開始，就已經預留了不少剛好用得上的架構，
**不需要重做**：

- `RitualRecord` + `UniversalSalvationDetail` + `UniversalSalvationEntry`
  ——普渡的「內部祭祀內容」完全不用動，繼續當正式主檔。
- `FinanceRecord` / `FinanceRecordStatus` / `FinanceRecordType`——V1 就
  預留的流水帳架構，剛好可以延伸當作規格要求的「LedgerEntry」，只需要
  新增幾個欄位（現金/銀行帳戶、付款方式、活動歸屬、報名群組），不需要
  另外建一張平行的分錄表。
- `Role` / `User` / `src/lib/permissions.ts`——財務權限矩陣（只有
  SUPER_ADMIN 能用財務功能）早就定義好，且註解本來就明確要求「後端 API
  一定要驗證權限，不能只藏前端按鈕」，跟這輪的規則完全一致。
- `AuditLog`——通用的 entityType/entityId 設計，新資料表的異動記錄可以
  直接沿用同一張表，不需要各自另外做稽核表。
- `@@unique([householdId, year, activityType])` 的年度隔離設計哲學、
  「複製前一年資料」的既有 API 模式、「畫面順序另外用排序陣列、不依賴
  enum 宣告順序」的既有慣例——這幾個模式全部可以直接複用在新的年度活動、
  年度價格、複製價格功能上。

詳細對照表見規格書第十節。

## 六、建議開發順序

見規格書第十二節，共 9 個階段，從風險最低的 `ActivityYear`/
`ActivityPriceItem` 開始，到風險最高的收款/流水帳邏輯，再到普渡/南北巡的
特化邏輯，最後是報表——每一階段都可以獨立開發、獨立測試，符合這個專案
一貫「一次只做一個模組」的方式。

## 七、風險與尚待確認事項

見規格書第十三節，六個需要你在正式開發前確認的設計決策，例如：「應收」
要不要獨立成一張表、普渡報名明細的數量要不要跟內部祭祀筆數自動同步、
南北巡的車次設定要不要即時同步成報名明細——這些都是「有不只一種合理做法」
的地方，本規格先給出建議做法，但實際開發前建議你確認一次，避免走錯方向
要重做。
