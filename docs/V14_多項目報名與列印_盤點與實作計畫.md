# 三玄宮 ERP V14｜活動報名多項目架構＋列印管理整合
## 第一階段：完整盤點與實作計畫（本文件不含任何程式修改）

> 依指令【十二】：第一階段只盤點、不改程式。以下為盤點結果、資料架構、
> 檔案清單、migration、測試項目與風險確認，供確認後一次性實作。

---

## 一、核心發現：目前「一活動一報名表」限制的真正成因

現行分派鏈（就是信眾詳情頁「按不下去」的來源）：

```
信眾詳情頁 → NewActivityRegistrationDialog
  → GET /available-activities
    → listAvailableActivitiesForMember()
      → 每個 TempleEvent 用 resolveRegistrationFormType(e.registrationFormType)
        → registrationFormType 為 null → { supported:false, reason:"尚未設定報名表" }
  → 前端 formSupported=false → 「建立並填寫內容」按鈕 disabled（死路）
```

**限制根源**：`TempleEvent.registrationFormType` 是**單一字串**，一個活動只能對
應一種編輯器（UNIVERSAL_SALVATION / PURIFICATION / LANTERN / GENERIC）。
`registrationFormTypes.ts` 的 `resolveRegistrationFormType()` 是唯一分派點。

同時兩個唯一鍵限制了資料形狀（**不動這兩把鍵**，見風險章）：

- `TempleEvent @@unique([activityType, year])`：一年一活動類型一筆主檔。
- `RitualRecord @@unique([householdId, year, activityType])`：一戶一年一活動類型一筆報名。

## 二、既有架構可沿用清單（不得重做第二套）

| 面向 | 既有資產 | 本輪沿用方式 |
|---|---|---|
| 活動主檔 | `TempleEvent`（`@@unique[activityType,year]`） | 沿用，作為「主活動」 |
| 報名主檔 | `RitualRecord`（`@@unique[household,year,activityType]`） | 沿用，作為某戶某活動的報名匯總 |
| 報名成員 | `RitualParticipant`（含農曆生日/虛歲快照） | 沿用 |
| 普渡牌位 | `UniversalSalvationDetail` + `UniversalSalvationEntry`（category 四類） | 沿用為「超拔祖先/乙位正魂/冤親債主/無緣子女」 |
| 寶袋 | `AdditionalPrintItem`（POCKET，含 subtotal、單價快照） | 沿用為「基本寶袋＋額外寶袋」 |
| 年度燈財務 | `LanternRegistration`（amountDue/Paid/Unpaid） | 沿用 |
| 祭改 | `PurificationEntry`（編號、貼紙、feeStatus） | 沿用 |
| 供品/福壽龜 | `OfferingType`(behaviorKind=TURTLE) + `ActivityOffering` + `OfferingClaim` | 沿用為福壽龜 |
| 爐主 | `StoveMasterRegistration` | 沿用 |
| 收款 | 6 個 `ReceivableSourceAdapter` + `PaymentTransaction/Allocation/Adjustment` | 沿用，新增項目掛新 sourceType |
| 收據 | `Receipt/ReceiptLine/ReceiptPrintLog` | 沿用 |
| 列印模板 | `TemplateDefinition` + `TemplateVersion`（category、activityType、版本、isActive、fileUrl） | **沿用並擴充版型設定欄位** |
| 統一報名編輯器 | `/registration/[ritualRecordId]` + `RegistrationEditor` | 沿用，改成「項目分頁」 |

**結論**：不需要新增第二套主檔。多項目是「在 RitualRecord 底下加一層項目」。

## 三、最小必要擴充（新增資料表／欄位）

### 3.1 `RegistrationItemType`（活動項目設定：主活動 → 可報名項目）
「哪個活動有哪些報名項目、每項的收費與列印類型」的設定表。取代單一
`registrationFormType`，讓一個活動可掛多個項目。

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | cuid | |
| activityType | ActivityType | 該項目屬於哪種活動 |
| key | String | 穩定碼，如 `US_ANCESTOR`,`US_SPONSOR`,`LANTERN_GUANGMING`,`CELEBRATION_TABLE` |
| name | String | 顯示名稱，如「超拔祖先」「贊普」 |
| contentKind | String | 報名內容型態：`TABLET`/`POCKET`/`SPONSOR`/`RICE`/`LANTERN`/`PURIFICATION`/`TURTLE`/`TABLE`/`STOVE`/`ROSTER`/`GENERIC` |
| feeMode | String | 收費方式：`NONE`/`FIXED`/`CUSTOM`/`PER_UNIT`/`FIXED_OR_CUSTOM` |
| defaultUnitPrice | Decimal? | 預設單價（如寶袋、贊普固定費用） |
| defaultQuantity | Int? | 預設數量（福壽大龜=1、福壽中龜=6，**不寫死於程式**，存這裡可改） |
| allowMultiplePerMember | Boolean | 同一人此項是否可多筆 |
| printDocumentKeys | String[] | 此項對應的列印品 TemplateDefinition.key 清單 |
| sortOrder | Int | |
| isActive | Boolean | |

以「種子資料」建立普渡 7 項、年度燈 4 項、宮慶 3 項、補褲、龍鳳燈的預設項目
（可由管理者調整數量/單價，不寫死）。

### 3.2 `RitualRegistrationItem`（每筆報名的項目實例，RitualRecord 子表）
「這一戶這個活動，實際報名了哪些項目、各自的內容連結與金額」。

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | cuid | |
| ritualRecordId | FK → RitualRecord (Cascade) | 匯總主檔 |
| registrationItemTypeId | FK → RegistrationItemType (Restrict) | 是哪個項目 |
| memberId | FK → Member (Restrict)? | 若為個人項目 |
| quantity | Int @default(1) | 數量（福壽龜、寶袋、白米） |
| customName | String? | 額外寶袋可自訂名稱（不必等於牌位名稱） |
| amountDue/Paid/Unpaid | Decimal | 本項財務（沿用既有金額欄位慣例；DRAFT 不進待收款） |
| feeChoice | String? | `FIXED`/`CUSTOM`（贊普二選一） |
| linkedEntryId / linkedEntryType | String? | 連回既有明細（US Entry / AdditionalPrintItem / OfferingClaim / PurificationEntry / StoveMaster），比照 AdditionalPrintItem 的 entityType/entityId 慣例，不建強制 FK |
| status, deletedAt, snapshots… | | 沿用既有慣例 |

> 設計要點：`RitualRegistrationItem` 不重複存牌位/寶袋內容，而是**指回**既有明細
> 表（US Entry、AdditionalPrintItem、OfferingClaim…）。它是「項目層的索引與財務」，
> 既有明細表仍是內容真相來源 → 不建立第二套內容資料。

### 3.3 `TemplateDefinition` 版型設定擴充（指令七：所有列印品可編輯版型）
在既有 `TemplateVersion` 上新增版型設定欄位（沿用既有版本機制，不新表）：

- `paperSize` (String, 如 A4/自訂mm)、`orientation`、`marginsJson`
- `layoutJson`（欄位位置、字型、字級、行距、對齊、資料欄位綁定；以 JSON 儲存，
  本輪先存結構，視覺化拖拉編輯器後續接上）
- 既有 `fileUrl` 已可存上傳版型檔（沿用）

> 本輪只建可擴充結構與入口，不做完整視覺化編輯器（指令七明示）。

### 3.4 `ReceivableSourceType` 新增列舉值（供新收費項目掛 adapter）
`RICE_REGISTRATION`（白米，若收費）、`CELEBRATION_TABLE`（訂桌）、
`DRAGON_PHOENIX_LANTERN`（龍鳳燈）、`STORAGE_TROUSERS`（補褲）—— 各配一個
受控並行 adapter。enum 新增值須**獨立 migration**（PostgreSQL 限制，見 V13.4 教訓）。

## 四、七大列印規則對照（指令二～七）

| 活動 | 報名項目 | 列印品 |
|---|---|---|
| 中元普渡 | 超拔祖先｜乙位正魂｜冤親債主｜無緣子女｜增加寶袋｜白米登記｜贊普 | 各類牌位、基本寶袋、額外寶袋、白米名單、贊普名單、各項目總名單 |
| 年度燈 | 光明燈｜太歲燈｜全家燈｜祭改 | 光明/太歲/全家燈牌、三種疏文、祭改既有貼紙、各項目總名單 |
| 宮慶 | 訂桌（信眾/友宮）｜福壽龜｜爐主/副爐主 | 訂桌名單、福壽龜名單、爐主名單、總名單 |
| 補褲 | 報名名單 | 收據＋報名總名單（無其他專用列印） |
| 龍鳳燈 | 報名 | 固定格式燈品、年底宮務名單、報名總名單、補印 |

普渡特別規則：超拔祖先/冤親債主/無緣子女**預設各帶 1 個基本寶袋**（沿用
`AdditionalPrintItem` + `pocketPricing.ts`）；贊普 = 固定費用 or 自訂金額
（`feeMode=FIXED_OR_CUSTOM`）；福壽大龜預設 1、中龜預設 6（存
`RegistrationItemType.defaultQuantity`，可改，不寫死）。

## 五、預計修改／新增檔案

**Schema / Migration**
- `prisma/schema.prisma`：新增 `RegistrationItemType`、`RitualRegistrationItem`；
  擴充 `TemplateVersion` 版型欄位；`ReceivableSourceType` 新增值。
- migration（分檔）：① 建兩張新表＋TemplateVersion 欄位；② enum ADD VALUE（獨立）；
  ③ 種子資料（普渡7/年度燈4/宮慶3/補褲/龍鳳燈項目，可 idempotent upsert）。

**Lib（新增）**
- `src/lib/registrationItems.ts`：項目設定查詢（取代單一 formType 分派）、
  `listItemsForActivity()`、種子定義。
- `src/lib/registrationItemRegistration.ts`：項目層報名寫入（沿用 RitualRecord/
  Participant，連回既有明細），受控並行、無 N+1。
- `src/lib/printDocuments.ts`：列印品登錄與「一鍵總名單」查詢（批次、共用查詢）。
- 新增 adapters（白米/訂桌/龍鳳燈/補褲）於 `receivableAdapters.ts`（掛進 ADAPTERS，受控並行）。

**Lib（修改）**
- `registrationFormTypes.ts`：保留相容，改為委派到 `registrationItems.ts`（不再是唯一分派）。
- `activityRegistration.ts`：`listAvailableActivitiesForMember()` 改回傳「活動＋其項目清單」；
  未設定項目的活動不再是死路（可選項目才建立報名）。
- `devotee360.ts`：已報名項目顯示（沿用受控並行/批次原則）。

**API（新增）**
- `GET /api/devotee-center/[memberId]/available-activities`：回傳活動＋項目。
- `POST /api/registrations/[ritualRecordId]/items`：新增/編輯項目（權限、operatorUserId）。
- `GET /api/registrations/[ritualRecordId]/items`：列出已報名項目。
- `GET /api/print-center/rosters/[activityType]/[year]/[itemKey]`：一鍵總名單。
- 版型：沿用既有 `/api/templates/*`，補版型設定欄位存取。

**UI（修改／新增）**
- `NewActivityRegistrationDialog.tsx`：兩段式「先選主活動 → 再選項目」，移除死路。
- `RegistrationEditor.tsx`：改為「項目分頁」；頂部顯示主活動名稱／年度／項目名稱／狀態。
- 沿用 `UniversalSalvationScreen`、`LanternRegistrationEditor`、`ParticipantSelector`、
  祭改導向、`AdditionalPrintItemsPanel`。
- 列印中心：項目總名單入口與補印（沿用 `PrintCenter`/`LanternPrintCenter`/模板中心）。

## 六、Migration 需求
1. `*_v14_registration_items`：CREATE `registration_item_types`、`ritual_registration_items`；
   ALTER `template_versions` ADD 版型欄位。純附加，無 DROP/UPDATE。
2. `*_v14_receivable_source_values`：`ALTER TYPE ReceivableSourceType ADD VALUE …`（**獨立檔、逐句**）。
3. `*_v14_seed_registration_items`：以 SQL/程式 idempotent upsert 種子項目（可重跑，不覆蓋既有）。

正式環境一律「程式回退、附加資料表保留」，不 DROP、不 TRUNCATE、不 Reset。

## 八、不破壞既有流程確認（指令十二.9）
- **不動** `TempleEvent @@unique[activityType,year]` 與 `RitualRecord
  @@unique[household,year,activityType]`（V8.1 起全模組依賴）。多項目掛在 RitualRecord 之下。
- 既有普渡（US Detail/Entry）、年度燈（LanternRegistration）、祭改（PurificationEntry）、
  供品/福壽龜、爐主、收款 6 adapter、收據流程 **完全沿用**，`RitualRegistrationItem`
  只是「指回」它們的索引＋項目財務層。
- `registrationFormType` 欄位保留（標 deprecated），舊資料不受影響。
- 效能延續 V13.4：受控並行、批次查詢、singleton、不提高連線數。
- 權限延續：新 API 一律 `assert*PermissionForOperator`、READONLY 拒寫、不吞錯回 0。

## 九、待實作階段確認的一個架構決策（唯一需要你點頭的點）
