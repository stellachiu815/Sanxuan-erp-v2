# 台北三玄宮行政規則書（V6.3 新增）

**狀態：行政規則文件，尚未實作任何 UI 或 API。**

`docs/FINANCE_AND_ACTIVITY_SPEC.md` 專注在**資料架構與 Schema**（哪些表、
哪些欄位、為什麼這樣設計）；本文件專注在**行政操作規則**（畫面上該怎麼
表現、什麼情況要提醒、按鈕該叫什麼名字、流程的先後順序）——兩份文件互相
對照，資料架構的部分不重複寫在這裡，只在需要說明「這條行政規則背後依賴
哪張資料表」時會註明並連回規格書對應章節。

---

## 一、年度燈正式規格

三玄宮年度燈**只有兩種**：

1. 光明燈
2. 太歲燈

**不得預設其他燈種**——未來畫面上的年度燈報名選項，固定只會出現這兩個
選項，不會有「安太歲燈」以外的其他燈種選項可以選（跟南巡/普渡等其他活動
「管理者可以自由新增收費項目」的彈性明確不同，年度燈是刻意收斂、固定的
兩種）。資料架構對照：`docs/FINANCE_AND_ACTIVITY_SPEC.md` 第十五節 15.1。

---

## 二、犯太歲規格

每一年度，管理者可以設定「當年度犯太歲的生肖」，例如：

- 115 年：馬、鼠、雞、兔

**不可寫死在程式，必須可以修改**（新增、刪除都要能做，且是每一年各自
獨立設定，不是全系統共用一份清單）。資料架構對照：規格書第十五節 15.2
（`TaiSuiYearZodiac` 資料表提案）。

---

## 三、信眾提醒

搜尋信眾後，系統自動判斷這位信眾的生肖（沿用既有的生肖換算邏輯，V5.0
起就有）：

- **如果今年犯太歲**：畫面以黃色提醒：

  > ⚠ 本年度犯太歲，建議確認是否安奉太歲燈。

- **如果今年不犯太歲**：不要顯示任何提醒。

判斷依據：這位信眾的生肖，是否出現在「二、犯太歲規格」當年度設定的清單
裡。資料架構對照：規格書第十五節 15.3。**本輪只定案這條規則的文字與觸發
條件，尚未開發任何畫面。**

---

## 四、延用去年報名

新增正式行政流程，取代口語化的「跟去年一樣」，正式按鈕名稱固定為：

> 「**延用去年報名**」

（**不是**「跟去年一樣」——這是刻意的用詞規範，往後畫面/文件/教育訓練
一律使用「延用去年報名」這個講法）。

### 按下之後的預設行為

按下「延用去年報名」後，系統預設帶入（打勾）：

- ☑ 光明燈
- ☑ 祭改
- ☑ 普渡

**不要自動帶入**：

- ☐ 太歲燈（不論這位信眾今年犯不犯太歲，太歲燈都不會被「延用去年報名」
  自動勾選）

### 太歲燈的另外詢問

如果這位信眾今年犯太歲（判斷依據同「三、信眾提醒」），系統在完成前面
三項（光明燈/祭改/普渡）的延用之後，**另外獨立詢問**：

> 是否新增太歲燈？

行政人員可以選擇是或否，這個詢問**跟前面三項的延用是分開的兩件事**——
即使行政人員在這一步選擇「否」，前面已經延用成功的光明燈/祭改/普渡三項
完全不受影響。

### 流程圖（文字版）

```
點擊「延用去年報名」
        │
        ▼
系統讀取這位信眾/家戶去年度的登記資料
        │
        ▼
自動勾選並複製：光明燈 + 祭改 + 普渡（不含太歲燈）
        │
        ▼
判斷：這位信眾今年是否犯太歲？
        │
   ┌────┴────┐
   是          否
   │           │
   ▼           ▼
另外詢問：      流程結束
是否新增太歲燈？
   │
 ┌─┴─┐
 是    否
 │     │
 ▼     ▼
新增   流程結束
太歲燈
報名
```

資料架構對照：規格書第十五節 15.4（跨活動類型批次複製 API 的設計提案，
以及與現有「複製前一年資料」既有模式的差異說明）。**本輪只定案這條流程，
尚未開發任何按鈕或 API。**

## 五、普渡正式順序

普渡項目固定順序：

1. 歷代祖先
2. 個人乙位正魂
3. 冤親債主
4. 無緣子女
5. 贊普
6. 白米

此順序**畫面、列印、報表、收費全部一致**——不會有「畫面上一種順序、
列印又是另一種順序」的情況。這個順序從 V4.0/V4.1 的牌位列印中心起就已經
是既有的固定順序（`src/lib/ritual.ts` 的 `ENTRY_CATEGORY_ORDER`、
`src/lib/labels.ts` 的 `universalSalvationEntryCategoryOrder`），V6.3
只是重新確認並延伸到「收費」這個新的面向也要保持一致。

---

## 六、贊普與白米

贊普、白米**不設定固定價格**，由行政人員每次報名時手動輸入實際金額
（不像光明燈/太歲燈等其他項目可以有年度標準價格可以複製沿用）。資料架構
對照：規格書第十一節 `ActivityPriceItem.unitPrice` 允許為空，正是為了
容納這兩個「金額不預設」的項目。

---

## 七、普渡收款（對外顯示規則）

收款畫面與未來收據，**只顯示**：

- 功德主姓名

**不要顯示**：

- 冤親債主
- 無緣子女
- 歷代祖先
- 個人乙位正魂

**祭祀內容只保留在系統內部**，對外（收款畫面、收據）一律不逐項顯示這
四個類別的任何內容，只顯示功德主姓名（與合計金額，收款/收據本來就需要
金額才有意義）。**這是 V6.3 對 V6.2 規則的收斂**：V6.2 原本只要求隱藏
冤親債主/無緣子女兩項，V6.3 明確擴大為四個類別全部不對外顯示，以此份
文件與規格書第五節為準。

## 八、南巡、北巡

南巡與北巡**不以個人作為主要財務統計**，以「**車次**」作為統計單位。

每一車可設定：

- 每人單價
- 人數
- 小計（自動計算 ＝ 每人單價 × 人數）

另外：

- **雙人房加價**：每間價格、間數、小計（自動計算）
- **海報贊助**：自由輸入金額

全部項目自動加總成這趟南巡/北巡的總金額。資料架構對照：規格書第六節、
第十一節 `TourVehicle`/`TourRoomCharge`。

---

## 九、南北巡名單

若未來建立乘車/住宿名單，名單用途**只有**：

- 乘車管理
- 住宿安排

**不得作為主要財務統計依據**（財務統計固定以車次/雙人房/海報贊助的金額
加總為準，不是靠名單人數反推金額）。

---

## 十、報名與收款

再次確認：**報名 ≠ 收款**。

只有實際收到款項，才能更新：

- 現金
- 銀行餘額
- 流水帳

這條規則從 V6.2 就已經定案（規格書第三節），V6.3 再次確認，沒有變更。

## 十一、報名當下立即付款（V7.0.3 新增）

報名畫面完成計價後（不論是代辦人模式或家戶快速報名模式），必須提供
兩個明確的操作選項，按鈕名稱固定：

- **【先存為未繳】**：只建立報名與應收金額，不建立任何收款紀錄，狀態
  維持「未繳」，之後可以再另外走既有的收款流程補收。
- **【立即收款】**：在同一次操作裡，除了建立報名與應收金額，同時建立
  一筆收款紀錄。

按下【立即收款】時，畫面必須讓行政人員填寫：

- 實收金額
- 付款方式
- 現金或銀行帳戶（收款進哪一個帳戶）
- 付款人（可能不是報名本人或代辦人，例如子女代繳）
- 收款日期
- 備註

**狀態自動判定，不需要行政人員手動選擇**：

- 實收金額 < 應收金額 → 自動判定為「部分繳款」。
- 實收金額 = 應收金額 → 自動判定為「已繳清」。
- 實收金額 > 應收金額 → 自動判定為「溢收待處理」，**系統不可以自己
  把多收的差額吞掉或忽略**，必須留下清楚的紀錄等管理者後續處理。

**同一筆報名不會因為「報名」跟「收款」同時完成而重複計算**：不論行政
人員選【先存為未繳】還是【立即收款】，都只會產生一筆應收紀錄；選
【立即收款】時，也只會產生一筆收款紀錄與一筆流水帳分錄，跟之後管理者
另外幫這筆報名補收款所用的是同一套底層邏輯，不會因為「當下就收款」
而多算一次。完整資料模型與流程圖見 `docs/FINANCE_CORE_SCHEMA.md` 第
4.5 節，架構決策見 `docs/ADR.md` ADR-0013。**本節只記錄行政操作規則，
本輪不開發實際的報名畫面或 API。**

## 十二、家戶快速報名與姓名明細規格（V6.4 新增）

### 11.1 核心設計原則

三玄宮實際行政說法通常是「幫王小華一家報名光明燈」「幫王小華一家報名
祭改，但是某某人不用」——這是**操作語言**，不是**儲存語言**。系統必須
遵守：

- **家戶＝快速叫出整組人員的操作入口**（方便行政人員一次找到一整戶人）。
- **姓名＝真正報名、計價、查詢與儲存的單位**（每一位姓名各自獨立成一筆
  報名明細）。

**不可只存一筆「王小華一家已報名」，最後必須拆成每位姓名的獨立報名
明細。** 這條原則適用於年度燈（光明燈）與祭改；普渡有自己既有的、以
功德主/報名群組為核心的架構，不套用這套「一戶一鍵勾選」流程（見 11.9）。

### 11.2 家戶快速報名流程

行政人員先搜尋任一家戶成員姓名（例如「王小華」），系統開啟王小華所屬
家戶，並提供「**幫這一家報名**」入口。

活動選項至少包含：光明燈、太歲燈、祭改。

選擇活動後，系統叫出該家戶全部成員姓名，逐一附上核取方塊，例如：

```
☑ 王小華
☑ 王太太
☑ 王大兒子
☑ 王二女兒
☑ 王阿嬤
```

行政人員可以：

- 取消今年不用的人
- 保留今年要報名的人
- 新增目前家戶成員（如果名單裡漏了）
- 直接輸入臨時姓名（見 11.7）
- 重新勾選取消過的人

**確認後，才建立今年的姓名層級報名明細**——在按下確認之前，這個畫面上的
勾選狀態純粹是操作介面的暫存狀態，不會建立任何資料。

### 11.3 家戶快速報名流程圖（文字版）

```
行政搜尋任一家戶成員姓名（例如「王小華」）
        │
        ▼
系統開啟王小華所屬家戶頁面
        │
        ▼
點擊「幫這一家報名」
        │
        ▼
選擇活動（光明燈 / 太歲燈 / 祭改）
        │
        ▼
系統叫出該家戶全部成員姓名，預設狀態依情境而定：
  - 全新報名（非延用去年）：預設全部勾選，太歲燈永遠預設不勾選（見 11.4）
  - 延用去年報名：預設帶入去年勾選的姓名，太歲燈永遠預設不勾選（見 11.6）
        │
        ▼
行政人員調整勾選狀態／新增臨時姓名／新增家戶成員
        │
        ▼
（只有這一步才是真正的資料異動）按下「確認」
        │
        ▼
只替「目前勾選」的姓名，建立/更新今年這個活動的姓名層級報名明細
（未勾選者不建立明細；先前已建立、後來被取消勾選者，明細狀態改為
 已取消，不是直接刪除，見 11.8）
```

### 11.4 常見操作情境

**情境一：「幫王小華一家報名光明燈，但是阿嬤不用。」**

```
☑ 王小華
☑ 王太太
☑ 王大兒子
☑ 王二女兒
☐ 王阿嬤
```

只替勾選者建立光明燈報名——王阿嬤這一年沒有光明燈的報名明細（如果她
之前已經有明細、現在被取消勾選，則她的明細狀態改為「已取消」，見
11.8，不是資料庫刪除）。

**情境二：「幫王小華一家祭改，另外增加王小寶。」**

```
先叫出原家戶名單，再按「＋新增姓名」，輸入「王小寶」
```

最後替勾選者與王小寶建立祭改報名——王小寶如果不是這一戶的既有成員，
就是一筆臨時姓名（見 11.7）。

**情境三：同一家戶可同時勾選光明燈與祭改，但每一位姓名仍可分別取消。**

```
王小華：☑ 光明燈　☑ 祭改
王太太：☑ 光明燈　☐ 祭改
王阿嬤：☐ 光明燈　☐ 祭改
```

同一人同時報名兩個活動時，儲存成**兩筆獨立的報名明細**（見 11.8 姓名
層級儲存流程圖），不是同一筆明細記錄兩個活動。

### 11.5 年度燈規格（更新）

三玄宮年度燈只有光明燈、太歲燈兩種（V6.3 已定案，本輪不變）。

- **光明燈**：可使用「一家快速帶入」流程（11.2）。
- **太歲燈**：
  1. **不可因去年有報名就自動勾選**（即使使用「延用去年報名」，太歲燈
     這一項永遠不會被預設勾選，見 11.6）。
  2. 每年由管理者設定犯太歲生肖（V6.3 的 `TaiSuiYearZodiac`，本輪範例
     沿用「四個犯太歲生肖」的說法，實際數量由管理者自由設定，不是固定
     四個）。
  3. 系統依每位姓名的生肖，對犯太歲者顯示**淡黃色提醒**（呼應本專案
     一貫的莫蘭迪柔和配色，不用高飽和黃色）。
  4. 提醒文字固定為：「本年度犯太歲，請確認是否安奉太歲燈」。
  5. **行政仍需手動勾選**——提醒只是提醒，不會自動打勾。
  6. **即使犯太歲，對方不要也不可強制報名**——這是信眾的自主決定，系統
     只負責提醒，不負責替信眾做決定。

### 11.6 年度燈範例流程（光明燈 + 太歲燈並列）

```
行政開啟王家「幫這一家報名」，選擇「光明燈」與「太歲燈」
        │
        ▼
系統叫出王家全部成員姓名：
  光明燈欄位：全部預設勾選（或延用去年名單，見 11.2）
  太歲燈欄位：全部預設不勾選（不論是否延用去年報名，永遠不預設勾選）
        │
        ▼
系統逐一判斷每位成員的生肖是否在今年度 TaiSuiYearZodiac 清單裡
        │
        ├── 是 → 該姓名旁顯示淡黃色提醒「本年度犯太歲，請確認是否安奉太歲燈」
        └── 否 → 不顯示提醒
        │
        ▼
行政人員看到提醒後，自行決定是否勾選該姓名的太歲燈欄位（系統不代為勾選）
        │
        ▼
確認 → 光明燈欄位勾選者 + 太歲燈欄位勾選者，各自建立獨立的姓名層級報名明細
（同一人如果光明燈、太歲燈都勾，會有兩筆明細；見 11.8）
```

### 11.7 祭改規格（更新）

祭改也支援「一家快速帶入」（11.2），但**真正儲存時，每位姓名必須是
獨立報名明細**。同一戶可以：

- 全家都祭改
- 部分成員祭改
- 臨時新增家戶外姓名
- 每位姓名獨立取消

**不得只存家戶層級的「本戶已祭改」**——跟光明燈/太歲燈的儲存規則完全
一致（都是姓名層級明細），差異只在「哪些活動可以快速帶入」跟「太歲燈
不能自動勾選」這類活動專屬規則。

### 11.8 姓名層級儲存流程圖

```
確認畫面上勾選的姓名清單（可能混合既有成員與臨時姓名）
        │
        ▼
逐一處理每個勾選的姓名：
        │
        ├── 這個人「本年度、這個活動」已經有報名明細？
        │       │
        │       ├── 是，且狀態是「已取消」→ 改回「有效」狀態（重新勾選）
        │       ├── 是，且狀態已經是「有效」→ 不重複建立（維持現狀）
        │       └── 否 → 新建一筆報名明細（狀態＝有效）
        │
        └── 這個人同時勾選了「另一個活動」（例如光明燈+祭改）？
                │
                ▼
          分別各自建立/更新一筆報名明細（不合併成一筆）
        │
        ▼
畫面上被取消勾選、但本來已經有明細的姓名：
  → 明細狀態改為「已取消」（不是刪除資料列），
    保留歷史軌跡，之後「重新勾選」可以直接復原
        │
        ▼
每一筆明細各自帶有：真正家戶/成員（或臨時姓名資訊）、活動類型、年度、
標準單價快照、實際單價、數量、小計——計價與彙總見 11.10
```

### 11.9 普渡差異說明（普渡不套用這套「一戶一鍵勾選」流程）

普渡與年度燈、祭改的行政語意本質不同：

- 普渡仍使用**功德主 + 報名群組**的既有架構（V6.2/V7.0 已定案）：一位
  功德主底下可以涵蓋**不同家戶**的祭祀對象，不是「同一戶內部姓名勾選」。
- 普渡的登記內容是**歷代祖先／個人乙位正魂／冤親債主／無緣子女／贊普／
  白米**，這些不是「這位信眾今年要不要參加」的勾選題，而是「要不要
  幫這位（可能已經往生、也可能是無緣子女這種沒有戶籍資料的）對象做
  祭祀登記」的行政作業，本質上是**祭祀內容的登記**，不是**信眾出席
  意願的勾選**。
- 因此**不可把普渡簡化成單純的全家姓名勾選名單**——這會混淆兩種完全
  不同的行政語意，也會破壞普渡既有「內部祭祀內容保留在系統內部、對外
  只顯示功德主姓名」的隱私規則（見 V6.2/V6.3 規則）。

### 11.10 計價規則

即使從家戶一次帶入多人，**計價仍以每位姓名的報名明細計算**。例如光明燈
單價 600 元，勾選 4 人：4 × 600 ＝ 2,400 元。

同一人同時報名光明燈與祭改，會分成**兩筆**報名明細，再由系統加總（不是
把兩個活動的價格合併成一筆）。

畫面（未來開發時）應可即時顯示：已選人數、每項單價、每項小計、全部應收
總額——本輪只寫規格，不做 UI。

### 11.11 臨時姓名處理規則

活動報名必須支援臨時姓名——可能尚未建立正式信眾資料、是親戚或朋友、不
屬於目前家戶、只參加一次活動。臨時姓名建立報名明細時：

- `displayName`：必填，行政人員直接輸入的姓名。
- `sourceHouseholdId`：記錄這筆臨時姓名是在「哪一戶的快速報名畫面」
  裡新增的（即使這個人不屬於這一戶，也保留這個操作脈絡，方便之後追查
  「這筆臨時姓名是誰、在什麼情境下新增的」）。
- `isTemporaryName`：布林值，標記這是臨時姓名而非正式信眾。
- `memberId`／`sourceMemberId`：一開始都是空的。

**如果日後確認臨時姓名為既有信眾，可以再連回正式 `Member`，但不得影響
原活動歷史紀錄**——也就是說，事後補連結 `memberId` 是允許的，但這筆
報名明細本來的 `displayName`／年度／金額等歷史資料不會因為補連結而
被覆蓋或消失，只是多了一個「後來確認是這位成員」的關聯，供之後查詢
時使用。

### 11.12 重複姓名提醒規則

同一活動、同一年度新增姓名時：

- 如果有 `memberId`（既有信眾），**優先以 `memberId` 判斷是否重複**——
  同一個 `memberId` 在同一年度同一活動已經有有效的報名明細，就是真正
  的重複。
- 如果沒有 `memberId`（臨時姓名），則用 `displayName` ＋ 來源家戶 ＋
  備註協助辨識可能重複（這只是輔助判斷，不是精確比對，因為臨時姓名
  沒有唯一識別）。

發現可能重複時，提醒文字：「本年度此活動名單中可能已有同名資料，是否
仍要新增？」**不可直接擋死**，因為可能存在同名信眾（例如兩個不同家戶
剛好都有一位「王小明」），行政人員確認後仍可以選擇新增。

### 11.13 延用去年報名（更新：姓名層級版本）

按鈕名稱固定為「**延用去年報名**」（V6.3 已定案的用詞規範）。延用時可以
**先以家戶叫出去年的姓名名單**。例如去年光明燈：

```
☑ 王小華
☑ 王太太
☑ 王大兒子
☑ 王二女兒
☑ 王阿嬤
```

今年行政可以：取消某一位、新增某一位、全部取消、只保留部分姓名。**按下
確認前，不得直接建立今年資料**——跟一般的家戶快速報名流程（11.3）共用
同一套「先勾選調整、後確認才寫入」的原則。

**太歲燈永遠不因去年資料預設勾選**——不論是全新報名還是延用去年報名，
太歲燈欄位一律從「全部不勾選」開始，這條規則優先於「延用去年報名」的
一般預設帶入邏輯。

## 十三、收款中心操作規則（V7.1 新增）

收款中心是七個活動共用的收款/查詢/對帳入口，完整規格與資料模型見
`docs/COLLECTION_CENTER.md`，本節只記錄**行政操作面**的規則，供未來
畫面設計對照：

1. **搜尋**：收款中心首頁提供單一搜尋框，可用姓名、家戶、贊助人、
   付款人、電話、地址、活動別、年度、狀態任一條件查詢，不需要記得
   「這是哪個活動的資料」才能查到。
2. **快速收款按鈕**：收款中心首頁提供獨立的【快速收款】按鈕，跟一般
   「查到報名後補收款」是兩條不同的入口——快速收款用於信眾臨櫃直接
   拿錢出來、但報名資料還沒備齊的情境，填寫實收金額/付款方式/收款
   帳戶/付款人姓名/概略描述即可先入帳，畫面上要清楚標示這筆錢「尚未
   連結正式報名」，並提供之後「連結到報名」的操作入口。
3. **全家燈勾選流程**：年度燈/祭改報名畫面除了既有的逐人勾選（見第
   十二節家戶快速報名），需要額外提供「改為全家燈整組報名」的切換
   選項，切換後畫面顯示的是「整組價格 + 名單清單」，不是逐人單價
   加總；名單可以新增/移除姓名，可以勾選跨戶的親屬。
4. **贊助人與付款人顯示規則**：畫面上預設只顯示贊助人（功德主）
   姓名；付款人欄位若與贊助人相同，不需要重複顯示，只在兩者不同時
   才額外顯示付款人姓名。
5. **南北巡車次顯示規則**：預設顯示車次 1、3、5、6、7、8、9；車次
   2、4 預設隱藏，需要管理者另外在設定裡啟用才會出現在收款中心的
   車次彙總畫面中（沿用既有規則，本輪不變）。
6. **狀態標籤色彩**：收款中心的付款狀態標籤（未繳/部分繳款/已繳清/
   溢收待處理/退款待處理/已退款/已作廢）依 `docs/COLLECTION_CENTER.md`
   第十三節的色彩系統顯示，已完成類狀態用鼠尾草綠、待處理/未繳類狀態
   用灰粉玫瑰色，不使用大面積純紅色標示「未繳」以避免長時間使用的
   視覺疲勞與莊嚴/警示感過重的問題。

**本節只記錄行政操作規則，本輪不開發收款中心的實際畫面或 API。**

## 十四、代收待繳回提醒操作規則（V7.2 新增）

代收待繳回是收款中心底下「代收對帳」頁籤的操作規則，完整規格與資料
模型見 `docs/AGENT_REMITTANCE.md`，本節只記錄行政操作面規則：

1. **登記代收款項**：管理者在既有報名（`Receivable` 狀態未繳或部分
   繳款）的畫面上，另外新增「登記代收款項」的動作，填寫代收人姓名/
   代收金額/代收日期/備註，**不是報名畫面上的第三個按鈕**，是收款
   中心對既有報名的補充操作。
2. **首頁提醒卡**：顯示尚未繳回總筆數、總金額、依代收人分組的筆數與
   金額、最久未繳回天數，提供「進入對帳」按鈕直達收款中心代收對帳
   頁籤。
3. **代收對帳頁籤四個子頁籤**：尚未繳回／本次對帳／已繳回紀錄／有
   爭議待確認，操作流程是「勾選尚未繳回項目 → 進入本次對帳工作區
   逐筆輸入實際繳回金額 → 確認對帳」，勾選動作本身不會改變繳回狀態，
   只有按下確認對帳才會。
4. **提醒排程**：每週三、週五晚上 7:30 自動產生系統提醒，
   已提醒次數累加，下一次提醒日期自動往後推算，**只有管理者勾選已
   繳回並確認對帳後才停止**，未勾選的款項不會自動消失或被系統自動
   視為已繳回。
5. **繳回後的資料保留**：確認繳回的款項移入「已繳回紀錄」，完整保留
   已提醒次數與對帳資料，不會被刪除或清空。

**本節只記錄行政操作規則，本輪不開發代收對帳的實際畫面或 API，也不
建立提醒排程的實際執行機制。**

## 十五、代收待繳回對帳與月底財務報表操作規則（V7.1.1 新增）

本節取代第十四節（V7.2）大部分內容，完整規格與資料模型見
`docs/AGENT_COLLECTION_RECONCILIATION.md`（代收對帳）與
`docs/MONTHLY_FINANCIAL_REPORT.md`（月底報表），本節只記錄行政操作
面規則：

### 15.1 代收待繳回對帳

1. **四種狀態的行政意義**：信眾尚未付款（一般未繳）／信眾已付款但
   款項在代收人手上（`AgentCollectionItem` 存在但未對帳）／代收人
   已繳回三玄宮但尚未核對（對帳批次 `DRAFT`）／三玄宮已完成核對與
   正式入帳（對帳批次 `CONFIRMED`）。行政人員在畫面上看到的「繳回
   狀態」欄位就是這四種狀態的白話呈現，不是技術欄位名稱。
2. **雙軌帳本對行政人員的意義**：信眾已付款會立刻反映在該筆應收的
   「已收金額」，但**不會**立刻反映在宮方銀行/現金餘額——這是刻意
   設計，行政人員若在對帳前查詢現金餘額，看到的數字不包含代收人手上
   尚未繳回的款項，屬於正常現象，不是系統錯誤。
3. **代收對帳頁籤四個子頁籤**（尚未繳回／本次對帳／已繳回紀錄／有
   爭議待確認）的操作流程：勾選尚未繳回項目 → 系統即時顯示勾選筆數
   與小計金額 → 輸入代收人實際繳回的總金額 → 系統比對是否相符 →
   相符則直接確認對帳，不相符則進入差額處理（見下）。
4. **部分繳回**：代收人本次只繳回一部分金額時，行政人員可以針對單筆
   款項輸入「本次繳回金額」，剩餘金額保留在「尚未繳回」，狀態維持
   `PARTIAL`，不會被誤判為已結清。
5. **差額處理**：對帳金額與勾選項目小計不相符時，行政人員可以選擇
   （a）調整本次勾選範圍重新核對、（b）標記為有爭議待確認並填寫原因、
   （c）以實際收到金額為準、差額另立備註、（d）取消本次對帳重新開始，
   四個選項都會被完整記錄，系統不會自動吞掉差額或自動判定其中一方
   正確。
6. **提醒排程**：每週三、週五晚上 7:30 自動產生系統提醒，
   已提醒次數累加，下一次提醒日期自動往後推算，**至少提醒五次，
   五次後若仍未確認繼續提醒**，只有具備「經授權對帳確認」權限的
   管理者完成對帳確認後才停止，未勾選的款項不會自動消失或被系統
   自動視為已繳回。
7. **權限規則**：只有 OWNER、SUPER_ADMIN、或經授權（勾選
   `canConfirmAgentReconciliation`）的 FINANCE 角色可以確認對帳，
   一般 STAFF 只能登記代收款項、查看清單，不能自行確認對帳，避免
   同一人「自己收錢、自己核對」。

### 15.2 月底財務報表

1. **草稿自動產生**：每月 1 日凌晨系統自動產生上個月的財務報表草稿，
   行政人員不需要手動觸發，草稿產生後狀態為「待對帳」，需要行政人員
   逐一核對各帳戶餘額後才能確認。
2. **首頁提醒卡**：月初出現「上月財務報表待對帳」提醒卡，顯示報表
   期間、目前對帳進度、進入報表頁按鈕，確認/鎖定後提醒卡自動消失。
3. **帳戶對帳**：系統計算的月底帳面餘額與行政人員盤點/銀行對帳單
   的實際餘額不同時，**不可以自行調整系統餘額去配合實際餘額**，
   必須把差額與原因記錄下來，需要正式修正時另外建立跨月調整
   （`PeriodAdjustment`），經過核准後才會生效。
4. **確認與鎖定分開**：報表核對完成後先「確認」，確認後仍可由具權限
   者「鎖定」，鎖定後的報表**不能再被直接修改**，任何之後才發現的
   錯誤一律透過跨月調整處理，不會回頭覆蓋已鎖定的報表。
5. **鎖定權限**：只有 OWNER、SUPER_ADMIN、或經授權
   （`canLockMonthlyReport`）的 FINANCE 角色可以鎖定報表，一般 STAFF
   只能查看、協助對帳，不能鎖定。
6. **報表編號**：每份報表有唯一編號（民國年度-月份-FR-四位流水號），
   作廢的報表編號不會被重複使用，需要補印時使用同一編號並標示
   「補印」，不會產生新編號。

**本節只記錄行政操作規則，本輪不開發代收對帳、月底報表的實際畫面或
API，也不建立提醒排程、月底自動產生報表的實際執行機制。**
