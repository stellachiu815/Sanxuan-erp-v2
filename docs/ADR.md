# 台北三玄宮 ERP — Architecture Decision Record（ADR）

本文件記錄影響資料架構/系統設計的重要決策，格式固定為：狀態／背景／決策／
後果。之後每一輪如果有新的架構決策，會用新的 ADR 編號繼續累加在本文件
後段，不會回頭修改已經定案的舊 ADR（如果舊決策被推翻，會新增一筆
「取代（Supersedes）」關係的新 ADR，而不是直接改寫舊的）。

## ADR-0001：普渡「內部祭祀內容」與「對外收款」拆成兩層架構

**狀態**：已採納（V6.2）

**背景**：普渡的內部祭祀內容（歷代祖先/個人乙位正魂/冤親債主/無緣子女）
從 V2.0 起就已經是 `RitualRecord` → `UniversalSalvationDetail` →
`UniversalSalvationEntry` 的既有架構。V6.2 需要新增「報名/收款/應收/
已收」的財務概念，但普渡對外收款只能顯示功德主姓名跟合計金額，不能洩漏
祭祀明細。

**決策**：不修改既有的普渡內部祭祀內容架構，另外新增一層
`RegistrationGroup`/`RegistrationItem` 財務容器，`RegistrationGroup`
用一個可選欄位 `ritualRecordId` 關聯回既有的 `RitualRecord`。對外的收款/
收據畫面永遠只讀 `RegistrationGroup`/`Payment` 的彙總欄位，架構上完全
不會碰到 `UniversalSalvationEntry` 的明細內容。

**後果**：
- 優點：既有普渡登記功能完全不受影響、不用重寫；對外隱藏規則不需要在
  每一支 API 額外加「過濾邏輯」，是架構層級天生保證，不容易因為漏寫一行
  過濾條件而外洩。
- 代價：普渡的財務資料分散在兩個地方（`RitualRecord` 的內部內容 +
  `RegistrationGroup` 的財務彙總），需要靠 `ritualRecordId` 關聯起來；
  之後如果要做「內部祭祀內容筆數變動時，同步更新報名明細數量」的自動化
  （見十三節第 2 項尚待確認事項），需要額外的同步邏輯。

## ADR-0002：財務流水帳沿用既有 `FinanceRecord`，不新建平行的 LedgerEntry 表

**狀態**：已採納（V6.2）

**背景**：V1 時代就已經預留 `FinanceRecord`（type/amount/occurredOn/
status/createdBy/voidedBy）架構，但完全沒有畫面或 API 使用過。V6.2 需要
「流水帳分錄／LedgerEntry」的概念，記錄現金/銀行帳戶異動、活動歸屬、
報名來源。

**決策**：擴充既有 `FinanceRecord`（新增 `financialAccountId`／
`paymentMethod`／`activityType`／`registrationGroupId`／`paymentId`
欄位），不新建一張平行的 `LedgerEntry` 表。

**後果**：
- 優點：直接沿用 V1 已經設計好的 `FinanceRecordStatus`
  （DRAFT/CONFIRMED/VOID，只能作廢不能刪除）與 `AuditLog` 稽核機制，
  不需要重新設計一套一模一樣的狀態流轉；避免系統裡出現兩張本質相同但
  彼此獨立的流水帳表，之後报表要合併查詢會很麻烦。
- 代價：`FinanceRecord` 這張表的職責會變得比較重（同時要承載「一般雜項
  收支」跟「報名收款產生的分錄」兩種來源），需要靠 `registrationGroupId`/
  `paymentId` 是否為空來區分來源，查詢時要留意。

## ADR-0003：同一筆收款只建立一次 `FinanceRecord`，靠資料庫 `@@unique` 約束防止重複計算

**狀態**：已採納（V6.2）

**背景**：規格要求「同一筆實際收款同時進入活動帳、總帳、現金/銀行帳戶，
但只能建立一次收款紀錄，不可重複計算」。如果靠 API/前端邏輯自己小心
不要重複寫入，長期維護容易因為某一支 API 漏寫防呆而重複入帳。

**決策**：`FinanceRecord.paymentId` 欄位加上 `@@unique` 約束，一筆
`Payment` 保證只能對應一筆 `FinanceRecord`；「活動帳」「總帳」「現金/
銀行帳戶餘額」三種視角，全部是同一張 `FinanceRecord` 表用不同欄位篩選
查詢出來的結果，不是三張分開維護的表。

**後果**：
- 優點：資料庫層級保證不會重複計算，不依賴任何一支 API 記得檢查；三種
  報表視角永遠同步（因為根本是同一份資料），不會有「活動帳跟總帳對不
  起來」的情況。
- 代價：如果之後真的有「一筆收款需要拆到兩個不同活動帳」的特殊情境
  （目前規格沒有這個需求），現有設計不支援，需要另外設計。

## ADR-0004：南巡/北巡的車次/房間小計，寫入同一套 `RegistrationGroup`/
`RegistrationItem` 財務容器，不另建平行管線

**狀態**：已採納（V6.2）

**背景**：南巡/北巡以「車次」為財務統計單位，跟其他活動「以人/戶為單位」
明顯不同，一開始考慮過為南北巡另外設計一套獨立的應收/收款資料模型。

**決策**：`TourVehicle`/`TourRoomCharge` 只負責車次/房間的**設定與小計
計算**，管理者確認後，小計寫入同一套通用的 `RegistrationGroup`/
`RegistrationItem`，海報贊助也是一筆 `RegistrationItem`
（`chargeItemKey = "POSTER_SPONSOR"`）。

**後果**：
- 優點：「活動獨立損益」（應收/已收/未收/支出/結餘）對所有活動（含
  南北巡）都是同一套查詢邏輯，不需要為南北巡另外寫一套應收計算規則；
  `Payment`/`FinanceRecord` 的收款流程也完全共用，不需要為南北巡另外
  設計收款 API。
- 代價：`TourVehicle`/`TourRoomCharge` 的小計要「確認」才寫入
  `RegistrationItem`（見十三節第 3 項尚待確認事項），這個「確認」時機點
  需要在正式開發 UI 時明確定義清楚，不然容易讓管理者以為調整車次/單價
  當下就已經生效。

## ADR-0005：犯太歲名單獨立成 `TaiSuiYearZodiac` 表，用年度+生肖字串儲存，
不寫死在程式、不建外鍵關聯

**狀態**：已採納（V6.3）

**背景**：三玄宮每一年度的犯太歲生肖清單會變動（例如 115 年跟 116 年不
一樣），規格明確要求「不可寫死在程式，必須可以修改」。系統既有的生肖
換算邏輯（`src/lib/lunar.ts`，V5.0 建立）已經有一套生肖字串顯示方式。

**決策**：新增 `TaiSuiYearZodiac`（`year` + `zodiac` 字串，
`@@unique([year, zodiac])`），不使用固定 enum 列舉犯太歲生肖組合，也
不跟 `ActivityYear`/`RegistrationGroup` 建立外鍵關聯——這張表純粹是
「今年官方犯太歲名單是什麼」的事實清單，透過 `year` 欄位邏輯對照即可，
不屬於任何一筆年度活動或報名資料的子資料。

**後果**：
- 優點：管理者新增/刪除/修改犯太歲生肖清單，是單純的資料表 CRUD，
  不需要改程式碼、不需要重新部署；`zodiac` 用字串沿用既有生肖換算邏輯的
  顯示文字，不會有兩套生肖名稱互相對不上的風險。
- 代價：因為沒有外鍵關聯，資料庫層級無法保證「犯太歲清單裡的生肖字串」
  一定是系統認得的合法生肖（例如手誤打錯字），這一層驗證需要在 API 層
  呼叫既有的 `getZodiacOptions()` 做輸入驗證，不能只靠資料庫約束。

## ADR-0006：「延用去年報名」是跨活動類型的批次動作，需要新的批次複製 API，
不疊加呼叫既有的單一活動複製 API

**狀態**：已採納（V6.3）

**背景**：「延用去年報名」一次涉及三個不同的 `ActivityType`（年度燈裡的
光明燈項目、祭改、普渡），且年度燈只複製「光明燈」，刻意不複製「太歲燈」。
現有「複製前一年資料」的既有模式（普渡的 `copy-from-previous-year`）是
針對單一活動類型設計的。

**決策**：正式開發時，另外設計一支跨活動類型的批次複製 API，在同一個
資料庫交易（transaction）裡，依序建立/複製年度燈-光明燈、祭改、普渡三個
`RegistrationGroup`，而不是在前端依序呼叫三次既有的單一活動複製 API。

**後果**：
- 優點：交易一致性——三個活動要嘛全部複製成功，要嘛都不動，不會有
  「複製到一半」的殘缺資料；行政人員操作體驗上是「按一次鈕，三件事一起
  完成」，符合「延用去年報名」這個正式行政流程被期待的樣子。
- 代價：這支批次 API 比三支獨立 API 疊加呼叫更複雜，需要處理「其中一個
  活動今年已經有報名資料、不該重複建立」之類的邊界情況，正式開發時需要
  額外設計這些邊界案例的處理方式（本輪只記錄決策方向，實際的邊界案例
  規則留待正式開發前再確認）。

## ADR-0007：普渡對外收款隱藏規則，從「隱藏兩類別」收斂為「隱藏全部四類別」

**狀態**：已採納（V6.3，取代 V6.2 對應規則的文字描述，資料架構不變）

**背景**：V6.2 規格原本寫「對外收款/收據不得顯示冤親債主、無緣子女」
（暗示歷代祖先、個人乙位正魂可以對外顯示）。V6.3 使用者更新規則為：
對外收款/收據**只顯示功德主姓名**，四個類別全部不對外顯示。

**決策**：更新 `docs/FINANCE_AND_ACTIVITY_SPEC.md` 第五節文字，明確為
「四個類別（歷代祖先/個人乙位正魂/冤親債主/無緣子女）全部不對外顯示」。

**後果**：
- 優點：因為 ADR-0001 的架構決策（對外收款畫面架構上天生就不會 join
  `UniversalSalvationEntry` 明細），這次規則收斂**不需要任何資料模型或
  程式碼變更**，純粹是文件用詞更新，之後正式開發收據/收款畫面時，直接
  照最新（V6.3）文字實作即可，不會有「架構設計時只防了兩類別，事後才
  發現漏防另外兩類別」的風險。
- 代價：無（這是本輪少數「純文件更新、對架構完全沒有影響」的決策）。

## ADR-0008：`Receivable`（應收）定案為獨立資料表，不是 `RegistrationGroup`
上的彙總欄位

**狀態**：已採納（V7.0，取代 ADR 未編號前 V6.2 尚待確認事項第 1 項的
「暫定做法」）

**背景**：V6.2 規格書當時建議把應收做成 `RegistrationGroup` 上的彙總
欄位（`receivableTotal` 等），理由是應收金額完全由 `RegistrationItem`
加總得出，沒有自己獨立的生命週期，但同時列為「尚待確認事項」請使用者
決定。V7.0 需求明確把 `Receivable` 列為財務核心九個模型之一，等於明確
決定了這個問題。

**決策**：新增獨立的 `Receivable` 模型，與 `RegistrationGroup` 一對一
（`@@unique`），欄位包含 `totalAmount`/`paidAmount`/`refundAmount`/
`unreceivedAmount`/`status`（`ReceivableStatus`：UNPAID/PARTIAL/PAID/
REFUND_PENDING）。這張表**完全不含任何 `FinancialAccount` 關聯**。

**後果**：
- 優點：「應收」有自己獨立的資料表跟生命週期，之後如果需要追蹤應收金額
  本身的異動歷史（例如報名明細事後修正），可以直接在 `Receivable` 這張
  表上加 `AuditLog` 記錄，不需要去拆解 `RegistrationGroup` 的彙總欄位
  異動軌跡；「應收不得更新銀行」這條規則因為 `Receivable` 天生沒有任何
  跟 `FinancialAccount` 相關的欄位或關聯，是架構層級的保證，不是靠
  程式邏輯自律。
- 代價：比 V6.2 原本的「彙總欄位」設計多一張表、多一次 join，但考量到
  這是財務核心資料，寧可多一點正規化換取資料完整性與稽核彈性。

## ADR-0009：流水帳分錄正式定名為 `LedgerEntry`，實作方式是既有
`FinanceRecord` 的正式改名＋擴充，沿用同一張實體資料表

**狀態**：已採納（V7.0，延伸 ADR-0002 的決策）

**背景**：ADR-0002（V6.2）決定「流水帳分錄沿用既有 `FinanceRecord`，
不新建平行表」，但當時沒有正式改名，Prisma model 名稱仍是
`FinanceRecord`。V7.0 需求明確把這個概念正式命名為 `LedgerEntry`
（財務核心九個模型之一）。

**決策**：Prisma model 正式從 `FinanceRecord` 改名為 `LedgerEntry`
（`FinanceRecordType`/`FinanceRecordStatus` 對應改名為
`LedgerEntryType`/`LedgerEntryStatus`，列舉值不變），但**沿用同一張
實體資料表**（`@@map("finance_records")` 不變），不是另建一張新表。
`User` 上的關聯名稱也對應從 `FinanceRecordCreatedBy`/
`FinanceRecordVoidedBy` 改為 `LedgerEntryCreatedBy`/
`LedgerEntryVoidedBy`。

**後果**：
- 優點：跟 ADR-0002 的理由一致（避免兩張本質相同的流水帳表）；因為
  `finance_records` 資料表目前在正式環境是空表（V6.1 已確認 row count
  = 0，且從來沒有任何 API 寫入過），這個改名在未來真正套用 migration
  時只是一個低風險的 `ALTER TABLE ... RENAME`/欄位新增操作，不涉及任何
  既有資料遷移。
- 代價：正式開發時的 migration 需要處理 Prisma model 改名產生的
  migration diff（Prisma 通常會偵測成「新建表+刪除舊表」而非
  「rename」，需要手動調整產生的 migration SQL 改成 `ALTER TABLE
  RENAME`，避免真的把空表刪除重建——這點列入正式開發前的注意事項）。

## ADR-0010：`ActivitySummary`（活動統計）設計為快取/彙總表，由應用邏輯
在相關資料異動時重新計算寫入，不是即時查詢視圖

**狀態**：已採納（V7.0）

**背景**：V6.2 規格書認為「活動獨立損益」（應收/已收/未收/支出/結餘）
只需要對 `RegistrationGroup`/`FinanceRecord` 做查詢彙總即可，不需要
額外的資料表。V7.0 需求明確把 `ActivitySummary` 列為財務核心九個模型
之一，要求「每個活動自動統計」。

**決策**：新增 `ActivitySummary`，與 `ActivityYear` 一對一，欄位為
`receivableTotal`/`receivedTotal`/`unreceivedTotal`/`expenseTotal`/
`balanceTotal` + `recalculatedAt`（最後一次重算時間）。定位為**快取表**：
應用邏輯在 `Receivable` 或 `LedgerEntry` 異動時，重新彙總計算並寫入這張
表，而不是每次讀取都即時運算聚合查詢。

**後果**：
- 優點：畫面讀取活動統計時，直接讀一筆 `ActivitySummary`，不需要每次
  都對 `Receivable`/`LedgerEntry` 做複雜的分組聚合查詢，效能較好，且
  「目前統計是幾點更新的」（`recalculatedAt`）本身也是有用的資訊；跟
  `Receivable`/`LedgerEntry` 解耦，之後如果統計邏輯要調整，重算一次
  `ActivitySummary` 即可，不影響原始交易資料。
- 代價：需要在應用邏輯層面正確地在每次相關資料異動後觸發重算，如果漏寫
  某個觸發點，`ActivitySummary` 可能會跟實際的 `Receivable`/
  `LedgerEntry` 資料短暫不同步——正式開發時建議用資料庫 trigger 或是
  集中在少數幾支「異動 API」裡統一觸發重算，不要分散在很多地方各自
  呼叫，降低漏寫的風險（這點列入正式開發前的注意事項，本輪只定案資料
  模型，不涉及觸發時機的實作細節）。

## ADR-0011：家戶作為快速報名入口，姓名作為實際報名單位

**狀態**：已採納（V6.4）

**背景**：V6.4 需求明確指出，行政人員實際操作語言是「幫王小華一家報名
光明燈」「幫王小華一家報名祭改，但是某某人不用」，如果系統只儲存一筆
「王小華一家已報名」，就無法回答「這一戶今年誰有報名、誰沒有」「某個人
今年報名了哪些活動」這類查詢，也無法針對單一個人取消或計價。同時，行政
人員的操作習慣仍然是「以家戶為單位」叫出名單，不會逐一手動輸入每個人的
姓名。

**決策**：

1. **為什麼不能只存家戶層級**：報名、計價、查詢、歷史紀錄的最小單位
   永遠是「姓名」，不是「家戶」。同一戶裡不同人可能報名不同活動組合
   （例如王小華光明燈+祭改都報，王阿嬤都不報），單價、數量、金額都是
   按人計算，家戶層級的單一狀態欄位無法表達這種組合，因此
   `RegistrationItem` 必須是「每個人、每個活動、每年」各自一筆獨立
   紀錄，`householdId` 只是這筆紀錄的**來源標記**之一，不是報名的
   儲存單位。
2. **為什麼管理端操作仍然需要家戶快速帶入**：行政人員的日常語言是
   「幫這一家報名」，如果系統要求逐一手動輸入每個成員姓名，會嚴重拖慢
   行政效率，違反本專案「所有功能都以行政人員最快完成工作為最高原則」
   的開發目標。因此系統在**操作層面**提供「搜尋任一成員 → 叫出整戶
   名單 → 勾選」的快速入口，但在**儲存層面**仍然逐一拆解成獨立的
   `RegistrationItem`，兩者並不衝突：家戶快速帶入只是減少手動輸入的
   UI 捷徑，不代表資料庫也用家戶層級儲存。
3. **整戶快速操作與逐人取消/新增如何並存**：勾選畫面預設帶出整戶名單
   （或去年名單），行政人員可以：取消勾選（不刪除資料，只是這次不建立
   該人的報名明細，若去年已有明細則將狀態改為「已取消」而非刪除）、
   維持勾選（建立/沿用該人的報名明細）、新增本次才要加入的人（可能是
   臨時姓名）。每個人的勾選狀態彼此獨立，同一戶可以同時報名多種活動，
   每種活動各自有獨立的勾選狀態矩陣（例如王小華光明燈✓祭改✓，王太太
   光明燈✓祭改✗）。這靠 `RegistrationItem.status`
   （`ACTIVE`/`CANCELLED`）與「每人每活動每年最多一筆有效紀錄」的邏輯
   來實現，取消是「狀態改變」不是「刪除」，之後重新勾選可以直接把狀態
   改回 `ACTIVE`，不會遺失原本的單價/備註等歷史資訊。**太歲燈是這套
   快速勾選機制下的例外**：即使有整戶快速帶入，太歲燈永遠不因為「去年
   有報名」或「今年犯太歲」而自動預先勾選，必須由行政人員每年手動確認，
   避免信眾在不知情的情況下被安奉太歲燈並收費。
4. **臨時姓名如何處理**：`RegistrationItem` 的 `memberId`/`householdId`
   可以為空，改用 `displayName`（必填，直接輸入姓名文字）、
   `sourceHouseholdId`（記錄這筆臨時姓名是從哪一戶的快速帶入畫面新增的，
   即使這個人本身不屬於該家戶正式成員）、`isTemporaryName = true` 標記。
   之後如果確認這位臨時姓名其實是系統裡已有的正式信眾，可以事後把
   `memberId` 補上、重新連結回正式 `Member`，但**不能回頭修改或覆蓋
   這筆歷史紀錄原本的活動內容/金額**，只是補上關聯，保留原始報名當下
   的事實紀錄。
5. **為什麼普渡不套用同一套邏輯**：普渡的核心單位是「功德主（代辦人）＋
   報名群組」，且同一個報名群組底下的四類祭祀對象（歷代祖先/個人乙位
   正魂/冤親債主/無緣子女）本質上是**祭祀內容的登記**，不是「這個人要不
   要參加」的出席勾選——冤親債主、無緣子女甚至不是真實存在的個人身份，
   沒有辦法用「勾選家戶成員」的方式表達。此外普渡的代辦人經常橫跨多個
   不相關的家戶（例如同一位功德主同時幫王家、陳家、李家報名），不符合
   家戶快速報名「同一戶內按姓名勾選」的前提。如果把普渡簡化成全家勾選
   清單，會破壞「內部祭祀內容 vs 對外收款只顯示功德主姓名」這條既有的
   隱私規則（見 ADR-0001），因此普渡維持既有的代辦人＋報名群組模式，不
   套用本 ADR 的家戶快速報名邏輯。

**後果**：
- 優點：資料儲存忠實反映「姓名才是報名單位」的業務事實，支援任意組合
  的個人化取消/新增，同時透過家戶快速帶入入口保留行政人員熟悉的操作
  語言，兩者不衝突；取消用狀態變更而非刪除，保留歷史軌跡且可還原。
- 代價：`RegistrationItem` 需要承擔比原設計更多欄位
  （`sourceHouseholdId`/`sourceMemberId`/`isTemporaryName`/`status`），
  查詢「這戶今年報名了誰」時需要用 `householdId` 或
  `sourceHouseholdId` 過濾再依 `status` 篩選，比單純的家戶層級狀態欄位
  多一層邏輯；重複姓名偵測（同一年同活動是否已有同名資料）需要額外的
  比對邏輯（優先用 `memberId`，否則用姓名+來源家戶+備註），無法完全
  自動化防呆，只能用非阻擋性提醒（見
  `docs/ADMINISTRATION_RULES.md` 11.12）。
- 本輪只定案資料模型與規格文字，`RegistrationItem` 的欄位擴充見
  `docs/FINANCE_CORE_SCHEMA.md`，**不修改 `prisma/schema.prisma`、
  不建立 migration**。

## ADR-0012：`ActivityYear` 擴充為完整的「年度活動引擎」，首頁切換邏輯
完全依資料狀態判斷，不依國曆月份寫死

**狀態**：已採納（V7.0.2）

**背景**：V7.0 的 `ActivityYear` 只有「這個活動今年是否舉辦」
（`isHeld`）這一個狀態欄位，不足以支撐「活動名稱可改」「顯示排序可改」
「報名期間」「活動日期（含農曆/國曆/神明指定）」這些行政上實際需要
逐年調整的資訊。V7.0.2 需求明確要求首頁「目前主打哪個活動」「下一個
活動是什麼、還有幾天」都必須是資料驅動，不能是程式碼裡判斷「現在是
幾月」的寫死邏輯，且新增「補庫」作為第七種年度活動。

**決策**：

1. **`ActivityYear` 擴充九個欄位**：`activityName`/`sortOrder`/
   `isRegistrationOpen`/`registrationStartDate`/`registrationEndDate`/
   `activityDate`/`lunarDate`/`solarDate`/`isDivineAssignedDate`，
   完整定義見 `docs/ACTIVITY_ENGINE.md` 第十三節。這是同一張表的欄位
   擴充，不是另建新表——延續 V7.0 以來「同一個實體概念只用一張表」的
   一貫做法（例如 ADR-0009 的 `LedgerEntry`/`FinanceRecord`）。
2. **`ActivityType` enum 新增 `TREASURY_REPLENISHMENT`（補庫）**：
   固定農曆十月十五，插入「普渡」與「北巡」之間，七種活動的固定顯示
   順序見 `docs/FINANCE_AND_ACTIVITY_SPEC.md` 第一節、
   `docs/ACTIVITY_ENGINE.md` 第三節。
3. **首頁「目前主打活動」判斷依 `ActivityYear` 的
   `isActive`/`isRegistrationOpen`/`registrationStartDate`/
   `activityDate` 等欄位比對「今天」計算，不依國曆月份**：這代表
   任何一年、任何一個活動的實際受理期間或日期改變，首頁行為都會自動
   跟著變，不需要改程式——這是本輪需求「不需要修改程式」的核心保證，
   完整判斷規則見 `docs/ACTIVITY_ENGINE.md` 第五、六節。
4. **`isHeld` 是否整併為 `isActive`**：本輪建議整併（語意重疊），
   列為尚待確認事項（`docs/ACTIVITY_ENGINE.md` 第十五節第 2 項），
   若你確認整併，正式開發時會是一個欄位改名的 migration，不影響資料
   內容。
5. **神明指定日期（`isDivineAssignedDate`）獨立於「日期是否已填」**：
   南巡/北巡允許行政人員先填草稿日期方便籌備，但只有這個欄位為
   `true` 時，首頁才會顯示「日期已確認」，否則一律顯示「等待神明指定
   日期」，避免信眾誤以為日期已經定案。

**後果**：
- 優點：新增/修改任何一年的活動安排（改名稱、改順序、改報名期間、
  改日期），全部是資料異動，不需要改程式碼或重新部署；首頁邏輯對七種
  活動一視同仁，之後如果要新增第八種活動只需要新增 enum 值＋建立
  `ActivityYear` 資料，不需要修改首頁判斷邏輯。
- 代價：`activityDate`/`lunarDate`/`solarDate` 三個欄位在多數情況下
  有值重疊（例如宮慶的 `activityDate` 通常直接等於 `solarDate`），
  需要應用邏輯維護一致性，這點已列入尚待確認事項；「目前主打活動」
  在多個活動同時受理中時（例如年度燈+祭改同期）如何呈現，本輪只定義
  查詢邏輯排序依據（`sortOrder`），實際畫面呈現方式留待未來 UI 設計。
- 本輪只定案資料模型與規格文字，完整 Schema 提案見
  `docs/ACTIVITY_ENGINE.md` 第十三節，**不修改 `prisma/schema.prisma`、
  不建立 migration**。

## ADR-0013：報名當下允許立即收款，`Payment` 直接攜帶收款帳戶與付款人，
不建立平行的第二套收款路徑

**狀態**：已採納（V7.0.3）

**背景**：實務上行政人員常常在報名的同一次櫃台作業裡就收到款項（信眾
報名當場付現金），如果系統強制「報名」跟「收款」永遠是兩個分開的畫面/
兩次操作，會讓最常見的當場繳費情境變得比實際流程更麻煩。V7.0.3 需求
明確要求報名完成計價後，提供「先存為未繳」與「立即收款」兩個選項，且
立即收款要能一次填完金額/付款方式/收款帳戶/付款人/收款日期/備註。

**決策**：

1. **`Payment` 新增 `financialAccountId`（必填）與 `payerName`
   （選填）欄位**：收款當下就指定現金/銀行帳戶，不必等到流水帳分錄
   才選——自動建立的 `LedgerEntry.financialAccountId` 直接沿用
   `Payment.financialAccountId`，避免同一筆收款在 `Payment` 與
   `LedgerEntry` 兩處各自選帳戶、可能選到不一致的風險。`payerName`
   記錄實際付錢的人（可能不是報名代辦人本人），純文字欄位，不強制
   連回 `Member`。
2. **「先存為未繳」與「立即收款」是同一支建立報名邏輯的分支，不是兩套
   平行 API**：兩者都會建立 `RegistrationGroup`/`RegistrationItem`/
   `Receivable`；「立即收款」只是在同一次操作裡，額外呼叫一次跟
   「之後另外幫這筆報名收款」完全相同的建立 `Payment` 邏輯。這保證
   ADR-0003（同一筆收款只建立一次 `FinanceRecord`/`LedgerEntry`）跟
   4.2 節「`@@unique([paymentId])`」的保證，在報名當下立即收款這個
   情境下依然成立，不需要為這個情境另外設計防重複機制。
3. **狀態自動計算依實收金額 vs 應收金額判斷，沿用既有
   `ReceivableStatus`，不新增值**：實收 < 應收 → `PARTIAL`；實收 =
   應收 → `PAID`；實收 > 應收 → `REFUND_PENDING`（溢收待處理）。V7.0
   的 `ReceivableStatus` 本來就有這四個值，本輪只是把「立即收款」這個
   新入口也綁定同一套既有的狀態計算規則，沒有引入新概念。
4. **溢收不可自行吞掉差額**：`REFUND_PENDING` 狀態的設計本意就是強迫
   留下一筆「多收了多少錢、還沒處理」的紀錄，不能因為差額很小就在
   應用邏輯裡直接忽略或自動核銷，必須留待管理者另外處理（退還或轉列
   其他用途），避免帳務對不起來卻查無原因。

**後果**：
- 優點：最常見的「報名當場繳費」情境只需要一次操作，不用先報名、
  存檔後再切換到另一個收款畫面重新查詢同一筆報名；`Payment` 直接帶有
  收款帳戶跟付款人，資料完整度比事後才收款更容易維持一致；沿用既有的
  防重複與狀態計算邏輯，沒有新增系統複雜度。
- 代價：`Payment` 多了 `financialAccountId` 必填欄位，代表所有既有的
  「建立 Payment」呼叫點（不只是報名當下立即收款這一個情境，還包含
  之後管理者另外幫舊報名收款）都要求呼叫端明確指定收款帳戶，正式開發
  API 時需要注意這個欄位沒有預設值、不能省略。
- 本輪只定案資料模型與規格文字，完整 Schema 與流程圖見
  `docs/FINANCE_CORE_SCHEMA.md` 第 4.5 節，**不修改
  `prisma/schema.prisma`、不建立 migration**。

## ADR-0014：報名與收款分離，但允許同一步完成——收款中心層級的正式擴充

**狀態**：已採納（V7.1）

**背景**：ADR-0013（V7.0.3）已經在「報名」這一個入口定案「報名與收款
分離、但允許同一步完成」的原則，但只涵蓋年度燈情境，且
`ReceivableStatus` 只有 4 個值，無法精確表達「溢收待處理」跟「退款
待處理」是不同階段、也無法表達「這筆報名整筆作廢」跟「已經收過款項
但要退款」的差異。V7.1 需要把這個原則正式擴充成**七個活動共用**的
收款中心規則，並補上更精確的狀態機。

**決策**：

1. **「報名與收款分離、允許同一步完成」正式定案為跨活動通用規則**，
   不是年度燈專屬——`docs/COLLECTION_CENTER.md` 第二節正式定案這是
   收款中心底下所有活動共用的行為，`docs/ADMINISTRATION_RULES.md`
   第十一節（V7.0.3 原始規則）保留作為最初定案的行政規則描述。
2. **`ReceivableStatus` 由 4 值擴充為 7 值**：`UNPAID`/`PARTIAL`/
   `PAID`/`OVERPAID_PENDING`（溢收待處理，新拆分自舊
   `REFUND_PENDING`）/`REFUND_PENDING`（退款待處理）/`REFUNDED`
   （已退款）/`VOIDED`（已作廢）。拆分理由：溢收不一定要退錢（可能
   補登項目吸收差額），需要跟「已經決定要退款、只是還沒付出去」的
   狀態分開，否則畫面上無法分辨「這筆錢多收了、待決定怎麼處理」跟
   「這筆錢已經確定要退、正在處理中」兩種完全不同的行政動作。
3. **新增獨立的 `PaymentStatus`（`ACTIVE`/`VOIDED`/`REFUNDED`）**：
   因為一筆 `Receivable` 可能對應多筆 `Payment`（分次繳款、退款
   沖銷款），`ReceivableStatus` 描述的是整筆應收的彙總狀態，無法
   標示「這一筆特定收款記錄本身」是否被作廢或已被退款沖銷，因此
   需要模型層級分開兩種狀態。
4. **明確禁止 `PAID`/`PARTIAL`/`OVERPAID_PENDING` 直接跳到
   `VOIDED`**：已經收過的錢一定要先走退款流程
   （`REFUND_PENDING`→`REFUNDED`）才能作廢，避免帳上出現「作廢了，
   但錢還在戶頭裡沒人知道要退」的漏洞——這是本輪新增的狀態機層級
   保護規則，完整狀態轉換圖見 `docs/COLLECTION_CENTER.md` 第三節。
5. **`Payment` 大幅擴充欄位**（`registrationGroupId` 冗餘存放、
   `payerNameSnapshot` 正式改名自 `payerName`、`payerMemberId`/
   `payerHouseholdId`、`status`、`createdById`），並新增
   `QuickCollectionEntry`（快速收款）與
   `FamilyLampGroup`/`FamilyLampMember`（全家燈整組計價）兩組新模型
   ——完整欄位與模型定義見 `docs/COLLECTION_CENTER.md` 第四、八、
   十二、二十節。

**後果**：
- 優點：狀態機更精確，行政人員可以清楚分辨「多收了、待處理」跟
  「確定要退、處理中」兩種不同待辦事項；`PaymentStatus` 讓單筆收款
  記錄的作廢/退款狀態不會跟整筆應收的彙總狀態混淆；`payerNameSnapshot`
  的改名明確表達這是歷史快照，不會因為之後修改正式姓名資料而讓歷史
  交易紀錄跟著變動。
- 代價：`ReceivableStatus` 從 4 值變 7 值，任何既有假設「只有 4 種
  狀態」的程式碼（未來實作時）都需要重新檢查；`Payment.receivableId`
  改為 nullable 是一個需要謹慎執行的欄位變更，需要在正式開發時盤點
  所有讀取這個欄位的程式碼位置（見 `docs/COLLECTION_CENTER.md` 第
  二十四節風險清單）。
- 本輪只定案資料模型與規格文字，完整 Schema、ER Diagram、流程圖見
  `docs/COLLECTION_CENTER.md`，**不修改 `prisma/schema.prisma`、
  不建立 migration**。

## ADR-0015：南巡／北巡以車次為主要財務統計單位，不提供逐人收據

**狀態**：已採納（V7.1）

**背景**：南巡/北巡的財務設計（`TourVehicle`/`TourRoomCharge`）從
V6.2/V7.0 就已經是「以車次為單位」計算小計，寫入
`RegistrationGroup`/`RegistrationItem`（見
`docs/FINANCE_AND_ACTIVITY_SPEC.md` 第六節），但這只是**設計傾向**，
沒有在收款中心的呈現層明確定案「不可以提供逐人收據」這個限制。V7.1
收款中心需要一個清楚的規則，避免未來開發收款中心搜尋/收據功能時，
順手加上「查詢某人這趟南巡繳了多少」的逐人收據功能，跟原本車次為主
的財務統計精神產生落差。

**決策**：

1. **收款中心對南巡/北巡只呈現「以車次為單位」的財務彙總**——應收/
   已收/未收都是按車次（1、3、5、6、7、8、9 預設顯示，2、4 需要
   管理者啟用）分組顯示，不提供「這位信眾這趟南巡/北巡繳了多少」的
   逐人收據或逐人收款操作介面。
2. **`RegistrationItem` 明細仍然存在，不刪減資料**——王小姐這趟南巡
   繳了多少錢的明細資料依然完整記錄在 `RegistrationItem`，只是收款
   中心的**呈現層**不把單一報名人當作可以獨立收款/退款/開立收據的
   財務單位；如果未來確實需要逐人收據，屬於另一輪需求，需要重新
   評估是否要打破這個限制。
3. **理由**：南巡/北巡的報名與收費實務上是跟著「車次/房間/海報」這些
   實體資源走的，不是跟著單一報名人走的（例如同車的人可能合買一個
   名額、或臨時併車），逐人收據容易跟實際收費邏輯脫節，造成對帳
   困難。

**後果**：
- 優點：收款中心的呈現邏輯單純，不需要為南北巡另外設計一套逐人收據
  介面；財務統計口徑跟既有 `TourVehicle` 車次設計完全一致。
- 代價：如果宮方未來確實需要開立逐人收據（例如某些信眾要求索取收據
  報稅或核銷），需要另外評估設計，本輪明確排除在範圍外。
- 本輪只定案呈現規則，不影響既有 `TourVehicle`/`TourRoomCharge`/
  `RegistrationItem` 資料模型，**不修改 `prisma/schema.prisma`、
  不建立 migration**。

## ADR-0016：代收款項採用獨立於 `Payment` 之外的「尚未繳回」追蹤層，
確認繳回後才建立正式 `Payment`

**狀態**：已採納（V7.2）

**背景**：三玄宮實務上常見代辦人/志工先向功德主收了款項，但現金還
沒有交回宮方——如果系統這時候就比照【立即收款】或快速收款
（ADR-0013、ADR-0014）立即建立 `Payment`、更新 `FinancialAccount`
餘額，會讓帳上現金餘額超過宮方實際能動用的金額，違反這個專案從第一輪
就定案的核心原則「未實際收到的款項，不可增加現金或銀行餘額」。

**決策**：

1. **新增 `AgentRemittance` 模型，作為 `Payment` 之外的獨立追蹤層**：
   一筆 `AgentRemittance` 記錄「某代收人向某功德主收了多少錢、還沒
   交回」這個事實，連回既有的 `RegistrationGroup`/`Receivable`，但
   **完全不觸碰 `LedgerEntry`/`FinancialAccount`**——這是跟
   `QuickCollectionEntry`（ADR-0014）刻意不同的設計：快速收款是「宮
   方當場已經拿到錢，只是報名資料不完整」，立即建立 `Payment` 天經
   地義；代收待繳回是「宮方根本還沒拿到錢」，絕對不能提前入帳。
2. **代收人是「付款人」角色的延伸應用，不是新概念**：直接沿用
   `docs/COLLECTION_CENTER.md` ADR 已定案的「功德主與付款人分開」
   設計（`Payment.payerNameSnapshot`），代收人在確認繳回後就是
   `Payment.payerNameSnapshot` 的值，不需要另外設計一套代收人專屬的
   付款人欄位。
3. **狀態機只有管理者的明確動作才能推進**（`PENDING`→`REMITTED`／
   `DISPUTED`），不會因為提醒次數、時間經過而自動轉換——這是刻意的
   保護設計，避免系統「自動假設」代收人已經把錢交回來，同一個原則
   延伸自 ADR-0013/0014「溢收不可自行吞掉差額」的精神：**任何影響
   現金/銀行餘額判斷的動作，都必須由人明確確認，系統不可以自己猜**。
4. **不修改報名畫面既有的兩個按鈕**：登記代收款項是管理者對一筆既有
   `Receivable` 的補充操作（在收款中心/報名詳細畫面新增「登記代收
   款項」動作），不是報名當下的第三個選項，避免影響 ADR-0013/0014
   已經定案的報名流程。
5. **提醒排程（每週三/五 19:30，累加 `reminderCount`，至少持續 5 輪
   且之後不自動停止）本身只更新 `AgentRemittance` 自己的欄位**，不會
   觸發任何財務相關的資料變動，確保提醒機制跟財務入帳邏輯完全解耦。

**後果**：
- 優點：宮方帳上現金餘額永遠反映「宮方實際能動用的金額」，不會因為
  代收人手上還握有現金就提前灌水；代收人概念直接沿用既有付款人欄位
  設計，不需要新增平行的角色概念；提醒機制跟財務入帳邏輯解耦，之後
  調整提醒排程規則（例如改成每週一次）不會動到任何財務計算邏輯。
- 代價：一筆已經被代收人收走但尚未繳回的款項，在 `Receivable` 層級
  仍然顯示為未繳/部分繳款（因為還沒有對應的 `Payment`），需要在畫面
  上額外提示「其中 NT$ X 已由代收人收取、尚未繳回」，避免管理者誤判
  這筆應收完全沒有任何進度（見 `docs/AGENT_REMITTANCE.md` 第十六節
  尚待確認事項第 5 項）。
- 本輪只定案資料模型與規格文字，完整 Schema、狀態機、流程圖見
  `docs/AGENT_REMITTANCE.md`，**不修改 `prisma/schema.prisma`、
  不建立 migration**。

> **後續發展（V7.1.1）**：本 ADR 的 `AgentRemittance`/3 態狀態機，在
> V7.1.1 已經演進為更完整的「雙軌帳本」設計，見 ADR-0017、ADR-0018
> 與 `docs/AGENT_COLLECTION_RECONCILIATION.md`。本 ADR 保留作為決策
> 演進脈絡，不再是實作依據。

## ADR-0017：信眾已付款不等於三玄宮已持有款項（雙軌帳本設計）

**狀態**：已採納（V7.1.1）

**背景**：ADR-0016 的 `AgentRemittance` 設計，代收款項在繳回確認前
完全不影響 `Receivable`，這代表信眾其實已經付款、但應收畫面仍然顯示
「未繳」，容易讓行政人員誤判進度、也難以支援「部分繳回」與「多筆
代收如何呈現在應收金額上」這兩個 V7.2 遺留的尚待確認事項。V7.1.1
需要更精確地區分「信眾是否已經付款」跟「宮方是否已經持有這筆現金」
兩件事，這是兩個獨立的事實，不能用同一個欄位表示。

**決策**：

1. **新增 `AgentCollectionItem` 取代 `AgentRemittance`**，記錄代收
   款項的完整生命週期（4 種狀態：`PENDING`/`PARTIAL`/`REMITTED`/
   `DISPUTED`），一建立就視為「信眾已付款」，**立刻反映在
   `Receivable.paidAmount`**（Track A：信眾付款帳本）。
2. **`FinancialAccount.balance` 與 `LedgerEntry`（Track B：宮方持有
   帳本）完全獨立更新**，只有在 `ReconciliationBatch` 被確認
   （`status=CONFIRMED`）之後，才會透過 `Payment`（帶
   `collectionItemId`）觸發 Track B 的餘額異動。
3. **兩軌之間允許暫時性落差**：信眾付款總額（Track A）可能大於宮方
   實際持有金額（Track B），這個落差就是「代收人保管中總額」，在
   首頁提醒卡與月底財務報表都需要明確呈現這個數字（見
   `docs/MONTHLY_FINANCIAL_REPORT.md` 第三節第 11 項），不能被隱藏
   或跟已入帳金額混在一起。
4. **這是刻意的複雜度增加，不是過度設計**：ADR-0016 選擇的「代收款項
   完全不碰 `Receivable`」雖然簡單，但無法回答「這筆應收到底收了
   多少」這個行政人員最常問的問題；雙軌帳本用兩個獨立但都即時更新的
   數字，同時滿足「信眾繳了多少」與「宮方拿到多少」兩種不同但都重要
   的行政需求。

**後果**：
- 優點：應收畫面可以正確顯示「已收金額」（含代收人尚未繳回部分），
  同時銀行/現金餘額不會提前灌水；支援部分繳回（`PARTIAL` 狀態）；
  月底報表可以明確拆出「代收人保管中總額」這個獨立數字，不再是
  ADR-0016 遺留的尚待確認事項。
- 代價：需要在 `Payment` 建立邏輯新增判斷（`collectionItemId` 存在
  時跳過重複更新 `Receivable.paidAmount`，見 ADR-0018），比 ADR-0016
  單純很多的設計多了一個需要小心處理的邊界情況。

## ADR-0018：只有對帳確認後才能轉入正式帳戶，且沿用同一筆 `Payment`
概念不重複計算

**狀態**：已採納（V7.1.1）

**背景**：ADR-0017 把代收款項拆成雙軌帳本後，衍生一個新問題：對帳
確認時要建立 `Payment` 觸發 Track B 餘額更新，但 `Payment` 建立的
既有邏輯（ADR-0013/0014）預設會同時更新 `Receivable.paidAmount`——
如果對帳確認時的 `Payment` 也照舊更新 `Receivable.paidAmount`，會
造成同一筆錢在 Track A 被計算兩次（一次在 `AgentCollectionItem`
建立時，一次在對帳確認建立 `Payment` 時）。

**決策**：

1. **`Payment` 新增 `collectionItemId`（nullable，非唯一）欄位**，
   標示這筆 `Payment` 是從代收對帳確認產生的，不是報名當下或快速
   收款產生的。
2. **`Payment` 建立邏輯新增判斷：`collectionItemId` 有值時，跳過
   `Receivable.paidAmount` 的重複累加**，因為這筆金額已經在
   `AgentCollectionItem` 建立當下計入過 Track A；這筆 `Payment` 的
   唯一職責是觸發 `LedgerEntry`/`FinancialAccount` 更新（Track B）。
3. **`Payment` 同時新增 `reconciliationBatchId`（nullable）欄位**，
   讓一筆對帳批次可以回溯查到它產生的所有 `Payment`，支援月底報表
   對帳、支援作廢/補印時的追蹤。
4. **對帳批次確認是唯一能把代收款項轉入正式帳戶的路徑**：沒有
   `ReconciliationBatch.status=CONFIRMED`，就不會有對應的 `Payment`，
   `FinancialAccount.balance` 就不會異動，這個規則跟 ADR-0016 的核心
   原則「未實際收到的款項，不可增加現金或銀行餘額」一致，只是把
   「實際收到」的判定點從「代收人交回現金」精確到「行政人員完成
   對帳確認」這個更明確的動作。

**後果**：
- 優點：同一筆錢在雙軌帳本中都只計算一次，不會有 Track A 或 Track B
  其中一邊被重複累加的風險；`Payment` 沿用既有欄位與既有查詢邏輯
  （收據、對帳查詢），不需要另外設計一套代收專屬的入帳機制。
- 代價：`Payment` 建立邏輯需要新增一個分支判斷，若未來有其他來源的
  `Payment` 也需要類似的「跳過重複累加」邏輯，需要注意判斷條件不要
  互相衝突（本輪只涵蓋 `collectionItemId` 這一種情境）。

## ADR-0019：月底報表先產生草稿，人工確認後才鎖定

**狀態**：已採納（V7.1.1）

**背景**：月底財務報表如果完全自動鎖定，遲到的資料（例如月底最後
一天補登記的代收款項、隔天才送達的銀行對帳單）就無法被正確反映，
但如果完全不自動化，行政人員每個月都要記得手動產生報表，容易遺漏。
需要一個「自動產生但不自動生效」的機制。

**決策**：

1. **`MonthlyFinancialReport` 採用 5 態狀態機**：`DRAFT`（系統自動
   產生）→ `PENDING_RECONCILIATION`（行政人員開始核對）→
   `CONFIRMED`（核對完成）→ `LOCKED`（正式鎖定，終態）；`VOID`
   （作廢，任何非終態都可以轉入）。
2. **只有 `DRAFT` 是系統自動產生，`CONFIRMED`/`LOCKED` 都必須是人工
   明確動作**，系統不會自動把 `DRAFT` 推進到後續狀態，即使已經超過
   一個月都不會自動確認或鎖定。
3. **報表核心欄位使用 `Json` 快照**（`financialSummarySnapshot` 等
   5 個快照欄位），在 `CONFIRMED` 時就凍結計算結果，之後即使
   `Receivable`/`Payment`/`LedgerEntry` 等來源資料被其他操作異動，
   已凍結的報表數字也不會跟著改變——這是延伸 ADR-0010
   `ActivitySummary` 的「快取/彙總表」哲學，但比 ActivitySummary
   更進一步，是完全不重新計算的「歷史凍結紀錄」。
4. **排程採用「每日凌晨檢查是否為每月 1 日」而非「每月固定觸發一次」
   的 cron 設計**：前者對單次觸發失敗有容錯（隔天凌晨還會再檢查一
   次），但需要額外邏輯確保重複執行不會重複產生報表（見
   `ScheduledJobRun`/`idempotencyKey`），這個取捨已在
   `docs/MONTHLY_FINANCIAL_REPORT.md` 第十九節標註為待你確認事項。

**後果**：
- 優點：草稿自動產生降低行政人員的手動負擔，同時鎖定前的任何階段
  都還有機會修正；凍結快照確保已確認/鎖定的報表數字不會被後續操作
  悄悄改變，符合財務報表「一旦確認就不能無聲無息變動」的基本要求。
- 代價：需要額外的 `ScheduledJobRun` 追蹤機制確保排程不會重複執行；
  `Json` 快照欄位犧牲了關聯式查詢的彈性（例如無法直接用 SQL 篩選
  快照內某個活動的金額），這是刻意的取捨，換取「歷史紀錄不可變」
  的保證。

## ADR-0020：已鎖定報表不得被遲到資料覆蓋，一律走 `PeriodAdjustment`

**狀態**：已採納（V7.1.1）

**背景**：報表鎖定後，難免還是會發現需要更正的情況（例如鎖定後才
發現某筆款項分類錯誤、或某個月的活動收入需要跨月重新分攤）。如果
允許直接修改已鎖定的報表或回頭修改當月的 `LedgerEntry`，會破壞
ADR-0019「鎖定後不可變」的保證，也會讓已經核對過的報表數字失去
可信度。

**決策**：

1. **新增 `PeriodAdjustment` 模型**，記錄「原始期間」「登錄期間」
   「調整金額」「調整原因」「建立人」「核准人」（建立人與核准人
   分開，避免自己調整自己核准），狀態機
   `PENDING_APPROVAL`/`APPROVED`/`REJECTED`/`APPLIED`。
2. **已鎖定的報表本身永遠不會被直接修改**，跨月調整一律計入**登錄
   當下所在月份**的報表（透過 `appliedToReportId` 關聯到套用調整的
   報表），並在該月報表的「前期調整金額」欄位（月底報表 20 個項目
   之一）明確揭露這筆錢是修正哪個期間的錯誤，不會偷偷併入當月正常
   收支。
3. **核准與套用分開**：`PeriodAdjustment` 核准後不會立刻自動套用到
   報表，要等到下一次月底報表產生（或行政人員手動觸發重新計算
   `DRAFT`）時才會反映，避免在報表核對過程中資料還在變動。
4. **這個規則同時保護代收對帳與月底報表兩邊**：`ReconciliationBatch`
   一旦 `CONFIRMED` 也不可以被直接修改，若對帳後才發現有誤，同樣
   走跨月調整或建立新的更正對帳批次（並在原批次備註關聯），不會
   直接改動已確認的批次。

**後果**：
- 優點：已鎖定/已確認的財務紀錄有明確的「不可變」保證，任何更正都
  留下完整的審計軌跡（誰發現問題、誰核准、套用到哪個月）；跨月調整
  的欄位設計同時適用於代收對帳與月底報表兩種情境，不需要重複設計。
- 代價：更正流程比直接修改多了「登記調整→核准→套用」三個步驟，
  對於行政人員來說比「打開報表直接改數字」麻煩，但這是財務資料
  正確性與可稽核性的必要代價，也是本輪明確要求的行為（「已鎖定
  報表不得被遲到資料覆蓋」）。
