## 三、收據以正式收款為唯一來源（對應指令三）——結構性強制，不是靠慣例

這是本輪最核心的架構要求，處理方式是**在 Prisma Schema 層級就不給第二套系統存在的
可能**：

- `Receipt` 沒有任何欄位直接指向 `OfferingClaim`／`PurificationEntry`／
  `UniversalSalvationDetail`／手動應收；唯一的正式來源是
  `Receipt.paymentTransactionId → PaymentTransaction`。
- `ReceiptLine.paymentAllocationId → PaymentAllocation` 是**每一筆收據明細**回查原始
  來源的唯一路徑（`ReceiptLine.sourceType`／`sourceId` 只是從對應的 `PaymentAllocation`
  複製過來的顯示用快照欄位，不是另開一條可以繞過收款資料的路）。
- 因此每張收據都能回查：收款主紀錄（`paymentTransactionId`）、收款分配明細
  （`ReceiptLine.paymentAllocationId`）、原始應收來源（透過 `PaymentAllocation.sourceType`／
  `sourceId`）、信眾或家戶（`householdId`／`memberId`）、所屬活動（`ReceiptLine.activityId`）、
  經手人（`createdByName`）、付款方式（透過關聯的 `PaymentTransaction.paymentMethod`）。

這不是靠程式邏輯「盡量」做到，而是資料庫外鍵結構本身就沒有讓收據繞過收款資料的欄位存在。

## 五、收據開立方式與防重複開立（對應指令五、十三）——本輪自己抓到並修正一個真實邏輯錯誤

系統支援指令要求的三種開立方式：暫不開立（`markNoReceiptRequired`）、合併開立
（`issueReceipt` 傳入同一 `paymentTransactionId` 底下多筆 `PaymentAllocation`）、
分項開立（對每個 `PaymentAllocation` 各自呼叫一次 `issueReceipt`）。

**防止超額開立的核心公式**（`computeReceiptableRemaining`，`src/lib/receiptRules.ts`）：

```
尚可開立收據金額 = 該筆 PaymentAllocation 金額
                 − 退款/轉款/作廢等調整扣減
                 − 已經有效佔用的收據金額
```

「已經有效佔用的收據金額」這一項，**在驗證階段才發現原本的寫法是錯的**：最初的實作
只排除 `status === "VOIDED"` 的收據，換開（`REPLACED`）沒有被排除。用真實 PostgreSQL
情境測試驗證「情境8：換開收據」之後，用獨立 SQL 重新核對尚可開立金額，發現一筆
`PaymentAllocation`（`PA_RCPT_SPLIT_2`，金額 500）算出 `remaining = -500`——出現負數，
代表舊收據（已換開、狀態 `REPLACED`）跟新收據（換開後的新收據）的 500 元被重複算了
兩次，違反使用者自己在指令五、十六寫下的規則。

已修正 `src/lib/receipt.ts` 三處金額計算，新增 `isActiveConsumingReceiptStatus(status)`
輔助函式（只有 `ISSUED`／`NO_RECEIPT_REQUIRED` 視為有效佔用金額，`VOIDED` 與
`REPLACED` 都要排除），修正後重新核對同一筆資料，`remaining` 正確變回 `0.00`。
完整的修正前/修正後 SQL 對照，見交付檔案
`V11.1_實際操作與併發驗收_SQL輸出紀錄.txt` 第五節。修正後重新執行
`tsc --noEmit`（`receipt.ts` 0 非噪音錯誤）與全部自動測試（206/206 通過），確認沒有
因此修正引入其他問題。

系統開立收據時一律在資料庫交易內**即時重新計算**尚可開立金額（不信任前端傳來的
金額），避免兩位行政人員同時操作造成的競態條件（TOCTOU）。

## 七、收據號碼管理（對應指令七）——資料庫層級安全取號，不是「查總筆數+1」

沿用本專案在收款編號機制已經驗證過的安全取號模式（`PaymentSequenceCounter`），
新增 `ReceiptSequenceCounter`，透過：

```sql
INSERT INTO receipt_sequence_counters ("yearKey", "currentValue")
VALUES ($1, 1)
ON CONFLICT ("yearKey") DO UPDATE SET "currentValue" = "currentValue" + 1
RETURNING "currentValue"
```

在同一個資料庫交易內原子性取號，不是先查詢筆數再加一（那種寫法在併發下會拿到重複
號碼）。

**真實兩個 PostgreSQL session 併發測試**（不是邏輯推演）：Session A 取號後在交易中停留
1.5 秒，Session B 幾乎同時嘗試取號，實際等待 1111.406 ms 之後才拿到下一個號碼——
證明鎖定機制真的有效，不會兩邊拿到同一個號碼。完整輸出見交付檔案第三節。

預設格式：`R-西元年度-六位流水號`（例如 `R-2026-000001`），與使用者範例完全一致。
提供 `ReceiptNumberingConfig`（單例設定表）與設定畫面
（`/receipt-center/settings`，`ReceiptSettingsScreen.tsx`）：前綴、民國年/西元年切換、
流水號位數、每年重新起號/連續編號、起始號碼、格式預覽——全部欄位對照指令逐項核對
無遺漏。已作廢號碼不會重新使用（`ReceiptSequenceCounter` 只會遞增，不會因作廢而回退）；
換開一定會透過同一套取號機制產生全新號碼；原號碼在作廢/換開後永久保留在
`Receipt.receiptNumber` 上（狀態改變，但這個欄位本身不會被清空或覆蓋）。

權限上，`RECEIPT_PERMISSIONS`（`src/lib/permissions.ts`）目前設定為只有 `SUPER_ADMIN`
角色擁有 `manageNumbering` 權限，符合指令「只有最高管理權限可修改收據編號規則」；
但如同專案其他所有模組，目前系統還沒有登入機制（`getCurrentUser()` 目前固定回傳
`null`），所以這個權限矩陣**目前是定義好但還沒有真正的登入畫面可以強制執行**，
這點如實標註，跟前面幾輪的財務/供品權限矩陣狀態完全一致，不是本輪新產生的落差。

## 十、補印功能（對應指令十）

## 十一、收據作廢（對應指令十一）

`voidReceipt()` 強制要求 `voidReason` 與 `approvedByName`（核准人），拒絕對已經是
`VOIDED` 或 `REPLACED` 狀態的收據重複作廢。作廢後：`status` 改為 `VOIDED`，
`receiptNumber` 完全不變（永久保留，不重新使用）、原始收據內容（`ReceiptLine` 明細）
完全保留可查看、`voidedAt`／`voidedByName`／`approvedByName` 記錄完整。

**真實 PostgreSQL 情境測試驗證**（情境7）：作廢前後 `PaymentAllocation.amount` 完全
不變（800.00 → 800.00）、所屬 `PaymentTransaction.totalAmount` 完全不變
（1300.00 → 1300.00）——證明作廢收據**不會**自動退款、不會自動取消收款、不會影響
財務收入，這幾件事是完全獨立的動作，符合指令「收據作廢與收款退款是兩個不同動作」
的要求。

## 十二、收據換開（對應指令十二）

`reissueReceipt()` 完整實作七步驟：① 舊收據狀態改為 `REPLACED`（不是 `VOIDED`，
理由見第四節說明）② 原收據號碼與內容完整保留 ③ 透過安全取號機制建立新收據
④ 新收據取得全新號碼 ⑤ `originalReceiptId` 讓新舊收據互相關聯（可雙向查詢）
⑥ 強制要求填寫換開原因（`reason`）與核准人（`approvedByName`）⑦ **明確拒絕**任何
明細金額被更改的換開請求——如果 `lineOverrides` 裡任何一項金額與原收據不同，會
直接拋出錯誤：「換開收據不得變更明細金額，收款金額不得重複計算——如需調整金額請走
收款中心的退款/轉款流程」，只允許更改品名/付款人顯示文字這類非金額欄位。

**真實 PostgreSQL 情境測試驗證**（情境8）：換開後舊收據狀態正確變成 `REPLACED`
（不是 `VOIDED`）、原號碼保留；新收據狀態 `ISSUED`、全新號碼、金額與舊收據完全相同
（500.00）、`originalReceiptId` 正確指向舊收據 ID。

## 十三、部分收款與收據（對應指令十三）

`computeReceiptableRemaining()` 天生支援部分收款分次開立：每次開立收據時只消耗
實際已收金額中「尚未被有效收據佔用」的部分，不要求第一次付款就開全額收據。

**真實 PostgreSQL 情境測試驗證**（情境4）：模擬同一應收項目分兩次付款（第一次3000、
第二次2000），各自開立收據後，兩張收據合計 5000，與 `UniversalSalvationDetail`
實際已收金額（`amountPaid = 5000`）完全一致，不多不少。

## 十四、退款與收據關係（對應指令十四）——軟性確認機制，不做自動連動

修改 `src/lib/collectionCenter.ts`：`createAllocationAdjustment()`（退款/轉款）與
`voidPaymentTransaction()`（作廢收款）在執行前，會呼叫新增的
`getAllocationReceiptImpact()` 檢查該筆收款是否已經有有效收據；如果有，且呼叫端沒有
帶入 `acknowledgeReceiptImpact: true`，會回傳 HTTP 409，附上受影響的收據號碼清單與
金額，訊息為：「此筆付款已開立收據，退款後需確認是否作廢或換開收據。」——如果帶入
確認參數，同一個請求就會正常執行。

這個設計刻意做成**可重新送出的軟性確認**，而不是硬性擋下：滿足指令「系統必須提示」
的要求，但不會自動作廢收據、不會自動刪除收據、也不會阻擋行政人員真正需要執行的
退款/作廢操作。對應的兩支 API 路由（`.../payments/[id]/void/route.ts`、
`.../allocations/[id]/adjustments/route.ts`）已同步修改，接受並轉傳這個確認參數。

## 十七、收據與財務中心界線（對應指令十七）

正式財務收入來源維持是收款中心（`PaymentTransaction`/`PaymentAllocation`），不是
收據中心。開立、補印、作廢、換開收據、修改備註、匯出 PDF、列印，這些動作在
`src/lib/receipt.ts` 裡**完全沒有寫入或修改任何** `PaymentTransaction.totalAmount`、
`PaymentAllocation.amount` 等會影響財務收入計算的欄位（唯一寫回 `PaymentAllocation`
的欄位是 `receiptStatus`／`receiptNumber`，純粹是顯示用的收據連結狀態標記，不是金額
欄位）。第十一節的真實資料庫測試已經直接驗證了這個邊界（作廢前後付款金額完全不變）。
