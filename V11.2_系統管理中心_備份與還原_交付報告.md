## 一、這一輪做了什麼

依照指令《台北三玄宮 ERP V2－V11.2 系統管理中心（Backup & Restore Center）
正式開發指令》的十七個章節，完整開發了：

- 新增主選單【系統管理】，7 個子項目：備份中心／還原中心／Google Drive連線／
  系統版本／系統健康檢查／系統Log／系統設定（對應指令「一」）。
- Google Drive 固定綁定機制：跟目前 Chrome 登入的帳號完全無關，授權資訊永久
  記錄在資料庫（對應指令「二」）。
- 備份資料夾結構 `三玄宮ERP_Backup/{Daily,Weekly,Monthly,Before_Update}`，
  程式自動建立，不需要使用者手動建資料夾（對應指令「三」）。
- 立即備份（涵蓋 17 項內容）、每日/每週/每月自動備份排程、更新前自動備份、
  一鍵還原（含二次確認＋輸入完整檔名的雙重防呆）、備份下載、首頁狀態卡片、
  備份Log、系統健康檢查、權限限制在 SUPER_ADMIN（對應指令「四」至「十四」）。

**沒有**開發 V12 財務中心，也**沒有**自行新增任何指令以外的宮務流程。

## 二、系統架構總覽

```
┌─────────────────────────────────────────────────────────────────────┐
│                          瀏覽器（Client）                             │
│  /system-center 主選單 → 7 個子頁面（皆為 Client Component）          │
│  身分：useOperator()／localStorage（沿用 V11.1.1 既有機制，不是登入） │
└───────────────────────────────┬───────────────────────────────────────┘
                                 │ fetch()，帶 operatorUserId
                                 ▼
┌─────────────────────────────────────────────────────────────────────┐
│               /api/system-center/* （Next.js Route Handlers）        │
│  每一支都呼叫 assertSystemPermissionForOperator() 重新查資料庫驗證     │
│  角色是不是 SUPER_ADMIN，前端隱藏按鈕只是體驗優化，不是安全機制。      │
└───────┬───────────────┬───────────────┬───────────────┬──────────────┘
        │               │               │               │
        ▼               ▼               ▼               ▼
  src/lib/backup.ts  src/lib/restore.ts  src/lib/googleDrive.ts  PostgreSQL
  (pg_dump/zip/上傳)  (下載/unzip/       (OAuth／上傳／下載／     (BackupLog／
                        pg_restore)       資料夾管理／健康檢查)    GoogleDriveConnection／
                                                                   SystemSetting)
```

**零新增 npm 套件的設計**：Google Drive 串接用原生 `fetch()`；refresh token
加密用內建 `node:crypto`（AES-256-GCM）；壓縮/解壓縮用系統 `zip`/`unzip`
指令（`child_process.execFile`）；資料庫備份/還原用系統 `pg_dump`/`pg_restore`
指令，而不是額外安裝 JS 套件或手刻橫跨 43+ 個 Prisma model 的 JSON 匯出。
這是刻意的工程選擇：這個沙盒環境從 V10.1 到現在從未能成功 `npm install`，
把整個備份功能設計成「零新增依賴」，讓它在任何已經能跑通現有專案的環境
（不需要額外安�裝任何套件）裡都能直接運作，降低風險。

## 三、備份流程圖（Backup Flow）

```
使用者按【立即備份】
        │
        ▼
POST /api/system-center/backup/run { operatorUserId }
        │
        ├─ assertSystemPermissionForOperator(userId, "runBackup")
        │     └─ 非 SUPER_ADMIN → 403，流程中止
        ▼
createBackup({ type: "MANUAL", executedByName, isAutomatic:false })
        │
        ├─ 1. BackupLog 建立一筆 IN_PROGRESS 紀錄
        │
        ├─ 2. 建立暫存目錄，依序產生：
        │      - database.dump  ← pg_dump --format=custom（PostgreSQL資料庫）
        │      - schema.prisma  ← 複製現有檔案（Prisma Schema版本）
        │      - migrations.json ← 查詢 _prisma_migrations（含收據流水號等
        │                            狀態都在資料庫欄位裡，隨資料庫一起備份）
        │      - version.json   ← package.json 版本（系統版本）
        │      - permissions-snapshot.json ← getFullPermissionSnapshot()
        │                            （使用者／權限矩陣快照）
        │      - uploads/       ← 已上傳圖片／PDF（目前為 no-op，見「已知
        │                            限制」一節）
        │      - manifest.json  ← 這份備份包含哪些檔案的清單
        │
        ├─ 3. zip -r 打包成 Backup_YYYY-MM-DD_HHMM.zip
        │
        ├─ 4. 呼叫 googleDrive.uploadFile() 上傳到
        │      三玄宮ERP_Backup／Daily（若資料夾不存在，
        │      ensureFolderStructure() 會先自動建立）
        │
        ├─ 5. BackupLog 更新為 SUCCESS（含檔名/大小/GDrive位置）
        │      或 FAILED（含失敗原因）
        │
        └─ 6. applyRetentionPolicy()：查詢 Google Drive Daily/Weekly
               資料夾實際內容，超過保留天數/週數的自動刪除（Monthly／
               Before_Update 永久保留，不執行這一步）
        │
        ▼
回傳結果給前端，畫面顯示 ✅ 成功／❌ 失敗＋原因
```

**自動備份**（每日 02:00／每週日 03:00／每月 1 號 04:00）走同一條
`createBackup()` 邏輯，差別只在：
1. 由外部排程服務呼叫 `POST /api/system-center/backup/scheduled-trigger?type=daily`
   （需要正確的 `x-backup-cron-secret` header），而不是使用者手動按按鈕。
2. `isAutomatic: true`、`executedByName: "系統排程（自動）"`。
3. `type` 對應 `DAILY`/`WEEKLY`/`MONTHLY`，上傳到對應的 Drive 子資料夾。

**更新前備份**（`scripts/preDeployBackup.ts`）在 `render.yaml` 的
`startCommand` 裡排在 `next start` 之前執行，`type: "BEFORE_UPDATE"`；
備份失敗時 `process.exit(1)`，讓整個部署流程（包含後面的
`migrate:deploy`／`next start`）中止，滿足指令「八」「備份失敗，不得
繼續更新」的要求。

## 四、還原流程圖（Restore Flow）

```
使用者在【還原中心】選擇資料夾（Daily/Weekly/Monthly/Before_Update）
        │
        ▼
GET /api/system-center/backup/browse?folder=Daily
        │  （驗證 viewSystemCenter 權限 → 呼叫 googleDrive 查詢該資料夾
        │    實際檔案清單，直接反映 Google Drive 當下真實內容）
        ▼
畫面列出檔案（檔名／大小／建立時間），使用者點【還原】
        │
        ▼
┌─────────────── 第一層防呆：確認彈窗 ───────────────┐
│ 「⚠️ 是否確定覆蓋目前資料？」                        │
│ 說明：即將完整覆蓋所有資料，這個動作無法復原           │
└─────────────────────┬───────────────────────────────┘
                       ▼
┌─────────────── 第二層防呆：輸入完整檔名 ─────────────┐
│ 使用者必須手動輸入跟選擇的備份完全相同的檔名，          │
│ 【確定還原】按鈕在輸入相符前保持 disabled              │
└─────────────────────┬───────────────────────────────┘
                       ▼
POST /api/system-center/backup/restore
      { operatorUserId, googleDriveFileId, fileName, confirmFileName }
                       │
                       ├─ 基本欄位驗證（route.ts）
                       ▼
restoreFromBackup()（src/lib/restore.ts）
                       │
                       ├─ assertSystemPermissionForOperator(userId, "restoreBackup")
                       │     └─ 非 SUPER_ADMIN → 403，中止
                       │
                       ├─ confirmFileName !== fileName → 400，中止
                       │     （伺服器端再次檢查，不是只靠前端擋）
                       │
                       ├─ getActiveAccessToken() + downloadFile()
                       │     從 Google Drive 下載該份 zip
                       │
                       ├─ unzip -o 解壓縮
                       │
                       ├─ pg_restore --clean --if-exists
                       │     --single-transaction --no-owner --no-privileges
                       │     --dbname <DATABASE_URL> database.dump
                       │     （單一交易：要嘛全部套用成功，要嘛整個回滾，
                       │      不會出現「只還原一半資料」的中間狀態，
                       │      對應指令「九」「不得只還原部分資料」）
                       │
                       └─ recordVersion({ entityType:"SystemRestore",
                             action:"RESTORE", operatorName, changeNote })
                             寫入稽核紀錄
                       │
                       ▼
畫面顯示「✅ 還原完成，資料已覆蓋為所選備份的內容」
```

## 六、已知限制與範圍說明（誠實揭露，非隱藏）

1. **已上傳圖片／PDF 備份目前是 no-op**：`copyUploadedFilesIfAny()` 目前
   不會真的複製任何檔案，因為這個專案目前沒有一個統一的「使用者上傳檔案」
   持久化儲存目錄可以讀取（收據 PDF 目前是前端即時產生，不是伺服器端存檔）。
   程式碼預留了 `UPLOAD_STORAGE_DIR` 環境變數，未來如果新增了持久化上傳
   儲存，只要設定這個環境變數，備份就會自動涵蓋。這是誠實揭露的範圍限制，
   不是自行縮減需求——manifest.json 會忠實記錄「這次備份沒有上傳檔案內容」。

2. **CSRF `state` 一次性驗證用記憶體 `Map`**：只在單一、長時間執行的
   Node 行程下正確（符合 Render 免費方案單一服務的部署模型）；如果未來
   改成多執行個體或無伺服器（serverless）部署，這個機制需要換成資料庫或
   Redis 儲存，目前的程式碼註解已明確寫出這個限制。

3. **排程機制依賴外部服務呼叫**：Render 免費方案沒有內建 Cron，本輪把
   排程動作做成受保護的 HTTP API，需要使用者自行選擇一個外部排程來源
   （付費 Render Cron Job，或免費的 cron-job.org／GitHub Actions），
   詳見 `render.yaml` 裡的完整說明。這是本輪的建議與程式碼準備，實際
   選哪一種排程來源需要使用者決定。

4. **系統管理中心整個選單的權限比其餘模組更嚴格**：只有 `SUPER_ADMIN`
   能看到／操作，包含一般 `ADMIN` 在內都不行。指令「十四」只明確提到
   「一般使用者不得看到Backup」，但沒有逐字列出「ADMIN 算不算一般使用者」，
   這裡採用比較保守的解讀（連 ADMIN 也不開放），如果之後需要讓 ADMIN
   也能看某些唯讀項目（例如系統版本、健康檢查），請明確告知，這屬於
   需要另外確認的範圍。

5. **Google Drive 帳號憑證管理方式**（回應使用者在 UI 開發期間提出的
   問題）：`GOOGLE_OAUTH_CLIENT_ID`／`GOOGLE_OAUTH_CLIENT_SECRET`／
   `GOOGLE_OAUTH_REDIRECT_URI` 是環境變數，程式碼與 Git 都不會出現實際值；
   Refresh Token 則是動態值（每次授权/Google輪替都可能改變），刻意不做成
   環境變數（Render 執行中的服務無法自己改環境變數，會讓「重新授權」
   「撤銷」等指令要求的功能無法動態運作），改為加密後存進資料庫，
   加密金鑰本身才是環境變數（`GOOGLE_TOKEN_ENCRYPTION_KEY`）——資料庫裡
   看到的是密文，沒有這組金鑰無法還原，同樣滿足「不寫死、不進 Git」。
