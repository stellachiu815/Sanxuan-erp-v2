## 一、與 V7.0～V7.1.1 舊版規格文件的關係（重要，請先閱讀）

在動手設計之前，我先完整檢查了現有 Prisma Schema、migration、API 與付款流程（依你的指示）。過程中發現專案裡還留著 `docs/COLLECTION_CENTER.md`（V7.1）、`docs/AGENT_REMITTANCE.md`（V7.2）、`docs/AGENT_COLLECTION_RECONCILIATION.md`／`docs/MONTHLY_FINANCIAL_REPORT.md`（V7.1.1）這幾份文件名稱看起來很像本次需求的舊規格文件。

我確認了這幾份文件**從頭到尾都明確標註「規格定案，尚未實作」**，而且我逐一比對 `prisma/schema.prisma`，確認裡面提到的 `Payment`／`Receivable`／`FinancialAccount`／`LedgerEntry`／`RegistrationGroup`／`RegistrationItem`／`FamilyLampGroup`／`QuickCollectionEntry`／`AgentRemittance`／`AgentCollectionItem`／`ReconciliationBatch`／`MonthlyFinancialReport` 這些模型，**沒有一個真正出現在現在的資料庫 schema 裡**——這條規劃路線在 V7.1.1 之後就沒有再繼續，實際開發改走了 V8.0（版本紀錄/回收區）→ V9.0（祭改）→ V9.1（附加列印項目）→ V10.0（宮務活動中心）→ V10.1（供品認捐中心）這條模組別、逐一落地的路線，也就是你這次明確要求我檢查的「現有 Prisma Schema、migration、API 與付款流程」。

因此，V11.0 這次是直接建立在**真正存在**的 V10.1 `OfferingClaim`／`OfferingPayment` 之上，**不是**依照 V7.1 那份未實作規格裡的 `Payment`／`Receivable` 模型去設計。這點在架構上非常關鍵，特此在報告最前面說明，避免你看到舊文件名稱以為兩者是同一套東西、或以為這次是重做 V7.1。若你之後想真的採用 V7.1 那條規劃路線（例如全面改用 `Payment`/`Receivable`），會是一個更大的架構變更，需要另外討論，本輪沒有動它們一絲一毫。

## 三、架構設計核心原則（對應你要求的「不得另外建立一套與既有資料互不相通的收款系統」）

1. **OfferingClaim 的收款分錄永遠只有一份**，寫在既有的 `offering_payments` 資料表。合併收款時，我在自己的資料庫交易（transaction）裡直接重新執行跟 `recordOfferingPayment()`／`refundOfferingClaim()` **完全相同的計算公式**（`round2`／`derivePaymentStatus`，這兩支函式是直接 import 自 `src/lib/offeringRules.ts`，不是重寫），因為那兩支既有函式各自會開一個獨立的資料庫交易，沒辦法巢狀共用同一個交易，所以無法直接呼叫——但寫入的資料表、欄位、計算方式跟既有邏輯逐行一致，不是另一套規則。
2. **一次真實收款事件只建立一筆 `PaymentTransaction`**，底下可能有多筆 `PaymentAllocation`。已用真實 PostgreSQL 資料驗證你舉例的情境（見下方第五節）。
3. **`ReceivableSourceType` 誠實區分「已串接」與「尚未串接」**：本輪只有 `OFFERING_CLAIM`（V10.1 供品認捐）與 `MANUAL`（收款中心自建的臨時應收項目）是真正可操作的來源；其餘 12 個列舉值（普渡贊普、祭改、平安燈、太歲燈、補庫、宮慶其他應收、神明聖誕、油香、功德、法會、誦經、其他宮務活動）只是列舉在 enum 裡的保留欄位，`isReceivableSourceWired()` 是唯一判斷「這個來源本輪是否真的可用」的地方，API 層會擋下對未串接來源建立收款的請求。
4. **收據中心／財務中心本輪都沒有真正串接**：`PaymentAllocation.receiptStatus` 固定是 `NOT_LINKED`（預留給 V11.1）；`financeSourceKey` 是預留的防重複入帳識別碼（格式 `{來源類型}:{來源id}:{交易id}`，有 unique 限制），但**沒有任何程式碼真正寫入財務中心資料表**——因為財務中心目前仍然沒有真正的流水帳新增畫面（見第七節）。

## 四、為什麼 UniversalSalvationDetail／PurificationEntry 這次沒有真正串接

調查結果（對應你「優先正式串接現有資料來源，不要假造未來模組資料」的指示）：

- **`UniversalSalvationDetail`**（普渡贊普）：有 `sponsorAmount` 等欄位，但**完全沒有收款分錄或應收/已收欄位**（沒有 `amountPaid`/`amountUnpaid`/`paymentStatus`），如果本輪就要串接，等於要先幫這個模型新增一整套付款追蹤欄位——這已經超出「串接現有來源」的範圍，變成「先幫另一個模組補開發付款功能」，不在本輪授權範圍內（你要求「不要擴增其他尚未確認的宮務模組」）。
- **`PurificationEntry`**（祭改）：有 `paymentStatus`／`paymentAmount`，但 `paymentAmount` 是單一欄位、透過 PATCH 直接覆蓋（沒有應收金額、沒有獨立收款分錄、沒有「每次收款留下一筆紀錄」的機制），跟 V10.1 `OfferingClaim` 的成熟度差距很大，直接串接會產生「這個來源的收款紀錄比其他來源少很多欄位」的不一致觀感，也一樣需要先擴充祭改模組本身。

這兩個都保留為 `ReceivableSourceType` 裡的列舉值（尚未串接），列在收款中心待收款項清單裡**不會出現**，避免顯示不完整或誤導的資料。若你確認要讓這兩個來源可以在收款中心真正收款，需要先個別幫它們補上應收/已收欄位（屬於祭改/普渡模組本身的擴充，不是收款中心的事），可以另外排一輪處理。

## 八、暫緩／簡化事項（誠實列出，不假裝已完成）

1. **月結報表匯出**：本輪只做了 JSON 預覽＋瀏覽器列印（`window.print()`）＋ CSV 下載（可用 Excel 開啟）。需求提到的正式 PDF 排版樣式，本輪沒有做——需要先確認正式報表版型（頁首/頁尾/宮徽等）才能開發，屬於「有需要時再做」的裝飾層，不影響資料正確性。
2. **收據中心串接**：只預留 `receiptStatus`／`receiptNumber` 欄位，固定 `NOT_LINKED`，如你指示留給 V11.1。
3. **正式財務中心串接**：只預留 `financeSourceKey` 防重複入帳識別碼欄位，沒有任何程式碼寫入財務中心資料表（因為財務中心還沒有真正的畫面）。
4. **退款/轉款的核准流程**：目前 `approvedByName` 是自由文字欄位（系統還沒有登入/session 機制，無法驗證核准人身份的真實性），跟專案裡其他 `operatorName`/`createdByName` 欄位的既有慣例一致，等系統做出登入機制後可以升級成真正的權限驗證。
5. **代收序號產生**：`transactionNo` 用「查詢目前年度筆數＋1」的方式產生，沒有做分散式鎖——小廟行政作業量不高，如果真的同時間有兩人送出收款造成序號衝突，資料庫的 unique 限制會擋下第二筆，使用者重新整理頁面重送即可，不會產生錯誤的重複序號。
6. **權限矩陣**：比照 `src/lib/permissions.ts` 既有的 `FinanceAction`/`PurificationAction` 慣例，本輪沒有新增對應的 `CollectionAction` 型別與矩陣——因為系統整體都還沒有登入/session 機制，加了也無法真正落地執行，等系統做出登入機制時，可以在同一支檔案裡用同樣的 pattern 補上（不影響現有程式碼結構）。
