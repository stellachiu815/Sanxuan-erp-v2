# V7.0「Finance Core（財務核心）」交付說明

本輪**沒有開發任何 UI、收款畫面、流水帳畫面、報表，沒有新增任何 migration，
沒有修改正式資料庫，沒有推送 GitHub，沒有觸發 Render 部署，沒有開發任何
API，沒有執行 Build**。目標是把 V6.2/V6.3 已經提案的財務規則，收斂成正式
定案的財務核心資料模型（九個模型）與完整 Schema。

完整內容見新文件 **`docs/FINANCE_CORE_SCHEMA.md`**——這是目前財務核心
Schema 的權威版本，以下是導覽摘要。

## 三、資料流程圖

見第四節，涵蓋四條流程：①設定價格 → 建立報名 → 產生應收（不碰銀行）→
收到真正款項 → 自動建立唯一一筆流水分錄 → 更新帳戶餘額 → 重新計算活動
統計；②「同一筆收款只入帳一次」的資料庫層級保證；③南巡/北巡車次/雙人房
確認後如何併入同一套主流程；④普渡如何在保留內部祭祀內容不變的前提下，
併入同一套主流程。

## 四、模組相依圖

見第五節，以圖示呈現九個財務核心模型彼此的上下游依賴關係（`ActivityYear`
最底層 → `RegistrationGroup`/`Item` → `Receivable` → `Payment` →

`TourVehicle`/`TourRoomCharge`/`TaiSuiYearZodiac`/`AuditLog`/`User`
這些橫向或獨立模組如何銜接主鏈。

## 五、哪些資料可以沿用

見第六節，重點：`finance_records` 實體資料表（既有、目前空表）直接沿用，
`LedgerEntry` 是它的正式改名+擴充，不是另建新表；`Household`/`Member`/
`RitualRecord`/`Role`/`User`/`AuditLog`/`permissions.ts` 全部可以直接
沿用；`ActivityType` enum 沿用並延伸三個新值；V6.2/V6.3 已提案的
`TourVehicle`/`TourRoomCharge`/`TaiSuiYearZodiac` 原封不動整合進本輪。

## 六、哪些資料要新增

見第七節，九個財務核心模型裡，`ActivityYear`/`ActivityPrice`/
`RegistrationGroup`/`RegistrationItem`/`Payment`/`FinancialAccount` 是
V6.2 就提案過、本輪定案的部分；`Receivable`（獨立資料表）跟
`ActivitySummary`（快取彙總表）是本輪相對於 V6.2 的**新決策**（V6.2
原本認為這兩者不需要獨立資料表，V7.0 需求明確要求獨立成表，等於解決了
V6.2 規格書留下的尚待確認事項）。

## 七、開發順序

見第八節，10 個階段：`ActivityYear`/`ActivityPrice` → `FinancialAccount`
→ 通用 `RegistrationGroup`/`RegistrationItem` → `Receivable` →
`Payment`+`LedgerEntry`（收款核心，建議測試最多） → 登入/session 機制 →
`ActivitySummary` → 普渡/南北巡特化 → 年度燈/犯太歲/延用去年報名 → 各
活動收款 UI/報表。

## 本輪對 V6.2 尚待確認事項的解決

V7.0 的需求明確回答了 V6.2 規格書留下的兩個關鍵開放問題：

1. **應收要不要獨立成表**——V6.2 原本建議做成彙總欄位，V7.0 明確要求
   獨立的 `Receivable` 模型，本輪已定案（`docs/ADR.md` ADR-0008）。
2. **活動統計要不要獨立成表**——V6.2 原本認為純查詢彙總就夠，V7.0 明確
   要求獨立的 `ActivitySummary` 模型，本輪已定案為快取表設計
   （`docs/ADR.md` ADR-0010）。

另外把流水帳分錄正式定名為 `LedgerEntry`（`docs/ADR.md` ADR-0009），
沿用既有 `finance_records` 實體表，不是另建新表。

其餘 V6.2/V6.3 尚未被本輪需求觸及的開放問題（普渡報名數量是否自動同步、
南北巡確認時機、代辦人自身是否算報名明細等）維持原本的建議做法，仍待你
在正式開發前確認。
