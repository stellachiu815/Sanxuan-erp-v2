【誠實說明：這份紀錄驗證的範圍】
src/lib/operator.ts 的 resolveOperator(userId) 實際程式碼是：

    const user = await prisma.user.findUnique({ where: { id: userId } });
    if (!user || !user.isActive) return null;
    return { id: user.id, name: user.name, role: user.role as Role };

因為這個沙盒環境沒有網路，無法安裝 @prisma/client，所以無法真的透過
Prisma／Next.js 執行這段 TypeScript 程式碼本身。這裡改用「等價的原始
SQL」直接對同一顆、同一份 schema 的真實資料庫執行，驗證的是「這個查詢
模式（依 id 查、檢查 isActive）在真實資料上的行為是否符合預期」，
不是驗證 TypeScript 程式碼的編譯或執行——程式碼本身的正確性另外由
V11.1.1_權限矩陣真實執行驗證結果.txt（這份是真的用 tsx 執行
src/lib/permissions.ts 原始碼，不是模擬）與人工程式碼審查佐證。

場景 1：有效且啟用的操作人員（模擬 resolveOperator('seed-super-admin')）
--------------------------------------------------------------------
> select id, name, role, "isActive" from users where id = 'seed-super-admin';

        id        |        name        |    role     | isActive
------------------+--------------------+-------------+----------
 seed-super-admin | 住持（最高管理員） | SUPER_ADMIN | t
(1 row)

判讀：user 存在且 isActive=true → resolveOperator() 會回傳
{ id: 'seed-super-admin', name: '住持（最高管理員）', role: 'SUPER_ADMIN' }，
符合預期。

--------------------------------------------------------------------
場景 2：存在但已停用的操作人員
（模擬 resolveOperator('seed-inactive-staff')，應視同「找不到」）
--------------------------------------------------------------------
> select id, name, role, "isActive" from users where id = 'seed-inactive-staff';

         id          |       name        | role  | isActive
---------------------+-------------------+-------+----------
 seed-inactive-staff | 已離職人員 測試用 | STAFF | f
(1 row)

判讀：查詢本身找得到這一列，但 isActive=false，依 resolveOperator() 的
`if (!user || !user.isActive) return null;` 判斷式，會回傳 null，
呼叫端（assertReceiptPermissionForOperator）會回 401「找不到有效的
操作人員身分」。也就是說「已停用的人員」即使拿到舊的 userId 也無法
再通過任何收據操作的權限檢查——這正是指令要求「未授權者即使直接呼叫
API，也必須回傳403並拒絕執行」的其中一種情境（人員已離職/停用）。

--------------------------------------------------------------------
場景 3：不存在的操作人員 id（模擬未授權呼叫端亂傳 userId）
--------------------------------------------------------------------
> select id, name, role, "isActive" from users where id = 'not-a-real-user-id';

 id | name | role | isActive
----+------+------+----------
(0 rows)

判讀：查無此人，user 為 null，resolveOperator() 回傳 null，
同樣導致 401。
