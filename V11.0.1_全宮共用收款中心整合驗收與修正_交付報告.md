## 4. 新增 migration

6. 新增 `payment_sequence_counters` 資料表，並用既有 `payment_transactions.transactionNo`

## 7. 普渡及贊普付款整合方式

**資料歸屬**：贊普是掛在既有 `UniversalSalvationDetail`（跟 `RitualRecord` 一對一），沒有
新增第二筆普渡紀錄，`isSponsor`/`sponsorAmount` 等既有欄位維持不變，只是**額外**加了
`amountDue/amountPaid/amountUnpaid` 三個欄位（`amountPaid` 從此只由付款分錄加總維護，
使用者不會直接填寫）。

**付款歷程**：新增 `UniversalSalvationPayment` 資料表（結構比照既有 `OfferingPayment`），
每一次收款/退款/轉款都是獨立一筆分錄，`amountPaid` 由 Adapter 重新加總所有分錄計算，
不會用單一欄位覆蓋前一次付款。

**跟 `ritual.ts` 既有維護邏輯的分工**：`src/lib/ritual.ts` 的
`updateUniversalSalvationDetail()` 負責在使用者修改 `sponsorAmount`（應收金額本身變動）
時同步 `amountDue/amountUnpaid`，但**絕對不會**去動 `amountPaid`——已收金額永遠只有
`receivableAdapters.ts` 的 `universalSalvationSponsorAdapter` 透過真正的付款分錄才能
異動，職責清楚切開，不會有兩邊都在改同一個欄位、互相打架的情況。

**畫面顯示的 `payerName`／`sourceUrl`**：贊普沒有獨立的成員層級資料（`isSponsor` 是掛在
「戶」層級），`payerName` 依序取 `yangshangName`（陽上姓名）→ 戶的 `contactName`（主要
聯絡人）→ 戶名稱；`sourceUrl` 導回 `/household/{householdId}`；如果用「信眾」（member）
搜尋來源，這個 Adapter 誠實地回傳空清單（不會強行湊一個不存在的 memberId 配對）。

**已用真實 PostgreSQL 驗證**：情境「未付款 → 部分付款（收 2000）→ 已付款（再收
3000）」，過程中付款分錄始終保留兩筆（沒有覆蓋），且超額收款會被原子條件式 UPDATE 擋下
（0 rows）。詳見第 17 項。

## 8. 祭改付款整合方式

**唯一祭祀資料原則**：`PurificationEntry` 仍然是唯一的祭改服務資料，沒有為了收款另外
複製第二筆紀錄。

**新增收費狀態欄位 `feeStatus`（`UNSET`/`CHARGEABLE`/`WAIVED`）**：這是這次特別針對
「不得將空白金額自動當成零元已付款」設計的三態欄位——`UNSET` 代表尚未決定是否收費，
`CHARGEABLE` 才會有 `amountDue`，`WAIVED` 是明確免收。`src/lib/purification.ts` 的
`updatePurificationRegistration()` 新增這個參數，並且規則是：設為 `CHARGEABLE` 卻沒有
填 `amountDue` 會直接回傳 400 錯誤「設定為收費時，請填寫應收金額」，不會自動當成 0 元。

**正式付款保存在 `PaymentTransaction`/`PaymentAllocation`**：新增 `PurificationPayment`
分錄資料表（結構同上），`purificationEntryAdapter.applyPayment()` 用原子條件式 UPDATE
同時更新 `amountPaid/amountUnpaid`，並**同步**既有 `paymentStatus`/`paymentAmount` 這兩個
舊欄位（讓還沒有機會改讀新欄位的既有祭改畫面，看到的資料不會顯得完全過時）——但新欄位
才是真正的權威資料來源，舊欄位只是同步顯示用途。

**祭改頁面顯示**：已收/未收/狀態一律由 `amountPaid/amountUnpaid` 加總取得，不再是單一
`paymentAmount` 覆蓋式寫入。

**已用真實 PostgreSQL 驗證**：情境「祭改分兩次付款」與「祭改與供品認捐合併付款（一筆
`PaymentTransaction` 底下兩筆 `PaymentAllocation`）」，詳見第 17 項。

## 9. 舊付款資料轉換方式

`prisma/migrations/20260716060000_collection_center_v2/migration.sql` 對既有
`purification_entries.paymentAmount`/`paymentStatus`（舊制單欄位覆蓋式付款）採取**誠實
分兩種情況**的轉換，不會為了方便就湊一個假的應收金額：

- **舊資料 `paymentStatus = PAID`**：可以合理推斷「收清代表應收金額就等於已收金額」，
  所以 `amountDue = amountPaid = 舊 paymentAmount`，`amountUnpaid = 0`，並補寫一筆
  `purification_payments` 分錄，備註標明「[V11.0.1 資料轉換：舊制付款金額搬移，原始
  paymentStatus=PAID]」。
- **舊資料 `paymentStatus = PARTIAL`**：真正的應收金額在舊制資料裡是無法還原的（舊制
  只記錄「目前收了多少」，沒有記錄「原本應該收多少」），這裡**刻意不猜測**，
  `amountDue` 留白（`NULL`），只搬移 `amountPaid`，並補寫一筆分錄，備註標明「應收金額
  尚待行政人員補填」，讓行政人員之後可以清楚知道哪些舊資料需要人工確認補齊，而不是被
  一個看起來正常、實際上是猜出來的數字誤導。

這個處理方式直接對應需求「不得將空白金額自動當成零元已付款」與整份專案一貫的誠實原則。

## 11. 防重複收款機制

每一筆分配透過 `PaymentAllocation.sourceType`/`sourceId`/`paymentTransactionId` 唯一
追蹤這次收款是哪個來源、哪個交易、金額多少。真正防止「兩人同時對同一筆應收收款」的機制
是每個 Adapter 的 `applyPayment()` 用**原子條件式 UPDATE**，例如：

```sql
UPDATE "purification_entries"
SET "amountPaid" = "amountPaid" + $amount,
    "amountUnpaid" = GREATEST(COALESCE("amountDue", 0) - ("amountPaid" + $amount), 0)
WHERE "id" = $sourceId AND "status" = 'ACTIVE' AND "feeStatus" = 'CHARGEABLE'
  AND "deletedAt" IS NULL AND "amountUnpaid" >= $amount
RETURNING ...
```

`WHERE` 子句本身就同時檢查「原始應收是否仍存在」「是否已取消」「是否為可收款狀態」
「本次金額是否超過最新未收金額」，PostgreSQL 保證這條 UPDATE 語句本身是原子操作——
兩個人同時對同一筆應收送出收款，同一時間只有一個人的 UPDATE 能真的把 `amountUnpaid`
扣下去，另一個人的 UPDATE 因為 `amountUnpaid` 已經不足會回傳 0 rows，程式碼偵測到
0 rows 之後才另外查詢一次資料判斷「是狀態不對還是金額不足」，回傳對應的錯誤訊息給
使用者，**不會**依賴前端畫面顯示的舊金額。所有分配全部包在同一個 `prisma.$transaction`
裡，任何一筆分配失敗（`applyPayment` 丟出例外）都會讓整筆合併收款回復，不會留下部分
寫入的髒資料。

**已用真實 PostgreSQL 驗證**：超額收款情境（贊普已收清後再收 1 元）確認 UPDATE 回傳
0 rows，見第 17 項。

## 12. 收款編號安全產生機制

新增 `PaymentSequenceCounter` 資料表（`year` 為主鍵，`currentValue` 為目前序號），取號
方式：

```sql
INSERT INTO "payment_sequence_counters" ("year", "currentValue")
VALUES ($year, 1)
ON CONFLICT ("year") DO UPDATE SET "currentValue" = "payment_sequence_counters"."currentValue" + 1
RETURNING "currentValue"
```

這一整條語句由 PostgreSQL 保證是原子操作（如果該年度還沒有列，直接建立為 1；如果已經
有列，原子遞增並取回新值），並且放在 `createMergedPaymentTransaction()` 同一個
`prisma.$transaction` 裡執行，取到序號後才建立 `PaymentTransaction`。兩個人同時收款，
一個人拿到 `currentValue = N`，另一個人一定拿到 `N+1`，不會有序號碰撞的情況，也就不需要
依賴 `transactionNo` 的 unique constraint 事後擋錯、要求使用者重新輸入整筆收款資料。

**已用真實 PostgreSQL 驗證**：連續取號 3 次（115 年度：3→4→5）與跨年度取號（116 年度
從 1 開始）都正確、互不影響，見第 17 項。

## 13. 代收對帳防重複機制

`createAgentReconciliation()` 改用 `SELECT ... FOR UPDATE` 在同一個資料庫交易裡先鎖定
「這個代收人目前所有待對帳的交易」，才計算應繳回金額、建立對帳紀錄：

```sql
SELECT "id", "totalAmount" FROM "payment_transactions"
WHERE "isAgentCollected" = true AND "agentName" = $agentName AND "status" = 'COMPLETED'
  AND "agentRemittanceStatus" IN ('PENDING', 'PARTIALLY_REMITTED')
FOR UPDATE
```

這些列在這個資料庫交易 commit 之前會保持鎖定狀態。如果另一個人同時對**同一個代收人**
送出第二個對帳批次，那個查詢會被擋住等待；等到可以繼續查詢時，
`agentRemittanceStatus IN ('PENDING','PARTIALLY_REMITTED')` 這個條件會重新核對，第一批
已經被改成 `RECONCILED` 的交易就不會再被第二個對帳批次認領到——不需要在資料庫額外新增
一個「對帳中」的中繼狀態，也不是只靠前端隱藏對帳按鈕。對帳完成後
`agentRemittanceStatus` 直接變成 `RECONCILED`，畫面上代收對帳頁查詢待對帳清單時這兩筆
自然就不會再出現，也就不可能「對帳完成後再次勾選繳回」。

**已用真實 PostgreSQL 驗證**：建立 2 筆代收交易、完成一次對帳後，再次查詢同一個代收人
的待對帳交易，結果為 0 筆，見第 17 項。
