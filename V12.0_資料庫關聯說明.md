# V12.0 資料庫關聯說明

對應指令「十五、資料庫設計要求」與「三、信眾資料來源原則」。

---

## 一、核心設計原則

**「信眾」＝既有的 `Member`（家戶成員），不是另一套獨立名單。**

`DevoteeProfile` 是 `Member` 的 1 對 1 延伸表，只存放 `Member` 原本沒有的欄位；姓名、性別、身份、生日（國曆/農曆）、生肖、是否往生一律直接讀 `Member`，**不重複儲存**。沒有 `DevoteeProfile` 的 `Member`（尚未被寫入過任何延伸資料的信眾）仍然可以正常出現在信眾名單／360 總覽，只是延伸欄位顯示預設值（`null`/`false`）。

## 二、新增的 5 張表

```
Member (既有) ──1:1(延遲建立)── DevoteeProfile (新增)
                                       │
                                       ├──1:N── DevoteeTagAssignment (新增) ──N:1── DevoteeTag (新增)
                                       ├──1:N── DevoteeInteraction (新增)
                                       └──1:N── DevoteeCareRecord (新增)
```

| 表名 | 用途 | 與既有資料的關聯 |
|---|---|---|
| `devotee_profiles` | 信眾延伸資料（手機/LINE/Email/公司/備註/停用狀態/關懷狀態快照） | `memberId` 唯一外鍵 → `members.id`（`ON DELETE CASCADE`）。**延遲建立**：只有第一次寫入延伸資料/標籤/互動紀錄時才會建立這筆資料，單純查看名單/360 總覽不會觸發建立（見 `getOrCreateDevoteeProfile()`）。 |
| `devotee_tags` | 標籤定義（11 個系統預設＋管理者自訂） | 獨立表，`name` 唯一。 |
| `devotee_tag_assignments` | 信眾與標籤的套用關係 | `devoteeProfileId` → `devotee_profiles.id`（CASCADE），`tagId` → `devotee_tags.id`（**RESTRICT**：標籤還有人在用時不能被硬刪除）。`(devoteeProfileId, tagId)` 唯一，防止同一標籤重複套用。 |
| `devotee_interactions` | 互動紀錄（電話/LINE/來宮/地址更新/關懷聯絡/詢問活動/詢問法會/其他），支援軟刪除 | `devoteeProfileId` → `devotee_profiles.id`（CASCADE）。 |
| `devotee_care_records` | 關懷狀態變更的歷史紀錄（標記/取消標記/聯絡/備註更新），append-only | `devoteeProfileId` → `devotee_profiles.id`（CASCADE）。與 `DevoteeProfile.careFlag`/`careReason`/`careNote` 等欄位是「目前狀態快照」＋「歷史事件紀錄」的關係，兩者都存在但用途不同：快照供高頻率查詢（名單/首頁統計），紀錄供稽核追溯。 |

## 三、與既有資料表的唯讀關聯（360 總覽／搜尋／統計使用）

以下既有資料表**沒有新增任何欄位或修改任何寫入邏輯**，V12.0 全部是唯讀查詢：

| 既有資料表 | 關聯方式 | 用途 |
|---|---|---|
| `households` | `members.householdId` | 家戶資料、家戶成員清單、家戶電話/地址（信眾本身沒有電話欄位，落回讀家戶電話）。 |
| `ritual_records` | `memberId`（可為空） | 宮務活動報名紀錄（普渡/宮慶/補庫/年度燈/其他，不含祭改）。 |
| `universal_salvation_details` / `universal_salvation_payments` | `ritual_records.id`（1:1） | 普渡贊普金額與收款分錄，用於捐款統計「贊普」「普渡」類別。 |
| `universal_salvation_entries` | `universal_salvation_details.id` | 歷代祖先／個人乙位正魂／冤親債主／無緣子女登記項目——全宮搜尋「祭祀資料」分類的資料來源。 |
| `purification_entries` | `memberId`（獨立欄位，不透過 `ritual_records.memberId`） | 祭改報名與收費紀錄。 |
| `offering_claims` / `offering_payments` | `sponsorMemberId` | 供品認捐紀錄與收款分錄。 |
| `payment_transactions` | `payerMemberId` | 收款交易主檔。 |
| `payment_allocations` → `receipt_lines` → `receipts` | `sourceType` + `sourceId` 多型參照 | 用於反查某筆應收項目（供品認捐/贊普/祭改）目前已開立且未作廢的收據號碼。 |
| `temple_events` | `ritual_records.templeEventId`（可為空） / `purification_entries.templeEventId`（必填） | 活動主檔，用於顯示活動名稱與搜尋結果跳轉連結。 |

## 四、家戶成員 1:1 唯一性保證

`devotee_profiles.memberId` 有 `@unique`（對應 PostgreSQL `devotee_profiles_memberId_key` 唯一索引），資料庫層級保證「一個 `Member` 最多一筆 `DevoteeProfile`」，即使應用層程式碼有漏洞也無法違反（已用真實 PostgreSQL 測試確認：對同一 `memberId` 插入第二筆會被資料庫拒絕）。

## 五、Cascade 刪除鏈

`households` → `members` → `devotee_profiles` → (`devotee_tag_assignments` / `devotee_interactions` / `devotee_care_records`) 全部是 `ON DELETE CASCADE`，與既有 `households`/`members` 的既有 cascade 慣例一致（刪除一戶會連動清除該戶信眾的全部延伸資料，不會留下孤兒資料）。已在 `sanxuan_v12_0_test` 測試資料庫用真實刪除操作＋筆數統計驗證過整條鏈會被正確清除。

## 六、捐款統計的資料涵蓋範圍（誠實揭露）

指令「六、8」要求依「添油香/贊普/普渡/宮慶/補庫/年度燈/祭改/花果/福壽龜/其他」十個類別統計，但目前系統只有 4 個類別有真實對應的登記/收費資料表：

| 類別 | 真實資料來源 | 是否有資料 |
|---|---|---|
| 花果 / 福壽龜 / 其他（供品） | `offering_claims`（依 `OfferingType.behaviorKind` 分類） | 有 |
| 贊普 / 普渡 | `universal_salvation_details`（同一組資料，非重複計算） | 有 |
| 祭改 | `purification_entries` | 有 |
| 添油香 / 補庫 / 年度燈單獨收費 / 宮慶獨立應收 | 無對應資料表 | **無**，固定顯示 0 並附文字說明 |

「普渡」與「贊普」在這套系統裡是同一組真實金額（普渡本身沒有另一筆獨立收費項目），`getDonationStats()` 把同一筆資料同時填入這兩個分類名稱下，**不是重複計算兩次**。
